## 摘要

面向嵌入式与边缘设备，日志系统既要满足可观测性需求，又要控制存储、CPU 与并发初始化开销。传统文本日志可读性高，但在高频写入场景下存在体积大、解析慢和跨进程格式同步成本高等问题。本文围绕“事件格式驱动 + 二进制编码 + 共享内存元数据”提出并实现了一套轻量日志方案 Optbinlog：通过解析 eventlog 格式文件构建事件描述表，将描述表写入共享文件并采用偏移量访问，日志写入阶段仅执行按类型定长/变长截断编码，读出阶段依据共享格式表完成解码与校验。

实现上，本文基于 C 语言完成了事件解析模块、共享元数据模块、二进制读写模块及多进程初始化控制逻辑，并构建了单机场景、多设备并发场景、初始化竞态场景和 round-trip 正确性场景的自动化实验脚本。实验采用重复运行、IQR 异常值过滤及均值/中位数/p95/标准差/95% 置信区间统计。结果表明，在本次“高负载单机”实验（`records=80000`）中，本地平台上 binary 相比 text 的端到端时间下降 **40.92%**、总空间下降 **69.23%**、吞吐提升 **67.75%**；Linux 平台上 binary 相比 text 的端到端时间下降 **50.85%**、吞吐提升 **102.75%**。在多设备场景（1/2/4/8 设备，每设备 1000 条）中，binary 在两平台均保持 **38.45%~65.73%** 的空间优势，但高并发下吞吐不占优，说明性能收益与平台调度/写入路径耦合较强。初始化竞态测试（本地 `procs=24`）显示 `create_success=1`、`open_existing_ok=23`、`init_done=1`，IQR 过滤后平均初始化耗时为 **21.243 ms**，说明共享文件状态机在更高并发下仍能维持单创建者收敛特性。

本文工作验证了“共享格式表 + 二进制编码”在日志系统中的可行性与工程收益，并给出了跨平台测试与外部基线补齐方案。后续将进一步补全 Linux x86_64 / ARM 嵌入式实机对照和截断日志异常检测能力。

关键词：二进制日志；共享内存；事件格式；并发初始化；性能优化

## Abstract

For embedded and edge systems, logging must balance observability and runtime overhead (storage, CPU, and concurrent initialization). Text-based logs are easy to read but often incur larger footprints and higher processing cost under high-frequency workloads. This thesis presents Optbinlog, a lightweight logging approach based on event-schema metadata in shared memory and binary encoding. Event definitions are parsed from eventlog files and materialized into a shared metadata file; runtime encoding/decoding uses offset-based access to avoid process-address dependency and reduce synchronization overhead.

The implementation includes an event parser, shared metadata manager, binary writer/reader, and concurrent initialization control in C. Automated benchmarks are built for single-node, multi-device concurrency, init-race, and round-trip correctness. Statistical reporting includes IQR outlier filtering and mean/median/p95/std/95% CI. In high-load single-node tests (`records=80000`), binary mode improves over text by **40.92% latency / 69.23% size / 67.75% throughput** on local platform, and **50.85% latency / 69.23% size / 102.75% throughput** on Linux platform. In multi-device tests (1/2/4/8 devices, 1000 records per device), binary mode keeps clear size savings but does not always preserve throughput advantage under heavier contention. Init-race results with `procs=24` show stable one-writer/multi-reader convergence (`create_success=1`, `open_existing_ok=23`, `init_done=1`).

The study confirms the practical value of schema-shared binary logging and provides a reproducible workflow for further Linux x86_64 and ARM embedded validation.

Keywords: binary logging; shared memory; event schema; concurrent initialization; performance optimization

### 1.1 研究背景

随着嵌入式设备和边缘计算场景的复杂度提升，日志系统不再仅用于故障排查，还承担状态追踪、性能分析和在线诊断等职责。传统文本日志在开发阶段友好，但在持续高频写入时会引入明显空间冗余和字符串处理开销，进而影响实时任务与系统稳定性。

开题目标强调了 Android EventLog/ftrace 相关思路与 Linux 嵌入式平台适配需求。本课题聚焦于以下核心矛盾：

1. 可读性与高效存储之间的矛盾。
2. 多进程并发写入下格式元数据的一致性与初始化竞争问题。
3. 工程可扩展性（新增事件字段、跨平台迁移）与实现复杂度之间的平衡。

1. 设计事件格式文件与统一事件元数据结构。

## 第2章 需求分析与总体设计

### 2.1 功能需求

系统需满足以下功能：

1. 读取 eventlog 目录中的事件定义文件。
2. 将事件定义初始化到共享文件，供写入/读取进程复用。
3. 支持按事件 ID 查找格式并进行二进制写入与读回。
4. 支持异常输入（非法 tag、元素个数不匹配）检测。

### 2.2 非功能需求

1. **性能**：降低写入耗时与端到端耗时，提升吞吐。
2. **空间**：降低日志总字节数（含共享元数据成本）。
3. **并发**：多进程初始化过程仅允许一个创建者，其余进程可安全复用。
4. **可扩展**：新增事件类型时尽量不改核心读写流程。

### 2.3 总体方案

结合技术文档与现有实现，系统采用“离线格式描述 + 在线共享复用 + 运行时编码”的路径：

1. 事件格式层：沿用 Android EventLog 思想，将 `unit` 改为 `bits`，得到 `(name|type|bits)`。
2. 元数据层：将事件数组、位图与元素描述写入共享文件。
3. 编解码层：写入时按 `type/len` 顺序编码，读取时按共享格式解码并回调。
4. 评测层：通过 Python 脚本统一执行与统计，输出 JSON + SVG。

## 第3章 关键技术与迭代过程

本章依据《hikbinlog技术文档》中的开发流程整理关键迭代。

### 3.1 事件格式从 Android EventLog 到 Hikbinlog

原 EventLog 采用 `(name|type|unit)`，本文改为 `(name|type|bits)`，使编码长度由类型与 bit 宽度直接决定，便于压缩与快速序列化。类型压缩为 `L/D/S` 三类，简化编码分支。

### 3.2 元数据结构迭代

技术文档显示，结构从“每事件单独 struct”演化为“通用 EventTag + EventTagEle + 二级数组 + Bitmap”。该演化的意义在于：

1. 避免事件种类增多后结构体爆炸。
2. 通过 Bitmap 进行 tag 合法性判定。
3. 通过数组分段控制查找复杂度。

### 3.3 共享文件与偏移量访问

共享文件初版存储指针地址时会受到进程映射基址差异影响。最终方案使用“首地址 + 偏移”替代裸指针，核心收益：

1. 跨进程映射地址不同仍可正确访问。
2. 数据结构可序列化、可迁移。
3. 与初始化状态机结合后更适合并发场景。

### 3.4 查找策略迭代

查找路径从 switch-case 到排序二分，再到二维数组索引与偏移直访。最终方案使用：

1. `arr = tag_id / 100` 与 `idx = tag_id % 100` 进行快速定位。
2. Bitmap 校验 tag 合法。
3. 偏移量获取 `EventTag` 与 `EventTagEle`。

该路径将查找代价稳定在常数级，降低大规模事件表下的开销。

## 第4章 系统实现

### 4.1 模块划分

基于当前工程实现，核心模块如下：

1. `optbinlog_eventlog`：解析事件格式目录，构建 `OptbinlogTagList`。
2. `optbinlog_shared`：共享文件初始化、状态控制、tag 查找。
3. `optbinlog_binlog`：二进制写入/读取与回调。
4. `run_bench.py / run_multi_bench.py / run_init_race.py / run_roundtrip.py`：实验驱动与统计可视化。

### 4.2 关键数据结构

`demo/include/optbinlog_shared.h` 给出以下关键结构：

1. `OptbinlogBitmap`：记录 tag 合法位。
2. `OptbinlogEventTagEle`：元素类型与长度（字节）描述。
3. `OptbinlogEventTag`：事件级描述，包含元素偏移。
4. `OptbinlogSharedTag`：共享头，包含状态与偏移入口。

### 4.3 初始化并发控制

`optbinlog_shared_init_from_dir()` 的核心流程为：

1. 若共享文件存在，先尝试只读映射。
2. 若状态为 `INITIALIZING`，短暂等待并重试。
3. 若不存在则创建文件、填充结构、最后置 `INITIALIZED`。

该流程在竞态测试中体现出“单创建者 + 多复用者”特征（第5章给出数据）。

### 4.4 二进制读写器

`optbinlog_binlog_write()` 与 `optbinlog_binlog_read()` 通过共享 tag 缓存减少重复查找，按元素类型写入：

1. `L`：按 `len` 字节写入无符号整数。
2. `D`：固定 8 字节写入 double。
3. `S`：按字段长度写入并截断/补零。

该实现避免每个 tag 定制函数，减少维护成本。

## 第5章 实验设计与结果分析

### 5.1 实验环境与数据来源

本章采用统一套件脚本 `demo/run_thesis_suite.py` 生成数据，结果位于：

1. `demo/results/thesis_suite_20260304_165518/`
2. `demo/results/latest`（软链接到最近一次实验）

本次实验口径：

1. 单机高负载（双平台）：`records=80000, repeats=8, warmup=2`。
2. 多设备模拟（双平台）：`devices=1,2,4,8`，`records_per_device=1000`，`repeats=6, warmup=2`。
3. 初始化竞争（仅本地）：`procs=24, repeats=12, warmup=2`。

### 5.2 评价指标与统计方法

1. 时间：`write_only_ms`、`end_to_end_ms`、`elapsed_ms`。
2. 空间：`bytes`、`shared_bytes`、`total_bytes`。
3. 吞吐：`throughput_*_rps`。
4. 稳定性：`mean/median/p95/std/95%CI`。
5. 异常值：IQR 过滤（默认系数 1.5）。

改进率定义：

- 时间改进率：`(baseline - target) / baseline * 100%`（越大越好）。
- 空间节省率：`(baseline - target) / baseline * 100%`（越大越好）。
- 吞吐提升率：`(target - baseline) / baseline * 100%`（越大越好）。

### 5.3 单机高负载对比（records=80000, repeats=8）

对照组：本地 `text/binary/syslog`；Linux `text/binary/syslog/ftrace`。  
双平台分别以各自 `text` 为基线。

| 平台/模式 | e2e mean (ms) | throughput_e2e (rps) | total bytes |
|---|---:|---:|---:|
| local text | 54.030 | 1,502,575 | 8,892,984 |
| local binary | 31.919 | 2,520,614 | 2,736,087 |
| local syslog | 37.979 | 2,114,858 | 5,716,000 |
| linux text | 211.580 | 381,522 | 8,892,984 |
| linux binary | 104.000 | 773,518 | 2,736,087 |
| linux syslog | 572.501 | 145,911 | 5,716,000 |

相对各平台 `text`：

1. local `binary`：e2e **+40.92%**，吞吐 **+67.75%**，空间 **+69.23%**。
2. local `syslog`：e2e **+29.71%**，吞吐 **+40.75%**，空间 **+35.72%**。
3. linux `binary`：e2e **+50.85%**，吞吐 **+102.75%**，空间 **+69.23%**。
4. linux `syslog`：e2e **-170.58%**，吞吐 **-61.76%**，空间 **+35.72%**。

解释：

1. `binary` 在两平台都体现了稳定的时延与吞吐优势。
2. Linux 上 `syslog` 明显退化，说明系统日志链路在高负载下成本较高。
3. 两平台 `total_bytes` 一致，说明空间收益主要来自编码格式本身。

### 5.4 多设备场景对比（repeats=6，records_per_device=1000）

本节报告 `d1/d2/d4/d8` 场景中 `elapsed / size / throughput` 的相对改进率（单位：%）。

本地平台（baseline=text）：

- `d1_r1000`：binary `-86.05 / +38.45 / -45.19`；syslog `-68.40 / +35.72 / -40.25`
- `d2_r1000`：binary `-44.86 / +54.04 / -30.59`；syslog `-33.11 / +35.72 / -24.96`
- `d4_r1000`：binary `-14.06 / +61.83 / -14.68`；syslog `-16.49 / +35.73 / -15.01`
- `d8_r1000`：binary `-21.25 / +65.73 / -16.88`；syslog `-29.07 / +35.73 / -23.61`

Linux 平台（baseline=text）：

- `d1_r1000`：binary `-368.63 / +38.45 / -78.28`；syslog `-45.45 / +35.72 / -30.98`
- `d2_r1000`：binary `-264.25 / +54.04 / -72.75`；syslog `-36.81 / +35.72 / -28.47`
- `d4_r1000`：binary `-218.66 / +61.83 / -68.60`；syslog `-124.72 / +35.73 / -55.44`
- `d8_r1000`：binary `-139.39 / +65.73 / -58.18`；syslog `-130.85 / +35.73 / -56.66`

结论：

1. `binary` 在两平台所有场景都保持显著空间优势（最高约 **65.73%**）。
2. 本轮高并发口径下，`binary` 与 `syslog` 在吞吐上均低于 `text`，说明多设备路径仍存在同步/调度瓶颈。
3. Linux 端退化更强，后续应重点优化跨进程锁竞争与系统调用路径。

### 5.5 初始化竞态测试（仅本地，procs=24, repeats=12）

IQR 过滤后：

1. `elapsed_ms` 均值 **21.243 ms**，p95 **26.071 ms**。
2. `wait_events` 均值 **3.417**，`wait_total_ms` 均值 **37.569 ms**。

## 第6章 结论与展望

### 6.1 结论

本文围绕嵌入式日志系统实现了共享格式元数据驱动的二进制日志方案，并完成了从数据结构、查找策略到并发初始化控制的系统化实现与验证。主要结论如下：

1. 在双平台单机高负载场景，binary 相比 text 保持稳定优势（local 与 Linux 均在时间/吞吐/空间上同时占优）。
2. 在双平台多设备场景，binary 的空间优势稳定，但吞吐在高并发口径下未体现优势，说明系统在并发写入路径仍有优化空间。
3. 初始化竞态实验（24 进程）表明共享状态机能稳定收敛到单创建者，但等待成本已具备统计显著性。
4. round-trip 与 bad-tag 检测通过，系统具备基础正确性保证。

### 6.2 后续工作

1. 按开题计划补齐 Linux x86_64 与 ARM（Raspberry Pi / i.MX）实机数据，并记录内核/文件系统/存储介质信息。
2. 引入更完整的外部基线（journald、Android EventLog 设备侧口径）并统一对照条件。
3. 统一采集 CPU 指标（如 `perf stat` 周期、上下文切换、cache miss）以解释多设备退化原因。
4. 完善截断日志检测、损坏恢复与异常路径测试覆盖。
5. 增加更长时长与更高并发压测，形成可复现实验矩阵。

### 6.3 下一步优化计划（对齐开题报告）

结合开题报告中 Android EventLog / Linux ftrace / 嵌入式平台目标，建议下一阶段优先级如下：

1. `P0`：补齐外部对照闭环。  
口径：`text`、`optbinlog binary`、`syslog/journald`、`ftrace`，可行时增加 Android EventLog 设备侧数据。
2. `P0`：多设备性能退化专项优化。  
重点：共享初始化锁等待、文件写入 flush 策略、进程调度与批量写策略。
3. `P1`：双平台实机复现。  
至少覆盖 x86_64 Linux 与 ARM Linux（Raspberry Pi / i.MX），并固化平台元信息。
4. `P1`：统计口径强化。  
除 mean/median/p95/std/95%CI 外，增加 p99、CPU 周期与上下文切换指标。
5. `P2`：工程化复现。  
保留 `run_thesis_suite.py` 为主入口，补充参数模板与自动报告导出。

## 摘要

面向嵌入式与边缘设备，日志系统既要满足可观测性需求，又要控制存储、CPU 与并发初始化开销。传统文本日志可读性高，但在高频写入场景下存在体积大、解析慢和跨进程格式同步成本高等问题。本文围绕“事件格式驱动 + 二进制编码 + 共享内存元数据”提出并实现了一套轻量日志方案 Optbinlog：通过解析 eventlog 格式文件构建事件描述表，将描述表写入共享文件并采用偏移量访问，日志写入阶段仅执行按类型定长/变长截断编码，读出阶段依据共享格式表完成解码与校验。

实现上，本文基于 C 语言完成了事件解析模块、共享元数据模块、二进制读写模块及多进程初始化控制逻辑，并构建了单机场景、多设备并发场景、初始化竞态场景和 round-trip 正确性场景的自动化实验脚本。实验采用重复运行、IQR 异常值过滤及均值/中位数/p95/标准差/95% 置信区间统计。结果表明，在 `records=20000` 的单机场景中，相比文本模式，二进制模式端到端时间均值下降 **73.88%**，总空间开销下降 **66.57%**，端到端吞吐提升 **283.31%**；相较系统日志对照（syslog），二进制模式在时间与吞吐上也表现更优。在多设备场景（1/2/4 设备，每设备 500 条）中，二进制模式在 1、2 设备下具备稳定时间优势，在 2、4 设备下体现空间优势；初始化竞态测试显示 `create_success=1`、`open_existing_ok=9`、`init_done=1`，IQR 过滤后的平均初始化耗时为 1.889 ms，说明共享文件初始化策略可避免大规模竞争冲突。

本文工作验证了“共享格式表 + 二进制编码”在日志系统中的可行性与工程收益，并给出了跨平台测试与外部基线补齐方案。后续将进一步补全 Linux x86_64 / ARM 嵌入式实机对照和截断日志异常检测能力。

关键词：二进制日志；共享内存；事件格式；并发初始化；性能优化

## Abstract

For embedded and edge systems, logging must balance observability and runtime overhead (storage, CPU, and concurrent initialization). Text-based logs are easy to read but often incur larger footprints and higher processing cost under high-frequency workloads. This thesis presents Optbinlog, a lightweight logging approach based on event-schema metadata in shared memory and binary encoding. Event definitions are parsed from eventlog files and materialized into a shared metadata file; runtime encoding/decoding uses offset-based access to avoid process-address dependency and reduce synchronization overhead.

The implementation includes an event parser, shared metadata manager, binary writer/reader, and concurrent initialization control in C. Automated benchmarks are built for single-node, multi-device concurrency, init-race, and round-trip correctness. Statistical reporting includes IQR outlier filtering and mean/median/p95/std/95% CI. In single-node tests (`records=20000`), binary mode reduces end-to-end latency by **73.88%**, cuts total size by **66.57%**, and improves throughput by **283.31%** versus text mode. In multi-device tests (1/2/4 devices, 500 records per device), binary mode shows consistent latency gain on low-to-medium concurrency and notable size benefit on larger workloads. Init-race results demonstrate robust one-writer/multi-reader convergence.

The study confirms the practical value of schema-shared binary logging and provides a reproducible workflow for further Linux x86_64 and ARM embedded validation.

Keywords: binary logging; shared memory; event schema; concurrent initialization; performance optimization

## 第2章 需求分析与总体设计

### 2.1 功能需求

系统需满足以下功能：

1. 读取 eventlog 目录中的事件定义文件。
2. 将事件定义初始化到共享文件，供写入/读取进程复用。
3. 支持按事件 ID 查找格式并进行二进制写入与读回。
4. 支持异常输入（非法 tag、元素个数不匹配）检测。

### 2.2 非功能需求

1. **性能**：降低写入耗时与端到端耗时，提升吞吐。
2. **空间**：降低日志总字节数（含共享元数据成本）。
3. **并发**：多进程初始化过程仅允许一个创建者，其余进程可安全复用。
4. **可扩展**：新增事件类型时尽量不改核心读写流程。

### 2.3 总体方案

结合技术文档与现有实现，系统采用“离线格式描述 + 在线共享复用 + 运行时编码”的路径：

1. 事件格式层：沿用 Android EventLog 思想，将 `unit` 改为 `bits`，得到 `(name|type|bits)`。
2. 元数据层：将事件数组、位图与元素描述写入共享文件。
3. 编解码层：写入时按 `type/len` 顺序编码，读取时按共享格式解码并回调。
4. 评测层：通过 Python 脚本统一执行与统计，输出 JSON + SVG。

## 第3章 关键技术与迭代过程

本章依据《hikbinlog技术文档》中的开发流程整理关键迭代。

### 3.1 事件格式从 Android EventLog 到 Hikbinlog

原 EventLog 采用 `(name|type|unit)`，本文改为 `(name|type|bits)`，使编码长度由类型与 bit 宽度直接决定，便于压缩与快速序列化。类型压缩为 `L/D/S` 三类，简化编码分支。

### 3.2 元数据结构迭代

技术文档显示，结构从“每事件单独 struct”演化为“通用 EventTag + EventTagEle + 二级数组 + Bitmap”。该演化的意义在于：

1. 避免事件种类增多后结构体爆炸。
2. 通过 Bitmap 进行 tag 合法性判定。
3. 通过数组分段控制查找复杂度。

### 3.3 共享文件与偏移量访问

共享文件初版存储指针地址时会受到进程映射基址差异影响。最终方案使用“首地址 + 偏移”替代裸指针，核心收益：

1. 跨进程映射地址不同仍可正确访问。
2. 数据结构可序列化、可迁移。
3. 与初始化状态机结合后更适合并发场景。

### 3.4 查找策略迭代

查找路径从 switch-case 到排序二分，再到二维数组索引与偏移直访。最终方案使用：

1. `arr = tag_id / 100` 与 `idx = tag_id % 100` 进行快速定位。
2. Bitmap 校验 tag 合法。
3. 偏移量获取 `EventTag` 与 `EventTagEle`。

该路径将查找代价稳定在常数级，降低大规模事件表下的开销。

## 第4章 系统实现

### 4.1 模块划分

基于当前工程实现，核心模块如下：

1. `optbinlog_eventlog`：解析事件格式目录，构建 `OptbinlogTagList`。
2. `optbinlog_shared`：共享文件初始化、状态控制、tag 查找。
3. `optbinlog_binlog`：二进制写入/读取与回调。
4. `run_bench.py / run_multi_bench.py / run_init_race.py / run_roundtrip.py`：实验驱动与统计可视化。

### 4.2 关键数据结构

`demo/include/optbinlog_shared.h` 给出以下关键结构：

1. `OptbinlogBitmap`：记录 tag 合法位。
2. `OptbinlogEventTagEle`：元素类型与长度（字节）描述。
3. `OptbinlogEventTag`：事件级描述，包含元素偏移。
4. `OptbinlogSharedTag`：共享头，包含状态与偏移入口。

### 4.3 初始化并发控制

`optbinlog_shared_init_from_dir()` 的核心流程为：

1. 若共享文件存在，先尝试只读映射。
2. 若状态为 `INITIALIZING`，短暂等待并重试。
3. 若不存在则创建文件、填充结构、最后置 `INITIALIZED`。

该流程在竞态测试中体现出“单创建者 + 多复用者”特征（第5章给出数据）。

### 4.4 二进制读写器

`optbinlog_binlog_write()` 与 `optbinlog_binlog_read()` 通过共享 tag 缓存减少重复查找，按元素类型写入：

1. `L`：按 `len` 字节写入无符号整数。
2. `D`：固定 8 字节写入 double。
3. `S`：按字段长度写入并截断/补零。

该实现避免每个 tag 定制函数，减少维护成本。

## 第5章 实验设计与结果分析

### 5.1 实验环境与数据来源

本文主结果来自以下文件：

1. 单机性能：`demo/bench/bench_result.json`。
2. 多设备并发：`demo/bench_multi/bench_multi_result.json`。
3. 初始化竞态：`demo/init_race/init_race_result.json`。
4. 正确性回归：`demo/roundtrip/roundtrip_result.json`。
5. 平台元信息：`demo/bench_platform/thesis_main_arm64/platform_report.json`。

当前采集到的平台信息为：`arch=arm64`、`kernel=25.3.0`、`filesystem=apfs`。这组数据用于开发机验证；Linux x86_64 与 ARM 嵌入式实机对照将在后续补齐。

### 5.2 评价指标与统计方法

1. 时间：`write_only_ms`、`end_to_end_ms`、`elapsed_ms`。
2. 空间：`bytes`、`shared_bytes`、`total_bytes`。
3. 吞吐：`throughput_*_rps`。
4. 稳定性：`mean/median/p95/std/95%CI`。
5. 异常值：IQR 过滤（默认系数 1.5）。

改进率定义：

- 时间改进率：`(baseline - target) / baseline * 100%`（越大越好）。
- 空间节省率：`(baseline - target) / baseline * 100%`（越大越好）。
- 吞吐提升率：`(target - baseline) / baseline * 100%`（越大越好）。

### 5.3 单机场景对比（records=20000, repeats=6）

对照组：`text`、`binary`、`syslog`，基线为 `text`。

| 模式 | e2e mean (ms) | write mean (ms) | total bytes | throughput_e2e (rps) |
|---|---:|---:|---:|---:|
| text | 8.402 | 8.003 | 1,645,389 | 2,385,332 |
| binary | 2.195 | 0.444 | 549,975 | 9,143,217 |
| syslog | 11.572 | 11.235 | 1,645,389 | 1,728,317 |

相对 `text`：

1. `binary`：e2e 降低 **73.88%**，write 降低 **94.46%**，空间降低 **66.57%**，吞吐提升 **283.31%**。
2. `syslog`：e2e 增加 **37.73%**，吞吐下降 **27.54%**，空间基本持平。

解释：

1. 文本与 syslog 主要瓶颈在字符串格式化与系统日志路径开销。
2. 二进制模式将字段按固定/截断长度序列化，减少字符串处理与体积冗余。
3. `write_only` 与 `end_to_end` 差异说明元数据访问与文件流程仍有固定成本，但总体被显著摊薄。

### 5.4 多设备场景对比（repeats=6，records_per_device=500）

#### 场景 A：d1_r500（1 设备）

- text：0.786 ms，41,126 B，640,248 rps
- binary：0.623 ms，47,499 B，803,567 rps
- syslog：0.976 ms，41,126 B，514,846 rps

相对 text：

1. binary 时间改进 **20.74%**，吞吐提升 **25.51%**，但总字节上升 **15.50%**。
2. 原因：小规模下共享元数据文件占比高，导致 binary `total_bytes` 被抬升。

#### 场景 B：d2_r500（2 设备）

- text：1.017 ms，82,274 B，986,400 rps
- binary：0.971 ms，60,383 B，1,063,207 rps
- syslog：1.317 ms，82,274 B，761,495 rps

相对 text：

1. binary 时间改进 **4.56%**，空间节省 **26.61%**，吞吐提升 **7.79%**。
2. 共享开销被并发工作量摊薄后，binary 的空间优势开始显现。

#### 场景 C：d4_r500（4 设备）

- text：1.365 ms，164,516 B，1,468,685 rps
- binary：1.627 ms，86,151 B，1,328,854 rps
- syslog：1.630 ms，164,516 B，1,230,539 rps

相对 text：

1. binary 空间节省 **47.63%**，但时间和吞吐不占优。
2. 结合 std 与 CI 可见 binary 在该场景波动较大（可能受进程调度、短时资源竞争影响）。

结论：

1. binary 在低中并发下表现出较稳定时间/吞吐优势。
2. 在更高并发下，空间优势仍明显，但性能收益受平台调度与测试噪声影响，需要在 Linux 实机与更长时长下继续验证。

### 5.5 初始化竞态测试（procs=10, repeats=6）

IQR 过滤后（保留 4 次）：

1. `elapsed_ms` 均值 **1.889 ms**，p95 **2.047 ms**。
2. `create_success=1`、`open_existing_ok=9`、`init_done=1`。
3. `wait_events` 与 `wait_total_ms` 统计均值为 0。

说明多进程场景下共享初始化过程基本符合预期：单进程创建，其余进程复用并快速打开。

### 5.6 正确性回归（round-trip）

`roundtrip_result.json` 结果：

- `roundtrip_ok=1`
- `bad_tag_detected=1`
- `truncated_detected=0`
- `records_checked=32`

说明：

1. 正向写入-读回一致性通过。
2. 非法 tag 能被正确识别。
3. 截断检测覆盖仍需增强（当前场景未触发预期检测路径），可作为后续改进项。

## 第6章 结论与展望

### 6.1 结论

本文围绕嵌入式日志系统实现了共享格式元数据驱动的二进制日志方案，并完成了从数据结构、查找策略到并发初始化控制的系统化实现与验证。主要结论如下：

1. 在单机场景，binary 相比 text 在时间、吞吐、空间三项指标均显著提升。
2. 在多设备场景，binary 的空间优势在并发提升后更明显；性能优势受并发级别与平台调度影响。
3. 初始化竞态实验表明共享文件状态机能有效控制多进程争用。
4. round-trip 与 bad-tag 检测通过，系统具备基础正确性保证。

### 6.2 后续工作

1. 按开题计划补齐 Linux x86_64 与 ARM（Raspberry Pi / i.MX）实机数据。
2. 引入 ftrace 与 journald 的更系统对照，并统一采集 CPU 指标（如 perf）。
3. 完善截断日志检测与恢复策略。
4. 增加更高并发与更长时长压测，构建平台无关的稳定性结论。

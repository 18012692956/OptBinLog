作者：邓泽宇（学号：2022302111098）  
单位：武汉大学

## 摘要

面向嵌入式与边缘设备，日志系统既要保障可观测性，又要控制存储、CPU 与并发初始化开销。围绕“开题报告中提出的 Android EventLog / Linux ftrace / 嵌入式平台对齐目标”，本文提出并实现 Optbinlog：采用事件格式描述文件 + 共享元数据 + 二进制编码路径，将格式解析成本从热路径迁出，并通过偏移量访问保证多进程映射一致性。

本文首先调查国内外代表性日志系统（Linux ftrace、LTTng、Zephyr logging、systemd-journald、NanoLog、RT-Thread ulog、OpenHarmony HiLog Lite）及其优化代码路径，再与 Optbinlog 在算法复杂度、数据结构、并发行为和工程适配性上做对比。随后基于“先理论、后实验”的方法，对不同日志模式推导日志条数 `N`、设备数 `D` 与时间/空间/吞吐关系，再用现有实验数据逐项验证。

实验由三轮组成：
1. 既有 thesis suite（2026-03-04）与 Linux 5/10 节点真实并发结果。
2. 2026-03-05 新增本地复现实验（`text/binary/syslog`，单机高负载 + 多设备扫描）。
3. binary 多设备竞争与 init-race 稳定性验证。

结果显示：在本地单机高负载（`records=100000`）中，binary 相对 text 时间下降 `38.50%`、吞吐提升 `62.72%`、空间节省 `69.31%`；在研究复现集中，`nanolog_like` 与 `hilog_lite_like` 分别达到 `-92.06%` 与 `-61.55%` 的时间降幅，而 `ulog_async_like` 由于“异步+文本后端”路径在当前负载下出现 `+40.54%` 时间增幅。Linux 真实并发 L1（5/10 节点）中，binary 相对 ftrace 的时间下降分别为 `7.03%` 与 `35.49%`，且空间节省稳定在 `67.87%`。进一步在语义拉齐组中，原先数量级差异收敛到同一量级，说明语义口径是跨系统比较的关键前提。理论推导与实验总体一致：空间随 `N` 线性增长，binary 的固定共享开销在大 `N` 下被摊薄；吞吐随 `D` 增长存在饱和区与回落点。

关键词：嵌入式日志；二进制编码；共享内存；并发初始化；性能建模

## Abstract

For embedded and edge systems, logging must balance observability with overhead (storage, CPU, and concurrent initialization). This thesis implements Optbinlog, a schema-shared binary logging approach aligned with the proposal targets (Android EventLog / Linux ftrace / embedded deployment). Event metadata are parsed once, materialized in shared memory, and accessed via offsets, so runtime hot-path cost is reduced and cross-process mapping remains stable.

The work first surveys representative domestic and international logging systems (Linux ftrace, LTTng, Zephyr logging, systemd-journald, NanoLog, RT-Thread ulog, OpenHarmony HiLog Lite) and analyzes their optimization code paths. Then a theory-first model is built to derive how time/space/throughput scale with log count `N` and device count `D`. Finally, measured results are compared against the derived trends.

Three experiment rounds are included: retained thesis-suite results (2026-03-04), new expanded cross-platform runs and research reproductions (NanoLog/Zephyr/uLog/HiLog-Lite-like) in 2026-03-05, and high-contention/init-race validations. On local high-load runs (`records=100000`), binary improves over text by `38.50%` in end-to-end time, `62.72%` in throughput, and `69.31%` in size. In real Linux concurrent runs, binary remains faster than ftrace at both 5-node and 10-node scales (`7.03%` and `35.49%` time reduction respectively, with stable size savings). Research reproductions show that cross-system comparisons must separate semantic mismatch from implementation overhead: large gaps under non-equivalent semantics shrink significantly after semantic alignment. Theoretical and empirical trends are largely consistent.

Keywords: embedded logging; binary encoding; shared metadata; concurrency; performance modeling

## 第1章 概述

### 1.1 研究背景

嵌入式系统日志常处于“高频写入、有限存储、多进程并发、实时约束”并存的环境。传统文本日志在可读性上有优势，但会带来：

1. 字符串格式化与解析开销高。
2. 单条日志冗余字节多，长期空间成本高。
3. 多进程共享格式信息时容易出现初始化竞争和一致性问题。

## 第2章 国内外研究调查与缺陷分析

### 2.1 调研范围与方法

本章基于官方文档、论文与代码仓库进行调研，聚焦“热路径优化机制”和“可复现实现入口”。

调研对象：
1. Linux ftrace / kernel ring buffer。
2. LTTng。
3. Zephyr logging subsystem。
4. systemd-journald。
5. NanoLog（USENIX ATC 2018）。
6. RT-Thread ulog。
7. OpenHarmony HiLog Lite。

### 2.2 国际代表系统与优化思路（代码级）

#### 2.2.1 Linux ftrace（内核追踪）

代码与文档显示其核心是 `per-CPU lockless ring buffer`。在 `ring_buffer.c` 中，写入路径围绕“预留 -> 填充 -> 提交”组织，关键是把竞争局部化到 CPU 本地缓冲，降低全局锁竞争。

1. 数据结构：每 CPU buffer + page 链式组织，维护 head/commit/tail。
2. 算法逻辑：优先本地写入、批量可见性提交，读写阶段解耦。
3. 场景定位：内核 tracing、高并发诊断；功能强但接入复杂度高于应用层日志库。

#### 2.2.2 LTTng（Linux Tracing Toolkit Next Generation）

LTTng 采用 tracepoint instrumentation + lockless buffer 管线，强调高吞吐追踪和离线分析。

1. 数据结构：channel/sub-buffer 等追踪缓冲结构。
2. 算法逻辑：将事件采集与后处理分层，追踪点开销可控。
3. 场景定位：系统级性能分析与可观测平台，部署和运维要求较高。

#### 2.2.3 Zephyr logging（RTOS 场景）

在 Zephyr `Kconfig.mode` 与 `log_msg.c` 可见，关键机制是 deferred logging。

1. 数据结构：`log_msg` 对象 + 消息池（MPSC packet buffer）。
2. 算法逻辑：前台线程只做封包入队，后台线程统一输出；支持 overflow/drop 策略。
3. 场景定位：RTOS 中实时任务优先，要求前台日志路径尽量短、可控。

#### 2.2.4 systemd-journald

其 `JOURNAL_FILE_FORMAT` 与 `journald-server` 路径强调二进制对象与索引。

1. 数据结构：header + objects + 哈希/索引。
2. 算法逻辑：写入与检索并重，保证一致性与查询能力。
3. 场景定位：通用 Linux 系统服务日志；对轻量嵌入式而言依赖偏重。

#### 2.2.5 NanoLog（研究型高性能日志）

从 `NanoLogCpp17.h`、`RuntimeLogger.cc` 可见，NanoLog 核心是“编译期提取格式 + 运行期仅记录动态参数”。

1. 数据结构：staging buffer、runtime logger 队列/缓冲。
2. 算法逻辑：前台路径尽量只做参数打包，压缩与格式恢复转移到后台/离线。
3. 场景定位：超高频日志、允许离线恢复文本的性能敏感场景。

### 2.3 国内生态系统与优化思路（代码级）

#### 2.3.1 RT-Thread ulog

`ulog.c` 提供异步输出和多 backend 抽象，思路是“前台快写缓冲，后台慢速输出”。

1. 数据结构：ringbuffer/rbb、backend 列表。
2. 算法逻辑：异步线程统一调度输出，减轻业务线程阻塞。
3. 场景定位：RT-Thread 设备侧调试与在线日志。

#### 2.3.2 OpenHarmony HiLog Lite

HiLog Lite 在格式能力上做约束（减少复杂格式），换取更低资源开销。

1. 数据结构：轻量日志缓冲和简化格式存储。
2. 算法逻辑：限制格式复杂度，降低格式化路径成本。
3. 场景定位：资源受限设备的系统日志基础设施。

### 2.4 NanoLog/Zephyr 与 Optbinlog 的核心差异

| 维度 | Optbinlog | NanoLog | Zephyr deferred |
|---|---|---|---|
| 语义来源 | eventlog schema（运行时可扩展） | 编译期格式抽取 | 运行时日志消息对象 |
| 热路径 | schema 查找 + typed 编码 + CRC | 参数打包 + 快速写缓冲 | 入队（queue） |
| 后台路径 | 主要是批量写盘（无独立日志后台线程） | 压缩/离线恢复 | 后台线程输出 |
| 主要优势 | 空间效率 + 共享元数据一致性 | 极低调用点开销 | 实时线程不阻塞 |
| 主要代价 | 初始化和 schema 路径有固定成本 | 工具链耦合较强 | 队列锁和线程调度开销 |

### 2.5 典型优化代码入口（可追溯）

| 系统 | 关键优化点 | 代码/文档入口 |
|---|---|---|
| Linux ftrace | 每 CPU 无锁 ring buffer | `kernel/trace/ring_buffer.c`，设计文档 `ring-buffer-design` |
| LTTng | tracepoint + lockless buffer | LTTng Concepts 官方文档 |
| Zephyr logging | deferred + 消息池 | `subsys/logging/Kconfig.mode`、`subsys/logging/log_msg.c` |
| systemd-journald | 二进制对象与索引文件 | `journald-server.c`、`JOURNAL_FILE_FORMAT` |
| NanoLog | 编译期格式抽取 + 运行时二进制参数 | `NanoLogCpp17.h`、`RuntimeLogger.cc` |
| RT-Thread ulog | 异步输出与多 backend | `components/utilities/ulog/ulog.c` + ulog 文档 |
| OpenHarmony HiLog Lite | 受限格式降低开销 | `hiviewdfx_hilog_lite` 文档/代码 |

### 2.6 现有方案缺陷归纳

从嵌入式角度可抽象出四类常见缺口：

1. **格式化开销仍在热路径**：text/syslog 在高频场景下受字符串格式化与内核接口调用影响明显。
2. **元数据复用不足**：多进程系统若每进程重复解析格式，启动和内存成本上升。
3. **并发初始化一致性风险**：共享资源初次创建缺少状态机时，容易出现竞态。
4. **跨平台复现实验缺失**：只在单平台结论容易失真，难支撑开题报告目标。

### 2.7 Optbinlog 对缺陷的对应关系

1. 用“事件格式文件 + 二进制编码”替代热路径字符串格式化。
2. 用共享元数据文件持久化 tag/element 描述，进程间偏移复用。
3. 用 `INITIALIZING -> INITIALIZED` 状态机控制并发初始化。
4. 用统一脚本输出 mean/median/p95/std/95%CI 与 IQR 过滤结果，支持多轮迭代对比。

### 2.8 本地复现尝试与可行性说明（2026-03-05）

本轮按“先读代码、再复现实验”的原则执行：

1. 已完成官方代码级阅读：ftrace、Zephyr、NanoLog、ulog 等核心实现入口。
2. 受网络与平台限制：无法完整拉取/编译全部外部系统并在本机跑 Linux tracing 全栈。
3. 已执行算法级复现：实现 `nanolog_like`、`zephyr_deferred_like`、`ulog_async_like`、`hilog_lite_like` 及“语义拉齐”版本（第5章给出数据）。

该约束已在第5章实验方法中显式记录，避免把“平台不可复现”误写成“算法结论”。

## 第3章 Optbinlog 设计迭代与工程实现

### 3.1 设计目标

1. 降低单条日志热路径开销（时间）。
2. 降低长期写入空间消耗（空间）。
3. 支持多进程共享元数据与安全初始化（并发）。
4. 对齐嵌入式可部署需求（工程可落地）。

### 3.2 事件格式与元数据结构

实现采用 `(name|type|bits)` 事件定义，类型简化为 `L/D/S`：

1. `L`：定长整数（按 bits 写入）。
2. `D`：双精度浮点（8 字节）。
3. `S`：字符串（按字段长度截断/补零）。

元数据在共享文件内由以下结构组织：

1. `OptbinlogSharedTag`：全局头与状态。
2. `OptbinlogBitmap`：tag 合法性位图。
3. `OptbinlogEventTag`：事件定义入口。
4. `OptbinlogEventTagEle`：元素类型与长度描述。

### 3.3 写入/读取路径

写入路径：
1. 依据 tag 查找共享格式。
2. 依元素序列编码 payload。
3. 写入 `len + payload + crc32`。

读取路径：
1. 读取帧长与 payload。
2. 校验 CRC。
3. 基于共享格式解码并回调。

### 3.4 关键迭代

1. 从“每事件独立结构”迭代为“统一结构 + 位图 + 偏移表”。
2. 从进程内指针共享迭代为“偏移量共享”（解决跨进程映射基址差异）。
3. 从 switch-case 查找迭代为 `arr=id/100, idx=id%100` 的常数级路径。
4. 增加并发初始化状态机和 init-race 压力验证。

### 3.5 与实际场景的匹配性

1. 小样本 `N`：固定共享成本占比高，时间优势不一定立刻出现。
2. 中大样本 `N`：固定成本被摊薄，binary 在时间/空间上更稳定。
3. 高并发 `D`：吞吐会进入饱和区，需结合调度/I/O 竞争做专项优化。

## 第4章 理论模型与数学推导

本章先做抽象推导，不代入具体事件文件数值。

### 4.1 符号定义

1. `N`：总日志条数。
2. `D`：并发设备数（或并发进程数）。
3. `r`：每设备日志条数，`N = D * r`。
4. `k`：事件类型（tag）索引，`k ∈ {1,...,T}`。
5. `p_k`：事件类型 `k` 的概率，`Σ_k p_k = 1`。
6. `e_k`：事件 `k` 的字段数。
7. `l_{k,j}`：事件 `k` 第 `j` 个字段编码字节数。
8. `L_k = Σ_{j=1}^{e_k} l_{k,j}`：事件 `k` 的有效载荷字节数。
9. `A_m`：模式 `m` 的固定时间开销（初始化、打开文件、预热）。
10. `C_m`：模式 `m` 的单条边际时间开销。
11. `B_m`：模式 `m` 的单条边际空间开销。
12. `S_{0,m}`：模式 `m` 的固定空间开销（例如共享元数据）。
13. `Q_m`：吞吐（records/s）。

### 4.2 从算法与数据结构到单条代价

为避免“只看结果不看机制”，先把每种模式拆成操作级成本。

#### 4.2.1 text

单条记录包含：
1. 读取输入字段。
2. `snprintf`/拼接格式串。
3. `write` 到文件。

时间可写为：

`t_text(k) = t_parse(k) + t_fmt(k) + t_write(bytes_text(k))`

空间可写为：

`b_text(k) = b_prefix + b_delim + b_num(k) + b_str(k)`

其中 `b_num(k)` 是数字转十进制后的字符长度，随数值大小缓慢变化。

#### 4.2.2 binary（optbinlog）

单条记录包含：
1. tag 查找（数组/偏移访问）。
2. 按字段类型编码 payload。
3. 计算 CRC。
4. 写帧 `len + payload + crc`。

时间：

`t_bin(k) = t_lookup + t_encode(k) + t_crc(L_k) + t_write(4 + header_bin + L_k + 4)`

空间：

`b_bin(k) = 4 + header_bin + L_k + 4`

固定空间：

`S_{0,bin} = S_shared`

#### 4.2.3 syslog / ftrace

抽象为“系统日志接口 + 内核路径”：

`t_sys(k) = t_format_sys(k) + t_syscall + t_kernel_path + t_write_kernel`

`t_ftrace(k) = t_format_ftrace(k) + t_trace_write + t_ring_or_file_path`

空间：

`b_sys(k) = b_sys_header + b_sys_payload(k)`

`b_ftrace(k) = b_ftrace_header + b_ftrace_payload(k)`

#### 4.2.4 nanolog_like / zephyr_deferred_like / ulog_async_like / hilog_lite_like

`nanolog_like`（固定字段二进制）：

`t_nano = t_pack_fixed + t_memcpy + t_write`

`b_nano = const`

`zephyr_deferred_like`（前台入队 + 后台批刷）：

`t_z_front = t_enqueue + t_signal`

`t_z_back = (t_dequeue + t_batch_write(k_b))/k_b`

`t_z = t_z_front + t_z_back`

`b_z = b_queue_node + b_fixed + b_batch_meta/k_b`

`ulog_async_like`（异步队列 + 文本 backend）：

`t_ulog = t_enqueue + t_cond_signal + (t_dequeue + t_format_text + t_backend_write)/k_b`

`b_ulog = b_queue_node + b_text_payload`

`hilog_lite_like`（受限字段 + 轻量头）：

`t_hilog = t_compact_format + t_write(header_lite + payload_lite)`

`b_hilog = b_header_lite + b_payload_lite`

### 4.3 空间随 `N` 的详细推导

#### 4.3.1 期望单条字节

对任意模式 `m`，定义：

`B_m = E[b_m(k)] = Σ_k p_k b_m(k)`

则总空间：

`S_m(N) = S_{0,m} + N * B_m`

这是严格的一阶线性模型，斜率为 `B_m`，截距为 `S_{0,m}`。

#### 4.3.2 binary 的显式表达

由 4.2.2 得：

`b_bin(k) = 8 + header_bin + L_k`

`B_bin = 8 + header_bin + Σ_k p_k L_k`

`S_bin(N) = S_shared + N * (8 + header_bin + Σ_k p_k L_k)`

#### 4.3.3 节省率与导数

binary 相对模式 `x` 的空间节省率：

`η_{bin/x}(N) = 1 - S_bin(N)/S_x(N)`

若 `S_{0,x}=0`，则

`η_{bin/x}(N) = 1 - (B_bin + S_shared/N)/B_x`

求导：

`dη_{bin/x}/dN = S_shared / (N^2 * B_x) > 0`

说明 `N` 越大，`S_shared/N` 越小，binary 空间优势越稳定。

#### 4.3.4 空间交叉点

当比较 `bin` 与 `x`：

`S_bin(N) <= S_x(N)`

`S_shared + N B_bin <= S_{0,x} + N B_x`

若 `B_x > B_bin`，可得交叉点：

`N*_{space}(bin,x) = (S_shared - S_{0,x}) / (B_x - B_bin)`

当 `N > N*_{space}`，binary 空间一定优于 `x`。

### 4.4 时间随 `N` 的详细推导

#### 4.4.1 从单条时间到总时间

定义：

`C_m = E[t_m(k)] = Σ_k p_k t_m(k)`

则总时间：

`T_m(N) = A_m + N C_m`

该形式直接对应实验中的 `elapsed/write_only/end_to_end` 指标。

#### 4.4.2 吞吐公式与单调性

`Q_m(N) = 1000N / T_m(N) = 1000 / (C_m + A_m/N)`

对 `N` 求导：

`dQ_m/dN = 1000 * A_m / (N^2 * (C_m + A_m/N)^2) > 0`

说明在模型中吞吐随 `N` 单调上升并渐近：

`lim_{N->∞} Q_m(N) = 1000 / C_m`

二阶导在常见参数下为负，表示“先快后慢”的收益递减，这与实验中大 `N` 后增益变缓一致。

#### 4.4.3 时间交叉点（binary 何时优于其他模式）

比较 `bin` 与 `x`：

`T_bin(N) <= T_x(N)`

`A_bin + N C_bin <= A_x + N C_x`

当 `C_x > C_bin` 时：

`N*_{time}(bin,x) = (A_bin - A_x) / (C_x - C_bin)`

解释：
1. 若 `A_bin` 较大（共享初始化重），小 `N` 可能不占优。
2. 随 `N` 增大，`N C` 项主导，只要 `C_bin < C_x`，最终会跨过交叉点。

#### 4.4.4 `C_m` 的字段复杂度展开

为把“常数项差异”写清楚，可把边际开销展开到字段数级别：

`C_m = c_{m,0} + c_{m,1} E[e_k] + c_{m,2} E[L_k] + c_{m,3} E[syscall_m]`

其中：
1. `E[e_k]` 决定字段遍历成本（编码循环次数）。
2. `E[L_k]` 决定 memcpy/CRC/写入字节相关成本。
3. `E[syscall_m]` 反映模式是否强依赖系统调用路径。

可得典型关系：
1. text/syslog：`c_{m,1}` 与 `c_{m,3}` 较大（格式化 + syscall）。
2. binary：`c_{m,1}` 中等，`c_{m,3}` 中等，另有 `A_bin` 较大（共享初始化）。
3. nanolog_like：`E[e_k]` 近似常数且小，`E[syscall_m]` 低，`C_m` 最小。
4. zephyr_deferred_like：前台 `C` 小，但高并发时等待项会转移到 `W_m(D,r)`。
5. ulog_async_like：前台 `C` 小，但后台文本格式化与后端写入使 `W_m(D,r)` 与 `C_m` 的常数项上升。
6. hilog_lite_like：通过限制格式能力降低 `E[e_k]` 与 `E[L_k]`，通常较 text/syslog 更轻。

### 4.5 多设备 `D` 的详细推导

#### 4.5.1 总时间分解

设每设备写 `r` 条：

`E_m(D,r) = A_spawn(D) + A_m + r C_m + W_m(D,r)`

其中 `W_m` 为竞争等待项（锁、队列、内核资源、I/O 饱和）。

#### 4.5.2 队列化近似（解释吞吐饱和）

把共享瓶颈抽象成单服务台，服务率 `μ_m`（records/s），到达率 `λ = D ν`（`ν` 为单设备提交率）。

稳定条件：

`ρ = λ/μ_m < 1`

等待时间近似随 `ρ` 增大而快速上升，可抽象为：

`W_m(D,r) ~ r * f_m(ρ),  f_m'(ρ) > 0,  f_m''(ρ) > 0`

因此当 `D` 接近

`D_sat,m ≈ μ_m / ν`

吞吐进入平台区；超过后出现明显回落。

#### 4.5.3 常用二次近似

在工程拟合中常写作：

`E_m(D,r) ≈ β_m + r C_m + α_m D + γ_m D^2`

总吞吐：

`Q_m(D,r) = 1000 D r / E_m(D,r)`

当 `γ_m > 0` 时，`D` 过大必然导致分母增长过快，出现峰值后下降。

进一步对 `D` 求导：

`dQ_m/dD ∝ (β_m + rC_m + α_m D + γ_m D^2) - D(α_m + 2γ_m D)`

整理得：

`dQ_m/dD ∝ β_m + rC_m - γ_m D^2`

故吞吐峰值点近似为：

`D_peak,m = sqrt((β_m + rC_m)/γ_m)`

这给出了可直接与实验扫描结果比对的“理论峰值设备数”。

#### 4.5.4 空间随 `D`

`S_m(D,r) = S_{0,m} + D r B_m`

因此空间对 `D` 严格线性，差异主要由斜率 `B_m` 与截距 `S_{0,m}` 决定。

### 4.6 模式间理论排序与可检验预测

#### 4.6.1 异语义比较（最短路径口径）

由单条代价可得典型关系：

`B_nano ~ B_z <= B_hilog < B_bin < B_sys ~ B_text <= B_ulog`

`C_nano` 常最小，`C_hilog` 次之；`C_z` 与 `C_ulog` 受队列/后台行为影响；`C_text/C_sys/ftrace` 受格式化与系统路径影响更大。

可检验预测：
1. nanolog_like 在时间与吞吐上常呈数量级优势。
2. hilog_lite_like 在“受限语义”下应明显优于 text/syslog。
3. ulog_async_like 在吞吐未饱和且后台轻负载时有机会受益；若后台文本后端成为瓶颈，可能慢于 binary/text。
4. 上述优势不代表语义等价下仍保持同倍率。

#### 4.6.2 语义拉齐比较（公平口径）

当统一为 `len + payload + crc` 并统一字段集时：

`B_sem(bin) ≈ B_sem(nano) ≈ B_sem(zephyr)`

`C_sem` 差异主要来自实现常数（编码循环、队列同步、缓存命中），不再是语义体量差异。

可检验预测：
1. 空间差异收敛到小常数级。
2. 时间差异收敛到同一量级，而非数量级差异。

### 4.7 与第5章指标的映射关系

为了让推导可被实验验证，建立公式到指标的映射：

1. `T_m(N)` 对应 `elapsed_ms` 或 `end_to_end_ms` 的均值趋势。
2. `Q_m` 对应 `throughput_rps` / `throughput_e2e_rps`。
3. `S_m` 对应 `total_bytes`。
4. 交叉点 `N*_{time}`, `N*_{space}` 对应“某场景下 binary 由劣转优”的阈值。
5. `D_sat,m` 对应“多设备扫描中吞吐峰值后回落”的拐点。

本章给出的是“由算法与数据结构直接推出的理论趋势”；第5章再代入实测数据检验这些趋势是否成立，以及偏离原因。

### 5.8 理论推导与实测对照

#### 5.8.1 空间模型对照

以扩展单机集 `N=100000` 为例（在此处才代入实例参数）：

1. `S_binary(N)=N*B_bin+S_shared`，实测 `3,411,447`，与线性模型一致。
2. `S_text(N)=N*B_text`，实测 `11,116,115`，与线性模型一致。
3. `S_syslog(N)=N*B_sys`，固定模板输入下与线性模型一致。

说明：binary/syslog 出现“0%误差”是因为当前基准输入模板与编码长度是确定性的，空间几乎无随机扰动；这不代表时间模型也会 0 误差。

#### 5.8.2 时间模型对照

1. 小样本（如 `d1_r500`）存在 binary 时间不占优情况，符合较大 `A_bin` 的预期。
2. 样本增大后（如 `d8_r2000`、`d12_r2000`）binary 时间优势转为稳定。
3. 高 `D` 下吞吐出现平台和回落（binary D=12 后回落），符合 `φ(D)` 上升导致的竞争退化。

#### 5.8.3 不一致点与原因

1. 不同平台的 syscall/内核路径差异显著，不能把“平台差异”误归因给“编码算法差异”。
2. 微秒级短任务受调度抖动影响大，IQR 过滤后仍会出现局部反直觉点。
3. 当记录条数较低时，固定准备成本与文件系统缓存行为会主导结果。

### 5.9 语义差异导致结果差异的机制解释

综合 5.7 与 5.8，可把差异拆解为：

1. **语义体量差异**：  
`nanolog_like/zephyr_deferred_like` 的记录字段和校验责任更小，单条字节更低。
2. **元数据与初始化差异**：  
optbinlog 有共享 schema 初始化与查找路径，这带来固定成本，但也带来跨进程一致性收益。
3. **并发模型差异**：  
Zephyr deferred 路径在本机负载下受队列锁与线程同步影响，语义拉齐后不一定比单线程批量编码更快。
4. **能力边界差异**：  
NanoLog 偏“极致热路径性能 + 离线恢复”，uLog 偏“异步线程与 backend 抽象”，HiLog Lite 偏“受限格式换低开销”，Optbinlog 偏“运行时可扩展 schema + 回放校验 + 多进程共享”。

因此论文中的结论应分两层表达：

1. 异语义层：展示不同系统设计目标下的性能上限差异。
2. 同语义层：展示等能力近似条件下的真实实现差异。

## 第6章 总结与后续计划

### 6.1 工作总结

本文完成了“调研 -> 设计 -> 理论 -> 实验 -> 对照”闭环：

1. 调研层：明确了国内外先进日志系统的共性优化手段（deferred、ring buffer、二进制结构化、编译期迁移热路径开销）。
2. 设计层：Optbinlog 通过共享格式与偏移访问实现跨进程一致性和低冗余写入。
3. 理论层：推导了 `N` 与 `D` 对时间/空间/吞吐的关系，并给出模式差异来源。
4. 实验层：多轮结果验证了“空间优势稳定、时间优势与样本规模/并发强相关”的核心结论，并新增了 NanoLog/Zephyr/uLog/HiLog Lite 的算法级复现与语义拉齐对比。

### 6.2 当前局限

1. 受网络与平台限制，尚未完成外部系统官方工程的全栈编译复现（当前为代码级阅读 + 算法级原型复现）。
2. Linux 实验仍以 VM 为主，尚未覆盖 Raspberry Pi / i.MX 实机。
3. 缺少 `perf` 级 CPU 微观指标，暂不能细粒度解释所有异常点。

### 6.3 下一步优化计划（对齐开题报告）

1. `P0`：补齐对照组闭环。  
口径统一为 `text / optbinlog-binary / syslog(journald) / ftrace`，可行时补 Android EventLog 设备侧。
2. `P1`：补齐平台覆盖。  
至少完成 x86_64 Linux 与 ARM Linux（Raspberry Pi / i.MX），并记录内核、文件系统、存储介质。
3. `P1`：补齐 CPU 指标。  
加入 `cycles`、`context-switch`、`cache-miss`，解释高并发退化。
4. `P2`：扩展容错实验。  
增加损坏帧、截断帧、恢复过程和校验失败路径统计。

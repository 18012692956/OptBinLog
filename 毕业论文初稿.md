## 摘  要

面向嵌入式与边缘设备，日志系统需要在可观测性、写入开销与并发一致性之间取得平衡。针对文本日志在高频场景下格式化成本高、空间冗余大以及多进程共享元数据易发生初始化竞争的问题，本文提出并实现 Optbinlog。该系统采用事件格式描述文件、共享元数据与二进制编码路径，将格式解析从热路径迁出，并通过偏移访问保证跨进程映射一致性。

本文围绕“机制分析—理论推导—分层实验验证”展开：首先调研 Linux ftrace、LTTng、Zephyr logging、systemd-journald、NanoLog、RT-Thread ulog 与 OpenHarmony HiLog Lite 等代表系统，并定位其关键优化路径；随后建立 `N`（日志条数）与 `D`（并发设备/节点数）驱动下的时间、空间与吞吐模型；最后在统一统计口径下完成工程近似层与严格公平层的对照验证。

实验数据由四个阶段构成：工程近似矩阵、严格语义对齐主套件（single、本地多设备 `D=5,10,20,50`）、真实多设备硬件 CRC32C 复测（`5/10/15/20` 节点）以及初始化竞争验证。结果表明：`binary` 相对 `text` 在本轮纳入语义组上均保持显著正收益；在严格公平口径下，Optbinlog 对 `ulog/hilog/syslog` 在三层场景总体稳定领先，对 `nanolog/zephyr` 在真实多设备中仍处负区；`ftrace` 在真实多设备中呈现“空间正收益、时间吞吐负收益”。新增空间扫描进一步显示，`ulog/hilog/syslog` 在 `N≈10` 即出现空间反超，而 `nanolog/zephyr` 在 `N<=100000` 仍未出现交叉点。初始化竞争实验稳定满足“单创建者、其余进程/节点复用已初始化共享文件”的正确性约束。理论趋势与实测结果总体一致。

关键词：嵌入式日志；二进制编码；共享内存；并发初始化；性能建模

## ABSTRACT

For embedded and edge systems, logging must balance observability with overhead (storage, CPU, and concurrent initialization). This thesis implements Optbinlog, a schema-shared binary logging approach for embedded deployment. Event metadata are parsed once, materialized in shared memory, and accessed via offsets, so runtime hot-path cost is reduced and cross-process mapping remains stable.

The work first surveys representative domestic and international logging systems (Linux ftrace, LTTng, Zephyr logging, systemd-journald, NanoLog, RT-Thread ulog, OpenHarmony HiLog Lite) and analyzes their optimization code paths. Then a theory-first model is built to derive how time/space/throughput scale with log count `N` and device count `D`. Finally, measured results are compared against the derived trends.

The retained dataset follows a stage-based workflow: engineering-approximate matrix rerun, strict-aligned suite (single and local multi-device scans at `D=5,10,20,50`), hardware-CRC32C real multi-node rerun (`5/10/15/20` nodes), and local/node-level initialization competition. Results show that `binary` consistently outperforms `text` across the included semantic groups. Under strict semantic alignment, Optbinlog remains stably ahead of `ulog/hilog/syslog`, while `nanolog/zephyr` still show persistent node-level penalties in real multi-node runs. In addition, `ftrace` in real multi-node runs shows a split pattern (space-positive but time/throughput-negative). The added space-scan experiment shows early crossover for `ulog/hilog/syslog` at around `N≈10`, while no crossover is observed for `nanolog/zephyr` up to `N=100000`. Initialization-race results satisfy the single-creator property consistently. Overall, theoretical trends and empirical observations are aligned.

Keywords: embedded logging; binary encoding; shared metadata; concurrency; performance modeling

## 1 绪论

### 1.1 研究背景

嵌入式系统日志常处于“高频写入、有限存储、多进程并发、实时约束”并存的环境。传统文本日志在可读性上有优势，但会带来：

1. 字符串格式化与解析开销高。
2. 单条日志冗余字节多，长期空间成本高。
3. 多进程共享格式信息时容易出现初始化竞争和一致性问题。

因此，本文并不局限于 `text` 与 `binary` 的局部比较，而是将问题置于“理论分析、实现机制与分组对照实验”统一框架下进行讨论。

### 1.2 研究问题

1. 现有日志系统的主流优化点是什么？是否仍有缺口？
2. Optbinlog 的实现路径能否覆盖这些缺口？
3. 从算法与数据结构出发，`N` 与 `D` 的时间/空间/吞吐趋势应如何推导？
4. 推导结论与实验结果哪些一致，哪些不一致，原因是什么？

### 1.3 本文工作

1. 完成国内外日志系统调研，并定位代表性优化代码入口。
2. 设计并实现 Optbinlog（事件格式解析、共享元数据、二进制读写、并发初始化状态机）。
3. 建立抽象数学模型，推导不同模式在 `N`、`D` 维度下的性能趋势。
4. 组织多轮实验（单机高负载、多设备模拟、真实多设备模拟（多台虚拟机节点）并发、init-race）并进行理论对照分析。

### 1.4 研究方法与工作中心

围绕前述研究问题，本文将工作中心放在三条主线上：

### 1.5 研究创新点

1. 提出“共享 schema + 偏移访问”的用户态二进制日志实现，重点解决多进程初始化一致性与重复解析问题。
2. 从算法与数据结构角度推导 `N` 与 `D` 对时间、空间和吞吐的影响，并给出可检验交叉点与饱和点表达式。
3. 建立“工程近似层 + 严格公平层”两级实验体系：前者保留真实系统路径差异，后者统一 schema、统一生成值与统一平台，避免把语义体量差或输入来源差误判为实现优劣。
4. 在真实多设备模拟（多台虚拟机节点）场景下验证初始化竞争与并发写入行为，形成可复现实验流程。

### 2.1 调研范围与方法

本章基于官方文档、论文与代码仓库进行调研，聚焦“热路径优化机制”和“可复现实现入口”。
为保证后续章节的横向可比性，调研统一按四个维度展开：数据结构组织、关键算法路径、并发行为特征与工程部署边界。

调研对象如下：
1. Linux ftrace / kernel ring buffer。
2. LTTng。
3. Zephyr logging subsystem。
4. systemd-journald。
5. NanoLog（USENIX ATC 2018）。
6. RT-Thread ulog。
7. OpenHarmony HiLog Lite。

### 2.2 国际代表系统与优化思路（代码级）

#### 2.2.1 Linux ftrace（内核追踪）

代码与文档显示其核心是 `per-CPU lockless ring buffer`。在 `ring_buffer.c` 中，写入路径围绕“预留 -> 填充 -> 提交”组织，关键是把竞争局部化到 CPU 本地缓冲，降低全局锁竞争。

1. 数据结构：每 CPU buffer + page 链式组织，维护 head/commit/tail。
2. 算法逻辑：优先本地写入、批量可见性提交，读写阶段解耦。
3. 场景定位：内核 tracing、高并发诊断；功能强但接入复杂度高于应用层日志库。

#### 2.2.2 LTTng（Linux Tracing Toolkit Next Generation）

LTTng 采用 tracepoint instrumentation + lockless buffer 管线，强调高吞吐追踪和离线分析。

1. 数据结构：channel/sub-buffer 等追踪缓冲结构。
2. 算法逻辑：将事件采集与后处理分层，追踪点开销可控。
3. 场景定位：系统级性能分析与可观测平台，部署和运维要求较高。

#### 2.2.3 Zephyr logging（RTOS 场景）

在 Zephyr `Kconfig.mode` 与 `log_msg.c` 可见，关键机制是 deferred logging。

1. 数据结构：`log_msg` 对象 + 消息池（MPSC packet buffer）。
2. 算法逻辑：前台线程只做封包入队，后台线程统一输出；支持 overflow/drop 策略。
3. 场景定位：RTOS 中实时任务优先，要求前台日志路径尽量短、可控。

#### 2.2.4 systemd-journald

其 `JOURNAL_FILE_FORMAT` 与 `journald-server` 路径强调二进制对象与索引。

1. 数据结构：header + objects + 哈希/索引。
2. 算法逻辑：写入与检索并重，保证一致性与查询能力。
3. 场景定位：通用 Linux 系统服务日志；对轻量嵌入式而言依赖偏重。

#### 2.2.5 NanoLog（研究型高性能日志）

从 `NanoLogCpp17.h`、`RuntimeLogger.cc` 可见，NanoLog 核心是“编译期提取格式 + 运行期仅记录动态参数”。

1. 数据结构：staging buffer、runtime logger 队列/缓冲。
2. 算法逻辑：前台路径尽量只做参数打包，压缩与格式恢复转移到后台/离线。
3. 场景定位：超高频日志、允许离线恢复文本的性能敏感场景。

### 2.3 国内生态系统与优化思路（代码级）

#### 2.3.1 RT-Thread ulog

`ulog.c` 提供异步输出和多 backend 抽象，思路是“前台快写缓冲，后台慢速输出”。

1. 数据结构：ringbuffer/rbb、backend 列表。
2. 算法逻辑：异步线程统一调度输出，减轻业务线程阻塞。
3. 场景定位：RT-Thread 设备侧调试与在线日志。

#### 2.3.2 OpenHarmony HiLog Lite

HiLog Lite 在格式能力上做约束（减少复杂格式），换取更低资源开销。

1. 数据结构：轻量日志缓冲和简化格式存储。
2. 算法逻辑：限制格式复杂度，降低格式化路径成本。
3. 场景定位：资源受限设备的系统日志基础设施。

### 2.6 现有方案缺陷归纳

从嵌入式角度可抽象出四类常见缺口：

1. **格式化开销仍在热路径**：text/syslog 在高频场景下受字符串格式化与内核接口调用影响明显。
2. **元数据复用不足**：多进程系统若每进程重复解析格式，启动和内存成本上升。
3. **并发初始化一致性风险**：共享资源初次创建缺少状态机时，容易出现竞态。
4. **跨平台复现实验缺失**：只在单平台得到的结论容易失真，外推边界较弱。

### 2.7 Optbinlog 对缺陷的对应关系

1. 用“事件格式文件 + 二进制编码”替代热路径字符串格式化。
2. 用共享元数据文件持久化 tag/element 描述，进程间偏移复用。
3. 用 `INITIALIZING -> INITIALIZED` 状态机控制并发初始化。
4. 用统一脚本输出 mean/median/p95/std/95%CI 与 IQR 过滤结果，支持多轮迭代对比。

### 2.8 复现边界与结论适用范围

## 3 Optbinlog 设计迭代与工程实现

### 3.1 设计目标

1. 降低单条日志热路径开销（时间）。
2. 降低长期写入空间消耗（空间）。
3. 支持多进程共享元数据与安全初始化（并发）。
4. 对齐嵌入式可部署需求（工程可落地）。

上述目标在后续章节中分别对应为可测指标：时间目标映射到 `elapsed/end_to_end`，空间目标映射到 `total_bytes`，并发目标映射到初始化竞争正确性与节点级稳定性，工程目标映射到跨场景复现与部署可迁移性。

### 3.2 事件格式与元数据结构

当前 Optbinlog 采用“共享元数据文件 + 进程内共享视图缓存 + 自描述二进制帧 + `CRC32C(hw)` + string-heavy 场景自动 `varstr`”的实现方式。

事件定义采用 `(name|type|bits)` 形式，字段类型简化为 `L/D/S`：

1. `L`：定长整数，按 schema 指定字节数写入。
2. `D`：双精度浮点，固定写入 8 字节。
3. `S`：字符串字段；当前实现同时支持固定宽度字符串编码和 `varstr` 编码。

共享元数据文件内仍由以下结构组织：

1. `OptbinlogSharedTag`：共享文件头与初始化状态。
2. `OptbinlogBitmap`：tag 合法性位图。
3. `OptbinlogEventTag`：事件定义入口。
4. `OptbinlogEventTagEle`：元素类型与长度描述。

在共享文件之上，系统进一步引入了**进程内共享视图缓存层**。该缓存按共享文件的 `dev/inode/size/mtime` 判定是否可复用，并缓存：

1. 已映射的共享文件视图。
2. 每个 tag 的 `OptbinlogTagCacheEntry`。
3. schema 级统计量，例如 tag 数、总 payload 字节数、字符串字段总宽度、字符串字段数和最大字符串字段宽度。

这些统计量会直接参与写路径中的 `varstr` 自动决策，因此它们属于当前热路径优化的一部分。

### 3.3 写入/读取路径

写入与读取路径遵循“先判定元数据状态、再执行编码/解码、最后进行一致性校验”的统一流程。

写入路径：
1. 通过 `acquire_shared_view()` 获取共享文件视图；若共享文件未变化，则直接复用进程内缓存的 tag cache。
2. 根据 schema 统计量判断是否启用 `varstr`；当字符串字段占比高或最大定长字符串字段较大时，默认切换到 `varstr`。
3. 依据 tag cache 顺序编码 payload。
4. 构造 4 字节自描述帧头，其中同时编码 `payload_len`、`varstr` 标志位和校验类型。
5. 对单条记录计算校验值；默认采用 `CRC32C(hw)`，仅在硬件不可用时回退到软件实现。
6. 将 `frame_header + payload + checksum` 写入用户态缓冲区，按批次落盘。

读取路径：
1. 读取 4 字节帧头，并解析 `payload_len`、`varstr` 标志位与 `checksum_type`。
2. 读取 payload 与校验字段，并按帧头声明的校验类型执行校验。
3. 按共享格式解码：整数和浮点按固定宽度读取，字符串则根据 `varstr` 标志选择“长度前缀 + 数据”或“固定宽度截断/补零”方式。
4. 将解码后的记录交给统一回调接口。

因此，当前记录格式可写作：

$$
\mathrm{frame} = \mathrm{frame\_header} \;\Vert\; \mathrm{payload} \;\Vert\; \mathrm{checksum}
$$

其中：

$$
\mathrm{frame\_header}
=
\mathrm{payload\_len}
\;\cup\;
\mathrm{varstr\_bit}
\;\cup\;
\mathrm{checksum\_type}
$$

### 3.4 关键迭代

1. 从“每事件独立结构”迭代为“统一结构 + 位图 + 偏移表”。
2. 从进程内指针共享迭代为“偏移量共享”（解决跨进程映射基址差异）。
3. 从 switch-case 查找迭代为 `arr=id/100, idx=id%100` 的常数级路径。
4. 从软件 `CRC32` 迭代为默认 `CRC32C(hw)`；在 ARMv8 与 SSE4.2 平台优先使用硬件指令，软件实现只作为回退路径。
5. 新增自描述帧头，使 `CRC32 / CRC32C / none` 与 `fixed-string / varstr` 可以在同一读取逻辑中兼容。
6. 新增进程内共享视图缓存和 tag cache，把共享文件 reopen、tag 遍历与 schema 统计从稳态热路径迁出。
7. 新增 schema 级自动 `varstr` 策略：当字符串字段占比高时，从固定宽度字符串编码切换到“长度前缀 + 实际长度”编码。
8. 增加并发初始化状态机和 init-race 压力验证，保证共享文件只被单创建者初始化，其余进程或节点复用既有元数据。

综合以上迭代，当前实现可归纳为三条关键收敛点：

1. **校验语义与热路径代价兼顾**：系统保留每条记录校验，同时通过 `CRC32C(hw)` 将校验成本压缩到更适合热路径的水平。
2. **共享元数据的稳态收益进入真实路径**：共享视图缓存和 tag cache 使 schema 解析与 tag 遍历不再重复进入每次写入。
3. **字符串主导 schema 的冗余写入被抑制**：在 string-heavy 场景下自动启用 `varstr`，把固定宽度补零和无效拷贝降到更低。

### 3.5 与实际场景的匹配性

1. 小样本 `N`：共享文件打开、共享视图校验、文件打开和缓冲准备等固定项占比仍然较高，时间优势不一定立刻出现。
2. 中大样本 `N`：固定项被摊薄，自描述帧头和 `CRC32C(hw)` 的额外代价收敛到常数级，binary 的时间和空间优势更稳定。
3. string-heavy schema：自动 `varstr` 会显著减少补零写入和冗余字节，因此空间优势更容易显现。
4. 高并发 `D`：即使校验和 schema 解析成本被压低，调度、文件竞争和写出路径仍会主导吞吐峰值与回落点。
5. 多进程/多节点部署：共享 schema、进程内缓存复用和节点级初始化竞争正确性已经形成闭环，因此该设计更贴近真实工程部署。

因此，第4章将把上述实现行为进一步抽象为 `A_m/C_m/B_m/W_m` 等参数形式，便于与第5章实验结果逐项对照。

### 3.6 工程化读取能力补全

为提升实验数据复核效率，本文在写入与解码库之外补充了 `optbinlog_read` 工具，用于“共享格式文件 + 二进制日志文件”的直接可视化读取。该工具具备三项能力：
1. 按 tag 与字段名输出记录，避免只看到匿名值序列。
2. 支持 `table/jsonl` 两种格式，便于命令行排错与自动化分析脚本复用。
3. 支持 `--limit` 与 `--summary`，可快速截取样本并统计各 tag 分布。

该工具把“论文中的统计结果”与“底层二进制记录”建立了可追溯链路，使实验结论不只停留在汇总图表层，而能够直接回溯到具体记录级证据。

## 4 理论模型与数学推导

本章先做抽象推导，不代入具体事件文件数值。
为保证推导可解释且可验证，本文采用以下前提：在单次统计窗口内事件分布近似稳定；各模式在同一场景下使用一致的输入语义；模型参数以期望意义解释而非单点极值解释。

### 4.1 符号定义

1. $N$：总日志条数。
2. $D$：并发设备数（或并发进程数）。
3. $r$：每设备日志条数，满足 $N = D \cdot r$。
4. $k$：事件类型（tag）索引，$k \in \{1,\dots,T\}$。
5. $p_k$：事件类型 $k$ 的概率，满足 $\sum_k p_k = 1$。
6. $e_k$：事件 $k$ 的字段数。
7. $l_{k,j}$：事件 $k$ 第 $j$ 个字段编码字节数。
8. $L_k = \sum_{j=1}^{e_k} l_{k,j}$：事件 $k$ 的有效载荷字节数。
9. $A_m$：模式 $m$ 的固定时间开销（初始化、打开文件、预热）。
10. $C_m$：模式 $m$ 的单条边际时间开销。
11. $B_m$：模式 $m$ 的单条边际空间开销。
12. $S_{0,m}$：模式 $m$ 的固定空间开销（例如共享元数据）。
13. $Q_m$：吞吐（records/s）。

### 4.2 从算法与数据结构到单条代价

为避免“只看结果不看机制”，先把每种模式拆成操作级成本。以下表达式均以单条记录期望开销为核心，再扩展到总量关系。

#### 4.2.1 text

单条记录包含：
1. 读取输入字段。
2. `snprintf`/拼接格式串。
3. `write` 到文件。

时间可写为：

$$
t_{\text{text}}(k) = t_{\text{parse}}(k) + t_{\text{fmt}}(k) + t_{\text{write}}(\mathrm{bytes}_{\text{text}}(k))
$$

空间可写为：

$$
b_{\text{text}}(k) = b_{\text{prefix}} + b_{\text{delim}} + b_{\text{num}}(k) + b_{\text{str}}(k)
$$

其中 $b_{\text{num}}(k)$ 是数字转十进制后的字符长度，随数值大小缓慢变化。

#### 4.2.2 binary（optbinlog）

单条记录包含：
1. tag 查找（数组/偏移访问）。
2. 按字段类型编码 payload。
3. 计算校验值。
4. 写帧 `len + payload + crc`。

时间：

$$
t_{\text{bin}}(k) = t_{\text{lookup}} + t_{\text{encode}}(k) + t_{\text{checksum}}(L_k) + t_{\text{write}}(4 + h_{\text{bin}} + L_k + 4)
$$

空间：

$$
b_{\text{bin}}(k) = 4 + h_{\text{bin}} + L_k + 4
$$

固定空间：

$$
S_{0,\text{bin}} = S_{\text{shared}}
$$

若 schema 触发 `varstr`，则字符串字段的有效载荷不再按固定宽度写出，而是写为“长度前缀 + 实际长度”：

$$
L_k^{(\text{varstr})}
=
\sum_{j \in \mathcal{N}_k} l_{k,j}
\;+\;
\sum_{j \in \mathcal{S}_k} \left( 2 + \hat{l}_{k,j} \right),
\qquad
\hat{l}_{k,j} \le l_{k,j}
$$

#### 4.2.3 syslog / ftrace

抽象为“系统日志接口 + 内核路径”：

$$
t_{\text{sys}}(k) = t_{\text{format\_sys}}(k) + t_{\text{syscall}} + t_{\text{kernel\_path}} + t_{\text{write\_kernel}}
$$

$$
t_{\text{ftrace}}(k) = t_{\text{format\_ftrace}}(k) + t_{\text{trace\_write}} + t_{\text{ring\_or\_file\_path}}
$$

空间：

$$
b_{\text{sys}}(k) = b_{\text{sys\_header}} + b_{\text{sys\_payload}}(k)
$$

$$
b_{\text{ftrace}}(k) = b_{\text{ftrace\_header}} + b_{\text{ftrace\_payload}}(k)
$$

#### 4.2.4 nanolog_like / zephyr_deferred_like / ulog_async_like / hilog_lite_like

`nanolog_like`（固定字段二进制）：

$$
t_{\text{nano}} = t_{\text{pack\_fixed}} + t_{\text{memcpy}} + t_{\text{write}}
$$

$$
b_{\text{nano}} = \text{const}
$$

`zephyr_deferred_like`（前台入队 + 后台批刷）：

$$
t_{z,\text{front}} = t_{\text{enqueue}} + t_{\text{signal}}
$$

$$
t_{z,\text{back}} = \frac{t_{\text{dequeue}} + t_{\text{batch\_write}}(k_b)}{k_b}
$$

$$
t_z = t_{z,\text{front}} + t_{z,\text{back}}
$$

$$
b_z = b_{\text{queue\_node}} + b_{\text{fixed}} + \frac{b_{\text{batch\_meta}}}{k_b}
$$

`ulog_async_like`（异步队列 + 文本 backend）：

$$
t_{\text{ulog}} =
t_{\text{enqueue}} + t_{\text{cond\_signal}}
+
\frac{t_{\text{dequeue}} + t_{\text{format\_text}} + t_{\text{backend\_write}}}{k_b}
$$

$$
b_{\text{ulog}} = b_{\text{queue\_node}} + b_{\text{text\_payload}}
$$

`hilog_lite_like`（受限字段 + 轻量头）：

$$
t_{\text{hilog}} = t_{\text{compact\_format}} + t_{\text{write}}(h_{\text{lite}} + p_{\text{lite}})
$$

$$
b_{\text{hilog}} = b_{\text{header\_lite}} + b_{\text{payload\_lite}}
$$

### 4.3 空间随 `N` 的详细推导

#### 4.3.1 期望单条字节

对任意模式 $m$，定义：

$$
B_m = \mathbb{E}[b_m(k)] = \sum_k p_k \, b_m(k)
$$

则总空间：

$$
S_m(N) = S_{0,m} + N B_m
$$

这是严格的一阶线性模型，斜率为 $B_m$，截距为 $S_{0,m}$。

#### 4.3.2 binary 的显式表达

由 4.2.2 得：

$$
b_{\text{bin}}(k) = 8 + h_{\text{bin}} + L_k
$$

$$
B_{\text{bin}} = 8 + h_{\text{bin}} + \sum_k p_k L_k
$$

$$
S_{\text{bin}}(N) = S_{\text{shared}} + N \left(8 + h_{\text{bin}} + \sum_k p_k L_k\right)
$$

#### 4.3.3 节省率与导数

binary 相对模式 $x$ 的空间节省率：

$$
\eta_{\text{bin}/x}(N) = 1 - \frac{S_{\text{bin}}(N)}{S_x(N)}
$$

若 $S_{0,x}=0$，则

$$
\eta_{\text{bin}/x}(N) = 1 - \frac{B_{\text{bin}} + S_{\text{shared}}/N}{B_x}
$$

求导：

$$
\frac{d\eta_{\text{bin}/x}}{dN} = \frac{S_{\text{shared}}}{N^2 B_x} > 0
$$

说明 $N$ 越大，$S_{\text{shared}}/N$ 越小，binary 空间优势越稳定。

#### 4.3.4 空间交叉点

当比较 $\text{bin}$ 与 $x$：

$$
S_{\text{bin}}(N) \le S_x(N)
$$

$$
S_{\text{shared}} + N B_{\text{bin}} \le S_{0,x} + N B_x
$$

若 $B_x > B_{\text{bin}}$，可得交叉点：

$$
N^*_{\text{space}}(\text{bin},x) = \frac{S_{\text{shared}} - S_{0,x}}{B_x - B_{\text{bin}}}
$$

当 $N > N^*_{\text{space}}$ 时，binary 空间一定优于 $x$。

### 4.4 时间随 `N` 的详细推导

#### 4.4.1 从单条时间到总时间

定义：

$$
C_m = \mathbb{E}[t_m(k)] = \sum_k p_k \, t_m(k)
$$

则总时间：

$$
T_m(N) = A_m + N C_m
$$

该形式直接对应实验中的 `elapsed/write_only/end_to_end` 指标。

#### 4.4.2 吞吐公式与单调性

$$
Q_m(N) = \frac{1000N}{T_m(N)} = \frac{1000}{C_m + A_m/N}
$$

对 $N$ 求导：

$$
\frac{dQ_m}{dN} = \frac{1000A_m}{N^2\left(C_m + A_m/N\right)^2} > 0
$$

说明在模型中吞吐随 $N$ 单调上升并渐近：

$$
\lim_{N \to \infty} Q_m(N) = \frac{1000}{C_m}
$$

在 $A_m>0$ 与 $C_m>0$ 条件下，吞吐增速会随 $N$ 增大而下降，表现为“先快后慢”的收益递减，这与实验中大 $N$ 后增益变缓一致。

#### 4.4.3 时间交叉点（binary 何时优于其他模式）

比较 $\text{bin}$ 与 $x$：

$$
T_{\text{bin}}(N) \le T_x(N)
$$

$$
A_{\text{bin}} + N C_{\text{bin}} \le A_x + N C_x
$$

当 $C_x > C_{\text{bin}}$ 时：

$$
N^*_{\text{time}}(\text{bin},x) = \frac{A_{\text{bin}} - A_x}{C_x - C_{\text{bin}}}
$$

解释：
1. 若 $A_{\text{bin}}$ 较大（共享初始化重），小 $N$ 可能不占优。
2. 随 $N$ 增大，$NC$ 项主导，只要 $C_{\text{bin}} < C_x$，最终会跨过交叉点。

#### 4.4.4 `C_m` 的字段复杂度展开

为把“常数项差异”写清楚，可把边际开销展开到字段数级别：

$$
C_m = c_{m,0} + c_{m,1}\mathbb{E}[e_k] + c_{m,2}\mathbb{E}[L_k] + c_{m,3}\mathbb{E}[\mathrm{syscall}_m]
$$

其中：
1. $\mathbb{E}[e_k]$ 决定字段遍历成本（编码循环次数）。
2. $\mathbb{E}[L_k]$ 决定 memcpy、校验和写入字节相关成本。
3. $\mathbb{E}[\mathrm{syscall}_m]$ 反映模式是否强依赖系统调用路径。

可得典型关系：
1. text/syslog：$c_{m,1}$ 与 $c_{m,3}$ 较大（格式化 + syscall）。
2. binary：$c_{m,1}$ 中等，$c_{m,2}$ 受字段编码与校验影响，另有 $A_{\text{bin}}$ 较大（共享文件和缓存准备）。
3. nanolog_like：$\mathbb{E}[e_k]$ 近似常数且小，$\mathbb{E}[\mathrm{syscall}_m]$ 低，$C_m$ 最小。
4. zephyr_deferred_like：前台 $C$ 小，但高并发时等待项会转移到 $W_m(D,r)$。
5. ulog_async_like：前台 $C$ 小，但后台文本格式化与后端写入使 $W_m(D,r)$ 与 $C_m$ 的常数项上升。
6. hilog_lite_like：通过限制格式能力降低 $\mathbb{E}[e_k]$ 与 $\mathbb{E}[L_k]$，通常较 text/syslog 更轻。

### 4.5 多设备 `D` 的详细推导

#### 4.5.1 总时间分解

设每设备写 `r` 条：

$$
E_m(D,r) = A_{\text{spawn}}(D) + A_m + rC_m + W_m(D,r)
$$

其中 $W_m$ 为竞争等待项（锁、队列、内核资源、I/O 饱和）。

#### 4.5.2 队列化近似（解释吞吐饱和）

把共享瓶颈抽象成单服务台，服务率 $\mu_m$（records/s），到达率 $\lambda = D\nu$（$\nu$ 为单设备提交率）。

稳定条件：

$$
\rho = \frac{\lambda}{\mu_m} < 1
$$

等待时间近似随 $\rho$ 增大而快速上升，可抽象为：

$$
W_m(D,r) \sim r f_m(\rho),
\qquad
f_m'(\rho) > 0,
\qquad
f_m''(\rho) > 0
$$

因此当 $D$ 接近

$$
D_{\text{sat},m} \approx \frac{\mu_m}{\nu}
$$

吞吐进入平台区；超过后出现明显回落。

#### 4.5.3 常用二次近似

在工程拟合中常写作：

$$
E_m(D,r) \approx \beta_m + rC_m + \alpha_m D + \gamma_m D^2
$$

总吞吐：

$$
Q_m(D,r) = \frac{1000Dr}{E_m(D,r)}
$$

当 $\gamma_m > 0$ 时，$D$ 过大必然导致分母增长过快，出现峰值后下降。

进一步对 $D$ 求导：

$$
\frac{dQ_m}{dD}
\propto
\left(\beta_m + rC_m + \alpha_m D + \gamma_m D^2\right)
-
D\left(\alpha_m + 2\gamma_m D\right)
$$

整理得：

$$
\frac{dQ_m}{dD} \propto \beta_m + rC_m - \gamma_m D^2
$$

故吞吐峰值点近似为：

$$
D_{\text{peak},m} = \sqrt{\frac{\beta_m + rC_m}{\gamma_m}}
$$

这给出了可直接与实验扫描结果比对的“理论峰值设备数”。

#### 4.5.4 空间随 `D`

$$
S_m(D,r) = S_{0,m} + DrB_m
$$

因此空间对 $D$ 严格线性，差异主要由斜率 $B_m$ 与截距 $S_{0,m}$ 决定。

### 4.6 模式间理论排序与可检验预测

#### 4.6.1 异语义比较（最短路径口径）

由单条代价可得典型关系：

$$
B_{\text{nano}} \sim B_z \lesssim B_{\text{hilog}} < B_{\text{bin}} < B_{\text{sys}} \sim B_{\text{text}} \lesssim B_{\text{ulog}}
$$

$C_{\text{nano}}$ 常最小，$C_{\text{hilog}}$ 次之；$C_z$ 与 $C_{\text{ulog}}$ 受队列和后台行为影响；$C_{\text{text}}/C_{\text{sys}}/C_{\text{ftrace}}$ 受格式化与系统路径影响更大。

可检验预测：
1. nanolog_like 在时间与吞吐上常呈数量级优势。
2. hilog_lite_like 在“受限语义”下应明显优于 text/syslog。
3. ulog_async_like 在吞吐未饱和且后台轻负载时有机会受益；若后台文本后端成为瓶颈，可能慢于 binary/text。
4. 上述优势不代表语义等价下仍保持同倍率。

#### 4.6.2 语义对齐比较（公平口径）

当统一为 `len + payload + crc` 并统一字段集时：

$$
B_{\text{sem}}(\text{bin}) \approx B_{\text{sem}}(\text{nano}) \approx B_{\text{sem}}(\text{zephyr})
$$

$C_{\text{sem}}$ 的差异主要来自实现常数（编码循环、队列同步、缓存命中和校验实现），不再是语义体量差异。

可检验预测：
1. 空间差异收敛到小常数级。
2. 时间差异收敛到同一量级，而非数量级差异。

### 4.7 与第5章指标的映射关系

为了让推导可被实验验证，建立公式到指标的映射：

1. $T_m(N)$ 对应 `elapsed_ms` 或 `end_to_end_ms` 的均值趋势。
2. $Q_m$ 对应 `throughput_rps` / `throughput_e2e_rps`。
3. $S_m$ 对应 `total_bytes`。
4. $N^*_{\text{time}}$ 与 $N^*_{\text{space}}$ 对应“某场景下 binary 由劣转优”的阈值。
5. $D_{\text{sat},m}$ 对应“多设备扫描中吞吐峰值后回落”的拐点。

本章给出的是“由算法与数据结构直接推出的理论趋势”；第5章再代入实测数据检验这些趋势是否成立，以及偏离原因。

### 4.8 本章小结

本章完成了三项工作：一是把不同日志模式统一到可比较的时间与空间参数形式；二是给出 `N` 与 `D` 两个核心自变量下的交叉点、峰值点与线性斜率含义；三是建立理论参数与实验指标的映射关系。基于这些结果，第5章将按“单机 -> 本地多设备 -> 真实多设备模拟 -> 日志量扫描”的顺序进行逐层验证。

### 5.5 日志量扩展：空间交叉点实验

### 5.6 理论—实证对照

综合 5.2～5.5 与第4章模型，可得到五点结论：

1. 空间模型与实测一致：`total_bytes` 随 `N` 线性增长，不同语义组对应不同 `B_m`。在 `nanolog/zephyr` 组中，binary 相对 peer 的空间仍为负；在 `ulog/hilog/syslog` 组中，binary 在 single、multi 与 l1 三层口径下均为正收益；在 l1 的 `ftrace` 组中，binary 空间同样保持稳定正收益。
2. 时间交叉点是“按语义组出现”的，而不是“按模式统一出现”的。`ulog/hilog/syslog` 在 single 与 multi 场景总体保持正收益；`zephyr` 在本地与真实多节点场景长期负区；`nanolog` 在 single 转正但在 multi/l1 波动并多点为负；`ftrace` 在 l1 时间维度持续深负。
3. 竞争项 `W_m(D,r)` 在本地多设备与真实多设备场景中都可被直接观察到。本地多设备出现的“点位振荡”与真实多设备出现的“持续负区/持续正区”，共同说明竞争项与冷路径项会在不同平台层级被放大为不同形态：前者更像局部抖动，后者更像结构性偏置。
4. “真实多设备”不是简单复制“本地多设备”。L1 场景额外引入了节点级冷启动、VM 调度噪声和每节点新进程启动，因此 `A_m` 的占比会明显抬升；这正是 `zephyr/nanolog` 在 L1 更难转正、`ulog/hilog/syslog` 在三指标上仍可保持正收益，以及 `ftrace` 呈现“空间优先但时延劣势”的主要原因。
5. 日志量增长实验进一步给出了空间交叉点阈值：`ulog/hilog/syslog` 在 `N≈10` 附近即可完成 `binary <= peer` 的空间反超，而 `nanolog/zephyr` 在 `N<=100000` 仍未出现交叉点，说明“空间何时反超”同样是语义组相关结论。

### 5.8 证据链组织与外推边界

为了让结论形成“可复核、可解释、可外推”的证据链，本文将数据集按用途组织为三类：
1. **工程近似层**：保留 `syslog/ftrace` 以及 `*_like` 路径，观察真实工程风格系统的整体代价排序。
2. **严格公平层**：统一使用 `text_semantic_like / binary / peer semantic_like`，在相同 schema、相同字段值和相同平台上报告单机、本地多设备与真实多设备结果。
3. **原因分析层**：保留 `binary_hotpath / binary_nocrc / binary_varstr` 等消融模式，用于解释当前实现中的主要时间来源和空间来源。

其中，工程近似层回答“真实系统路径如何排序”，严格公平层回答“同语义条件下 Optbinlog 与 peer 如何分化”，原因分析层回答“差异主要来自哪些结构性开销”。三类数据分工明确，可避免把部署差异、语义差异与实现差异混写在同一结论中。

因此，严格公平层主结论由 `nanolog/zephyr/ulog/hilog/syslog` 在 single/multi/l1 三层结果与 `ftrace` 在 l1 结果共同支撑；工程近似层仅承担工程部署视角的补充说明。

### 5.9 机制解释与应用启示

结合工程近似对照、严格对齐套件和消融结果，可将 `Optbinlog` 的差异点归纳为三层：
1. **文本扩展型语义组**：`ulog` 的主要矛盾是文本扩展与记录膨胀，因此 binary 在 single、local multi 和 real multi-node 三层场景中都保持稳定优势；`hilog` 在 single 与真实多设备中也保持明显正收益，但在本地高设备数点位会被短记录热路径压缩到近似持平。这仍然是 Optbinlog 当前最稳的受益区间。
2. **极短热路径语义组**：`nanolog/zephyr` 已经把热路径压得很短。对于这类 peer，Optbinlog 的剩余通用开销主要来自 schema 访问、字段遍历、frame 组织和冷启动，因此本地多设备与真实多设备都更容易出现负区或振荡；空间扫描也显示这两组在 `N<=100000` 范围内仍不存在交叉点。
3. **平台与并发层**：进入真实多设备后，`W_m(D,r)` 和冷启动项会把细小常数项放大，因此“单机反超”不应被直接外推为“节点级并发立即反超”。`syslog` 在该层体现为“三指标均可转正”，`ftrace` 体现为“空间正收益但时间吞吐深负”，说明平台路径差异会直接改变三指标的收益方向。

Optbinlog 的应用优势主要体现在：
1. **跨进程一致性**：共享 schema + 偏移访问保证多进程解码一致，适合网关/边缘设备的多服务部署。
2. **工程可迁移性**：不依赖编译期代码生成链路，新增日志事件只需更新 schema 文件，适合迭代频繁的业务系统。
3. **可验证性**：二进制帧包含长度与 per-record CRC32C，损坏检测与回归测试路径清晰，便于工程质量控制。
4. **对文本型 peer 的稳定优势**：`ulog/hilog` 三层场景的对齐结果表明，binary 对文本扩展更重的 peer 风格路径具有稳定时间、空间和吞吐优势。
5. **优化路径清晰**：实验已经把剩余开销收敛到 `cold-path + 通用打包路径 + 稳态缓存收益释放不足`，后续优化可以围绕长生命周期进程、steady-state fast path 和批量提交策略展开。

完整分析表、最新可视化和关键结论汇总见严格对齐套件生成的报告；工程近似层的补充说明见实验补充分析报告。

## 6 结论与展望

### 6.1 研究结论

本文形成了“调研 -> 设计 -> 建模 -> 验证 -> 解释”的完整研究链条：

1. 调研层：识别了代表系统的共性优化机制（deferred、ring buffer、结构化二进制、热路径迁移）及其适用边界。
2. 设计层：Optbinlog 通过共享 schema 与偏移访问实现跨进程一致性，并以自描述帧与硬件 CRC32C 保持可校验性。
3. 理论层：建立了 `N`、`D` 与时间/空间/吞吐之间的关系模型，给出可检验的交叉点与饱和点预测。
4. 实验层：在 single、本地多设备与真实多设备模拟三层场景完成严格语义对齐验证，并通过初始化竞争和消融实验解释差异来源。

从结果上看，`binary` 相对 `text` 在本轮纳入的各语义组上均为正收益；对 `ulog/hilog/syslog` 的优势在三层场景中总体稳定；`ftrace` 在真实多设备中表现为“空间正收益、时间吞吐负收益”。空间扫描显示 `ulog/hilog/syslog` 在 `N≈10` 即出现反超，而 `nanolog/zephyr` 在 `N<=100000` 仍未出现空间交叉。上述结论与第 4 章模型保持一致，说明 Optbinlog 更适合“长期在线、版本持续演进、强调跨进程一致性”的嵌入式网关与边缘控制器场景。

### 6.2 当前局限

1. 受网络与平台限制，尚未完成外部系统官方工程的全栈编译复现（当前为代码级阅读 + 算法级原型复现）。
2. Linux 实验仍以 VM 为主，尚未覆盖 Raspberry Pi / i.MX 实机。
3. 严格公平语义套件已经扩展到 20 节点 VM 级“真实多设备”模拟，但仍未迁移到真实 ARM 板卡或 bare-metal Linux 实机。
4. 当前严格公平套件采用“新进程 + 单次写入”统计口径，因此新加入的进程内 shared/tag cache 收益只在代码层完成接入，尚未通过长生命周期 steady-state 基准充分量化。
5. 缺少 `perf` 级 CPU 微观指标，暂不能细粒度解释所有异常点。

### 6.3 未来展望

未来工作可从以下方向持续推进：
1. 在更广泛的平台与负载条件下开展验证，进一步提升结论的外部有效性与可迁移性。
2. 在保持语义一致性的前提下继续优化热路径与稳态路径，缩小与极短路径 peer 在特定场景下的差距。
3. 完善日志可靠性与可恢复性评估，形成覆盖损坏、截断与异常恢复过程的系统化证据。
4. 结合更细粒度观测手段（如 CPU 与调度行为分析），建立从机制到性能现象的更强解释闭环。

## 摘  要

面向嵌入式与边缘设备，日志系统需要在可观测性、写入开销与并发一致性之间取得平衡。针对文本日志在高频场景下格式化成本高、空间冗余大以及多进程共享元数据易发生初始化竞争的问题，本文提出并实现 Optbinlog。该系统采用事件格式描述文件、共享元数据与二进制编码路径，将格式解析从热路径迁出，并通过偏移访问保证跨进程映射一致性。

本文围绕“设计机制—理论推导—实证验证”展开：先从日志热路径与并发初始化问题出发设计 Optbinlog，再建立 `N`（日志条数）与 `D`（并发设备/节点数）驱动下的时间、空间与吞吐模型，最后在统一统计口径下进行严格语义对齐验证。

本文重点报告最新一版结果：single、本地多设备 `D=5,10,20,50`、真实多设备 `5/10/15/20` 节点（硬件 `CRC32C`）及空间交叉点扫描（`N=1～100000`）。结果表明：`binary` 相对 `text` 在全部语义组上保持稳定正收益；在“Optbinlog 对齐 peer 语义、peer 保持原生存储路径”的严格公平口径下，`binary` 在真实多设备中对 `syslog/ulog` 保持稳定综合优势，对 `hilog` 呈现“空间占优、时间略优、吞吐近持平”的中间形态，对 `nanolog/zephyr` 仍未完成节点级反超，`ftrace` 则表现为“空间正收益、时间吞吐负收益”。空间扫描显示 `syslog/ulog` 在 `N≈10` 出现反超，`hilog` 在 `N≈50` 左右出现反超，`nanolog/zephyr` 在 `N<=100000` 仍无交叉。初始化竞争实验持续满足“单创建者、其余进程或节点复用共享文件”的正确性约束。理论趋势与实测结果总体一致。

关键词：嵌入式日志；二进制编码；共享内存；并发初始化；性能建模

## ABSTRACT

For embedded and edge systems, logging must balance observability with overhead (storage, CPU, and concurrent initialization). This thesis implements Optbinlog, a schema-shared binary logging approach for embedded deployment. Event metadata are parsed once, materialized in shared memory, and accessed via offsets, so runtime hot-path cost is reduced and cross-process mapping remains stable.

The work first surveys representative domestic and international logging systems (Linux ftrace, LTTng, Zephyr logging, systemd-journald, NanoLog, RT-Thread ulog, OpenHarmony HiLog Lite) and analyzes their optimization code paths. Then a theory-first model is built to derive how time/space/throughput scale with log count `N` and device count `D`. Finally, measured results are compared against the derived trends.

This thesis emphasizes the latest strict-aligned evidence: single-node tests, local multi-device scans (`D=5,10,20,50`), real multi-node runs (`5/10/15/20` nodes with hardware CRC32C), and space crossover scans (`N=1...100000`). `binary` consistently outperforms `text` across all included semantic groups. Under native-path fairness (Optbinlog aligned to peer semantics while peers keep their original storage paths), `binary` remains clearly ahead of `syslog/ulog` in real multi-node runs, stays mixed against `hilog`, and still trails `nanolog/zephyr` in node-level comparisons; `ftrace` keeps a split pattern (space-positive but time/throughput-negative). Space scans show crossover near `N≈10` for `syslog/ulog`, near `N≈50` for `hilog`, and no observed crossover for `nanolog/zephyr` up to `N=100000`. Initialization-race results consistently satisfy the single-creator property. Overall, theoretical predictions and empirical trends are aligned.

Keywords: embedded logging; binary encoding; shared metadata; concurrency; performance modeling

## 1 绪论

### 1.1 研究背景

嵌入式与边缘设备日志系统通常同时面临高频写入、存储受限、并发访问与实时约束。\textsuperscript{[4,12,14,20]} 传统文本日志在可读性与生态兼容性方面具有优势，但在高负载场景下，字符串格式化、冗余字符存储与跨进程重复解析会持续放大时间与空间开销；当多个进程或节点共享同一格式元数据时，还可能出现初始化竞态与一致性风险。\textsuperscript{[1,3,7,8]} 基于此，本文将研究重点放在“结构化二进制编码与共享元数据机制”上，并将其置于统一的理论—实证框架中进行检验。

### 1.2 研究目标与问题定义

围绕上述背景，本文聚焦以下四个核心问题：

1. 现有代表性日志系统在热路径优化、并发组织与空间控制方面分别采用了哪些机制，其适用边界是什么。
2. Optbinlog 的共享元数据与二进制编码路径能否在保持语义完整性的前提下弥补现有方案缺口。
3. 在日志条数 `N` 与并发设备（节点）数 `D` 变化时，时间、空间与吞吐应呈现怎样的理论趋势与交叉关系。
4. 理论预测与实验观测在何处一致、何处偏离，偏离由哪些实现常数项与系统路径因素导致。

### 1.3 主要贡献

本文的主要工作与贡献如下：

1. 形成面向嵌入式多进程场景的 Optbinlog 方案，核心机制为共享 schema、偏移访问、自描述帧与并发初始化状态机。
2. 建立 `N`、`D` 驱动的统一模型，给出时间、空间、吞吐的可检验趋势与交叉点表达。
3. 完成最新一版严格语义对齐实验闭环：single、本地多设备 `D=5,10,20,50`、真实多设备 `5/10/15/20` 节点、空间交叉点扫描与初始化竞争验证。
4. 基于最新结果给出关键分组结论：`syslog/ulog` 稳定占优，`hilog` 呈中间形态，`nanolog/zephyr` 在当前口径下仍未跨越节点级交叉点。

### 1.4 研究方法与技术路线

本文采用“问题驱动、机制先行、分层验证”的研究路径，具体包括三条主线：

1. **机制分析主线**：以算法与数据结构为起点，先解释代价来源，再进行性能比较。
2. **公平对照主线**：在保留工程近似层的同时，引入严格语义对齐层，避免将语义差异误判为实现差异。
3. **理论—实证闭环主线**：统一统计口径与复现实验脚本，按“理论推导→单机→多设备→真实多设备”逐层验证。

### 1.5 创新点与研究价值

1. 提出面向用户态的“共享 schema + 偏移访问”二进制日志机制，缓解多进程初始化一致性与重复解析问题。
2. 给出 `N`、`D` 驱动下的时间/空间/吞吐统一推导框架，并提供可直接检验的交叉点与峰值点表达。
3. 建立“工程近似层 + 严格公平层”的双层证据体系，提高结论的可解释性与可迁移性。
4. 在真实多设备节点级并发场景下完成共享格式文件与初始化竞争验证，形成可复现的工程化实验流程。

## 2 国内外研究现状与问题归纳

### 2.1 研究对象与评估维度

本章基于官方文档、公开论文与代码仓库开展调研，重点关注“热路径优化机制”与“可复现实现入口”。\textsuperscript{[1,3-4,7-8,12,14]} 为保证后续分析可比，所有系统统一按四个维度展开：数据结构组织、关键算法路径、并发行为特征与工程部署边界。

调研对象如下：
1. Linux ftrace / kernel ring buffer。
2. LTTng。
3. Zephyr logging subsystem。
4. systemd-journald。
5. NanoLog（USENIX ATC 2018）。
6. RT-Thread ulog。
7. OpenHarmony HiLog Lite。

### 2.2 国际代表系统：机制与实现路径（代码级）

#### 2.2.1 Linux ftrace（内核追踪）

代码与文档显示其核心是 `per-CPU lockless ring buffer`。在 `ring_buffer.c` 中，写入路径围绕“预留 -> 填充 -> 提交”组织，关键是把竞争局部化到 CPU 本地缓冲，降低全局锁竞争。\textsuperscript{[1-2,15]}

1. 数据结构：每 CPU buffer + page 链式组织，维护 head/commit/tail。
2. 算法逻辑：优先本地写入、批量可见性提交，读写阶段解耦。
3. 场景定位：内核 tracing、高并发诊断；功能强但接入复杂度高于应用层日志库。

#### 2.2.2 LTTng（Linux Tracing Toolkit Next Generation）

LTTng 采用 tracepoint instrumentation + lockless buffer 管线，强调高吞吐追踪和离线分析。\textsuperscript{[3]}

1. 数据结构：channel/sub-buffer 等追踪缓冲结构。
2. 算法逻辑：将事件采集与后处理分层，追踪点开销可控。
3. 场景定位：系统级性能分析与可观测平台，部署和运维要求较高。

#### 2.2.3 Zephyr logging（RTOS 场景）

在 Zephyr `Kconfig.mode` 与 `log_msg.c` 可见，关键机制是 deferred logging。\textsuperscript{[4-6]}

1. 数据结构：`log_msg` 对象 + 消息池（MPSC packet buffer）。
2. 算法逻辑：前台线程只做封包入队，后台线程统一输出；支持 overflow/drop 策略。
3. 场景定位：RTOS 中实时任务优先，要求前台日志路径尽量短、可控。

#### 2.2.4 systemd-journald

其 `JOURNAL_FILE_FORMAT` 与 `journald-server` 路径强调二进制对象与索引。\textsuperscript{[7,16]}

1. 数据结构：header + objects + 哈希/索引。
2. 算法逻辑：写入与检索并重，保证一致性与查询能力。
3. 场景定位：通用 Linux 系统服务日志；对轻量嵌入式而言依赖偏重。

#### 2.2.5 NanoLog（研究型高性能日志）

从 `NanoLogCpp17.h`、`RuntimeLogger.cc` 可见，NanoLog 核心是“编译期提取格式 + 运行期仅记录动态参数”。\textsuperscript{[8-10]}

1. 数据结构：staging buffer、runtime logger 队列/缓冲。
2. 算法逻辑：前台路径尽量只做参数打包，压缩与格式恢复转移到后台/离线。
3. 场景定位：超高频日志、允许离线恢复文本的性能敏感场景。

### 2.3 国内代表系统：机制与实现路径（代码级）

#### 2.3.1 RT-Thread ulog

`ulog.c` 提供异步输出和多 backend 抽象，思路是“前台快写缓冲，后台慢速输出”。\textsuperscript{[12-13]}

1. 数据结构：ringbuffer/rbb、backend 列表。
2. 算法逻辑：异步线程统一调度输出，减轻业务线程阻塞。
3. 场景定位：RT-Thread 设备侧调试与在线日志。

#### 2.3.2 OpenHarmony HiLog Lite

HiLog Lite 在格式能力上做约束（减少复杂格式），换取更低资源开销。\textsuperscript{[14]}

1. 数据结构：轻量日志缓冲和简化格式存储。
2. 算法逻辑：限制格式复杂度，降低格式化路径成本。
3. 场景定位：资源受限设备的系统日志基础设施。

### 2.4 Optbinlog 与代表性 peer 系统的对比

#### 2.4.1 代表性 peer 对比与场景分工

需要说明的是，前述调研覆盖了更广的国内外系统范围，而第5章的定量复现只选择其中具有代表性的六类 peer：`syslog/ftrace/nanolog/zephyr/ulog/hilog`。选择依据是机制覆盖度、用户态可比性与实验可复现性。\textsuperscript{[1,4,8,12-15]} `LTTng` 与 `systemd-journald` 在本文中承担机制参照作用，不直接进入第5章统一口径下的数值对比。

| 系统 | 核心机制 | 典型使用场景 | 主要优势 | 主要代价 | 与 Optbinlog 的关系 |
|:---:|---|---|---|---|---|
| Linux ftrace | 内核 tracing ring buffer | 内核/驱动诊断、时序追踪 | 内核路径短、追踪能力强 | 接入复杂、与业务日志模型存在语义差异 | Optbinlog 适合业务日志结构化记录，Linux ftrace 更适合底层追踪 |
| Zephyr logging | deferred 入队 + 后台批刷 | RTOS 实时任务日志 | 前台阻塞小 | 后台线程调度与队列竞争 | 工程复现中以 native-like deferred 路径逼近其原生机制；Optbinlog 不依赖后台线程，行为更确定 |
| syslog | 文本日志 + 系统日志接口路径 | 通用 Linux 服务日志 | 生态成熟、接入通用 | 文本格式化与系统路径开销较高 | Optbinlog 在高频写入场景更强调紧凑编码与可控开销 |
| NanoLog | 固定字段二进制直写 | 超高频、可离线恢复文本 | 调用点开销极低 | 语义扩展与工具链耦合强 | 工程复现中以 native-like 固定字段二进制路径逼近其原生机制；Optbinlog 可扩展性更强，但热路径常数项更高 |
| RT-Thread ulog | 异步队列 + 多 backend | 嵌入式设备在线调试 | 工程集成灵活 | 后台文本格式化与后端竞争 | 工程复现中以 native-like 异步队列路径逼近其原生机制；Optbinlog 在结构化与一致性方面更强 |
| OpenHarmony HiLog Lite | 受限字段 + 轻量头 | 资源受限系统日志 | 资源占用较低 | 语义表达能力受限 | 工程复现中以 native-like 轻量头路径逼近其原生机制；Optbinlog 提供更完整字段语义和校验能力 |
| **Optbinlog** | 共享 schema + typed 编码 + CRC | 多进程边缘网关/控制器 | 跨进程一致性、结构化可扩展、空间可控 | 初始化和通用编码固定成本 | 作为“语义完整 + 工程可迁移”的折中方案 |

: Optbinlog 与六类 peer 系统对比 {#tbl:peer-compare}

如表\ref{tbl:peer-compare}所示，Optbinlog 在设计上更偏向“语义完整、跨进程一致和工程可迁移”的折中路线，因此与追求极限热路径或系统级 tracing 的方案形成了明显分工。

1. 若目标是**内核态追踪**，`Linux ftrace` 仍是优先选择；若目标是**应用层结构化日志**，Optbinlog 更匹配。
2. 若目标是**极限热路径时间**，`NanoLog/Zephyr logging` 这类强调短热路径的方案更有优势；若目标是**语义可扩展与跨进程一致性**，Optbinlog 更稳定。
3. 若系统强调**运维生态兼容**，`syslog` 路径更易快速接入；若系统强调**嵌入式长期在线与空间效率**，Optbinlog 更具针对性。

#### 2.4.2 二进制日志优化研究特征与 Optbinlog 对比

结合第2章调研对象与公开实现路径，可以把“二进制日志优化研究”的主流特征归纳为三类：
1. **热路径极短化**：通过编译期提取静态模板、运行期只记录动态参数，最大化降低调用点常数开销（典型如 NanoLog）。\textsuperscript{[8-10]}
2. **并发写出解耦化**：通过无锁环形缓冲或 deferred 队列，将前台写入与后台落盘拆分，降低前台阻塞（典型如 ftrace/LTTng/Zephyr，以及 ETW 采集优化工具 Kellect）。\textsuperscript{[1,3-4,15,21]}
3. **后处理集中化**：把文本恢复、检索或压缩能力后移到离线/后台，以换取在线写入吞吐（典型如 NanoLog、集群日志无损压缩研究与 Binlog 工程方案）。\textsuperscript{[3,8,22-23]}

对应地，这类研究的优化目标通常集中在两点：其一是降低单位日志写入时间（尤其是 p95/p99 热路径）；其二是在高并发下保持吞吐稳定，避免锁竞争放大。相比之下，Optbinlog 的目标函数更偏“多目标折中”：在保持较低写入开销的同时，保留跨进程共享语义、记录级校验与在线可读恢复能力。

| 研究/系统方向 | 优化重点与目标 | 与 Optbinlog 的关键差异 |
|:---:|---|---|
| NanoLog 类编译期优化 | 目标是极限热路径时延；通过编译期拆分模板降低运行期开销 | Optbinlog 不是纯离线恢复路径，而是保留共享 schema、在线结构化解码与记录级校验，热路径常数项更高但语义治理能力更强 |
| ftrace/LTTng 类 tracing 缓冲 | 目标是高并发低侵入追踪；通过无锁/低锁缓冲提升采集吞吐 | tracing 更偏系统观测而非业务日志语义一致性；Optbinlog 更强调应用层 event schema 与跨进程一致映射 |
| Kellect（ETW 采集优化） | 目标是“无丢失 + 低开销 + 实时”采集链路；优化采集与解析管线 | 更偏“日志采集器层”优化，不直接约束应用写入编码格式；Optbinlog 属于应用写入与共享格式层 |
| Zephyr deferred/ulog 异步队列 | 目标是缩短前台路径、把代价后移到后台线程 | Optbinlog 采用共享元数据 + 批量写出，不依赖独立后台日志线程，稳态行为更确定，但初始化与通用编码固定成本更明显 |
| Lossless compression for large-scale cluster logs | 目标是提升大规模日志压缩比与压缩效率，降低存储成本 | 属于“存储后处理层”优化，不直接降低单条写入热路径；Optbinlog 重点在写入阶段的结构化与一致性 |
| Binlog（结构化二进制工程方案） | 目标是元数据/数据分离与高效序列化，兼顾实时写入与后续解码 | 与 Optbinlog 在结构化方向接近，但 Binlog 更偏单进程库生态；Optbinlog 更强调共享格式文件与跨进程一致映射 |
| Optbinlog（本文） | 目标是“可复现、可校验、可扩展”的工程折中；强调共享 schema 与一致性 | 在 `syslog/ulog` 组可形成稳定综合优势；在 `nanolog/zephyr` 这类极短路径对照下仍需继续压缩单位记录常数项 |

: 二进制日志优化研究特征与 Optbinlog 对比 {#tbl:binary-research-vs-optbinlog}

如表\ref{tbl:binary-research-vs-optbinlog}所示，现有工作的“突破点”往往分布在不同层级。NanoLog、ftrace、LTTng 与 Kellect 更偏采集链路与热路径；Lossless compression 更偏存储后处理；Binlog 更偏结构化编码库。\textsuperscript{[1,3-4,8,15,21-23]} Optbinlog 的取舍更接近工程部署场景：在语义对齐、并发一致、空间统计可追溯与记录级校验同时成立的约束下追求综合最优。这也是第5章中“对部分 peer 稳定占优、对极短路径 peer 仍有优化空间”这一结果结构的直接原因。

### 2.6 现有方案的共性不足

从嵌入式角度可抽象出四类常见缺口：

1. **格式化开销仍在热路径**：text/syslog 在高频场景下受字符串格式化与内核接口调用影响明显。
2. **元数据复用不足**：多进程系统若每进程重复解析格式，启动和内存成本上升。
3. **并发初始化一致性风险**：共享资源初次创建缺少状态机时，容易出现竞态。
4. **跨平台复现实验缺失**：只在单平台得到的结论容易失真，外推边界较弱。

### 2.7 Optbinlog 的针对性设计

1. 用“事件格式文件 + 二进制编码”替代热路径字符串格式化。
2. 用共享元数据文件持久化 tag/element 描述，进程间偏移复用。
3. 用 `INITIALIZING -> INITIALIZED` 状态机控制并发初始化。
4. 用统一脚本输出 mean/median/p95/std/95%CI 与 IQR 过滤结果，支持多轮迭代对比。

### 2.8 复现边界与结论适用范围

本文按照“代码阅读先行、实验复现跟进”的顺序开展验证工作：

1. 已完成官方代码级阅读：ftrace、Zephyr、NanoLog、ulog 等核心实现入口。
2. 受网络与平台限制：无法完整拉取/编译全部外部系统并在本机跑 Linux tracing 全栈。
3. 已执行算法级复现：实现 `nanolog_like`、`zephyr_deferred_like`、`ulog_async_like`、`hilog_lite_like` 及“语义对齐”版本（第5章给出数据）。

上述约束已在第5章实验方法中显式记录，用于区分“工程可复现边界”与“算法层结论边界”。因此，第5章中的排序结论均在对应口径下成立，不对未覆盖平台作超范围外推。

## 3 Optbinlog 系统设计与工程实现

### 3.1 设计目标

1. 降低单条日志热路径开销（时间）。
2. 降低长期写入空间消耗（空间）。
3. 支持多进程共享元数据与安全初始化（并发）。
4. 对齐嵌入式可部署需求（工程可落地）。

上述目标在后续章节中分别对应为可测指标：时间目标映射到 `elapsed/end_to_end`，空间目标映射到 `total_bytes`，并发目标映射到初始化竞争正确性与节点级稳定性，工程目标映射到跨场景复现与部署可迁移性。

### 3.2 事件格式与元数据结构

当前 Optbinlog 采用“共享元数据文件 + 进程内共享视图缓存 + 自描述二进制帧 + `CRC32C(hw)` + string-heavy 场景自动 `varstr`”的一体化实现路径；其事件定义思路与通用事件日志标签机制保持一致。\textsuperscript{[11,20]}

```{=latex}
\begin{figure}[H]
\centering
\includesvg[width=0.94\linewidth]{./demo/results/paper_dataset/supplement/figures/optbinlog_design_overview.svg}
\caption{Optbinlog 设计总览图（共享元数据、热路径与读取校验链路）}
\label{fig:optbinlog-design-overview}
\end{figure}
```

如图\ref{fig:optbinlog-design-overview}所示，系统以共享格式文件为中心，把“schema 解析、tag 组织、字段统计”前移到共享与缓存层，稳态写入只保留编码、帧构造、校验与批量落盘这四类关键操作。该组织方式直接对应第4章中的“固定项与边际项分离”思想。

事件定义采用 `(name|type|bits)` 形式，字段类型简化为 `L/D/S`：

1. `L`：定长整数，按 schema 指定字节数写入。
2. `D`：双精度浮点，固定写入 8 字节。
3. `S`：字符串字段；当前实现同时支持固定宽度字符串编码和 `varstr` 编码。

共享元数据文件内仍由以下结构组织：

1. `OptbinlogSharedTag`：共享文件头与初始化状态。
2. `OptbinlogBitmap`：tag 合法性位图。
3. `OptbinlogEventTag`：事件定义入口。
4. `OptbinlogEventTagEle`：元素类型与长度描述。

在共享文件之上，系统进一步引入了**进程内共享视图缓存层**。该缓存按共享文件的 `dev/inode/size/mtime` 判定是否可复用，并缓存：

1. 已映射的共享文件视图。
2. 每个 tag 的 `OptbinlogTagCacheEntry`。
3. schema 级统计量，例如 tag 数、总 payload 字节数、字符串字段总宽度、字符串字段数和最大字符串字段宽度。

这些统计量会直接参与写路径中的 `varstr` 自动决策，因此它们属于当前热路径优化的一部分。

```{=latex}
\begin{figure}[H]
\centering
\includesvg[width=0.94\linewidth]{./demo/results/paper_dataset/supplement/figures/optbinlog_data_structure.svg}
\caption{Optbinlog 关键数据结构与记录布局示意图}
\label{fig:optbinlog-data-structure}
\end{figure}
```

如图\ref{fig:optbinlog-data-structure}所示，左侧共享元数据布局保证“跨进程可移植”，右侧记录帧布局保证“按帧可校验、按 schema 可还原”，二者共同构成第5章实验统计可追溯的结构基础。

### 3.3 写入/读取路径

写入与读取遵循“元数据状态判定→编码/解码执行→一致性校验”的统一流程。

```{=latex}
\begin{figure}[H]
\centering
\includesvg[width=0.95\linewidth]{./demo/results/paper_dataset/supplement/figures/optbinlog_rw_flow.svg}
\caption{Optbinlog 写入—读取运行流程图}
\label{fig:optbinlog-rw-flow}
\end{figure}
```

如图\ref{fig:optbinlog-rw-flow}所示，写入与读取并非两条割裂路径，而是由同一共享 schema 贯通。该流程设计使“语义对齐实验”和“记录级复核”能够使用同一套格式定义，减少统计口径漂移。

写入路径：
1. 通过 `acquire_shared_view()` 获取共享文件视图；若共享文件未变化，则直接复用进程内缓存的 tag cache。
2. 根据 schema 统计量判断是否启用 `varstr`；当字符串字段占比高或最大定长字符串字段较大时，默认切换到 `varstr`。
3. 依据 tag cache 顺序编码 payload。
4. 构造 4 字节自描述帧头，其中同时编码 `payload_len`、`varstr` 标志位和校验类型。
5. 对单条记录计算校验值；默认采用 `CRC32C(hw)`，仅在硬件不可用时回退到软件实现。
6. 将 `frame_header + payload + checksum` 写入用户态缓冲区，按批次落盘。

读取路径：
1. 读取 4 字节帧头，并解析 `payload_len`、`varstr` 标志位与 `checksum_type`。
2. 读取 payload 与校验字段，并按帧头声明的校验类型执行校验。
3. 按共享格式解码：整数和浮点按固定宽度读取，字符串则根据 `varstr` 标志选择“长度前缀 + 数据”或“固定宽度截断/补零”方式。
4. 将解码后的记录交给统一回调接口。

因此，当前记录格式可写作：

$$
\mathrm{frame} = \mathrm{frame\_header} \;\Vert\; \mathrm{payload} \;\Vert\; \mathrm{checksum}
$$

其中：

$$
\mathrm{frame\_header}
=
\mathrm{payload\_len}
\;\cup\;
\mathrm{varstr\_bit}
\;\cup\;
\mathrm{checksum\_type}
$$

### 3.4 核心迭代与收敛

为突出与最终结果直接相关的改进，本文将迭代收敛为三个阶段：

1. **结构一致性阶段**：将“每事件独立结构”统一为“共享结构 + 位图 + 偏移表”，并将跨进程共享从指针语义改为偏移语义，解决映射基址不一致导致的可移植性问题。
2. **热路径压缩阶段**：将 tag 查找改为常数级索引路径，引入进程内共享视图缓存与 tag cache，把共享文件 reopen、tag 遍历与 schema 统计迁出稳态写路径。
3. **可靠性与空间协同阶段**：从软件 `CRC32` 迭代为默认 `CRC32C(hw)`，引入自描述帧头与 `varstr` 自动策略，并通过初始化状态机与 init-race 验证保证“单创建者 + 多复用者”并发语义。

对应到最终版本，关键收敛点可概括为：

1. **可靠性不降、校验成本下降**：保留每条记录校验语义，同时以 `CRC32C(hw)` 降低热路径校验开销。
2. **共享元数据收益进入稳态**：共享视图缓存使 schema 解析与 tag 遍历不再重复进入每次写入。
3. **字符串冗余持续收缩**：string-heavy 场景下自动启用 `varstr`，减少固定宽度补零与无效拷贝。

### 3.5 工程场景适配性

1. 小样本 `N`：共享文件打开、共享视图校验、文件打开和缓冲准备等固定项占比仍然较高，时间优势不一定立刻出现。
2. 中大样本 `N`：固定项被摊薄，自描述帧头和 `CRC32C(hw)` 的额外代价收敛到常数级，binary 的时间和空间优势更稳定。
3. string-heavy schema：自动 `varstr` 会显著减少补零写入和冗余字节，因此空间优势更容易显现。
4. 高并发 `D`：即使校验和 schema 解析成本被压低，调度、文件竞争和写出路径仍会主导吞吐峰值与回落点。
5. 多进程/多节点部署：共享 schema、进程内缓存复用和节点级初始化竞争正确性已经形成闭环，因此该设计更贴近真实工程部署。

因此，第4章将把上述实现行为进一步抽象为 `A_m/C_m/B_m/W_m` 等参数形式，便于与第5章实验结果逐项对照。

### 3.6 工程化读取能力补全

为提升实验数据复核效率，本文在写入与解码库之外补充了 `optbinlog_read` 工具，用于“共享格式文件 + 二进制日志文件”的直接可视化读取。该工具具备三项能力：
1. 按 tag 与字段名输出记录，避免只看到匿名值序列。
2. 支持 `table/jsonl` 两种格式，便于命令行排错与自动化分析脚本复用。
3. 支持 `--limit` 与 `--summary`，可快速截取样本并统计各 tag 分布。

该工具把“论文中的统计结果”与“底层二进制记录”建立了可追溯链路，使实验结论不仅停留在汇总图表层，也可直接回溯到记录级证据。

## 4 理论模型与数学推导

本章先做抽象推导，不代入具体事件文件数值。
为保证推导可解释且可验证，本文采用以下前提：在单次统计窗口内事件分布近似稳定；各模式在同一场景下使用一致的输入语义；模型参数以期望意义解释而非单点极值解释。

### 4.1 符号定义

1. $N$：总日志条数。
2. $D$：并发设备数（或并发进程数）。
3. $r$：每设备日志条数，满足 $N = D \cdot r$。
4. $k$：事件类型（tag）索引，$k \in \{1,\dots,T\}$。
5. $p_k$：事件类型 $k$ 的概率，满足 $\sum_k p_k = 1$。
6. $e_k$：事件 $k$ 的字段数。
7. $l_{k,j}$：事件 $k$ 第 $j$ 个字段编码字节数。
8. $L_k = \sum_{j=1}^{e_k} l_{k,j}$：事件 $k$ 的有效载荷字节数。
9. $A_m$：模式 $m$ 的固定时间开销（初始化、打开文件、预热）。
10. $C_m$：模式 $m$ 的单条边际时间开销。
11. $B_m$：模式 $m$ 的单条边际空间开销。
12. $S_{0,m}$：模式 $m$ 的固定空间开销（例如共享元数据）。
13. $Q_m$：吞吐（records/s）。

除特别说明外，本章的“空间”均对应实验统计中的 `total_bytes`：对 `binary` 为“日志文件 + 共享格式文件且共享文件只计一次”，对其他模式为其实际输出文件大小；“时间”默认对应单轮写入阶段的均值趋势，若切换到 `end_to_end` 口径，则把进程/节点启动与调度准备吸收到固定项中。

### 4.2 从算法与数据结构到单条代价

为避免“只看结果不看机制”，先把每种模式拆成操作级成本。以下表达式均以单条记录期望开销为核心，再扩展到总量关系。

#### 4.2.1 text

单条记录包含：
1. 读取输入字段。
2. `snprintf`/拼接格式串。
3. 写入用户态缓冲并最终落盘。

时间可写为：

$$
t_{\text{text}}(k) = t_{\text{parse}}(k) + t_{\text{fmt}}(k) + t_{\text{write}}(\mathrm{bytes}_{\text{text}}(k))
$$

空间可写为：

$$
b_{\text{text}}(k) = b_{\text{prefix}} + b_{\text{delim}} + b_{\text{num}}(k) + b_{\text{str}}(k)
$$

其中 $b_{\text{num}}(k)$ 是数字转十进制后的字符长度，随数值大小缓慢变化。

#### 4.2.2 binary（optbinlog）

当前 `optbinlog_binlog_write()` 的稳态路径可分为四步：
1. 复用共享视图与 tag cache，并按 `tag_id` 完成常数级查找。
2. 按字段类型编码 payload。
3. 根据校验模式计算记录校验。
4. 将记录写入用户态批量缓冲区，缓冲区满后统一写出。

与代码一致，单条记录的物理布局为：

$$
\mathrm{frame}
=
\mathrm{frame\_header}_{4B}
\;\Vert\;
\mathrm{payload}
\;\Vert\;
\mathrm{checksum}_{4B}
$$

其中 payload 并不只是 schema 字段本身，还额外包含：

$$
\mathrm{payload}
=
\mathrm{timestamp}_{8B}
\;\Vert\;
\mathrm{tag\_id}_{2B}
\;\Vert\;
\mathrm{ele\_count}_{1B}
\;\Vert\;
\mathrm{fields}
$$

记记录内部固定元数据开销为：

$$
g_{\text{bin}} = 8 + 2 + 1 = 11
$$

若字符串字段按固定宽度写出，则 payload 中 schema 字段的有效载荷仍记为 $L_k$；若 schema 触发 `varstr`，则对应地记为 $\widetilde{L}_k$。于是单条时间可写为：

$$
t_{\text{bin}}(k)
=
t_{\text{lookup}}
+
t_{\text{encode}}(k)
+
t_{\text{checksum}}(k)
+
\bar{t}_{\text{write}}(b_{\text{bin}}(k))
$$

其中 $\bar{t}_{\text{write}}(\cdot)$ 表示“经用户态缓冲摊薄后的平均写出代价”，比逐条系统调用更接近当前实现。

单条空间则可写为：

$$
b_{\text{bin}}(k) = 4 + g_{\text{bin}} + \widetilde{L}_k + 4
$$

固定空间只在共享格式文件层面体现，因此有：

$$
S_{0,\text{bin}} = S_{\text{shared}}
$$

若 schema 触发 `varstr`，则字符串字段的有效载荷不再按固定宽度写出，而是写为“长度前缀 + 实际长度”：

$$
\widetilde{L}_k
=
\sum_{j \in \mathcal{N}_k} l_{k,j}
\;+\;
\sum_{j \in \mathcal{S}_k} \left( 2 + \hat{l}_{k,j} \right),
\qquad
\hat{l}_{k,j} \le l_{k,j}
$$

还需说明两点与代码实现直接对应的口径：
1. 默认 `CRC32C(hw)` 覆盖的是“帧头 + payload”；兼容模式下 `CRC32` 覆盖 payload，`none` 则不做校验。
2. 共享文件的打开、映射与 tag cache 构建只在视图失效时重新执行，因此它们主要进入固定项 $A_{\text{bin}}$，而不是每条记录的边际项。

#### 4.2.3 syslog / ftrace

抽象为“系统日志接口 + 内核路径”：

$$
t_{\text{sys}}(k) = t_{\text{format\_sys}}(k) + t_{\text{syscall}} + t_{\text{kernel\_path}} + t_{\text{write\_kernel}}
$$

$$
t_{\text{ftrace}}(k) = t_{\text{format\_ftrace}}(k) + t_{\text{trace\_write}} + t_{\text{ring\_or\_file\_path}}
$$

空间：

$$
b_{\text{sys}}(k) = b_{\text{sys\_header}} + b_{\text{sys\_payload}}(k)
$$

$$
b_{\text{ftrace}}(k) = b_{\text{ftrace\_header}} + b_{\text{ftrace\_payload}}(k)
$$

#### 4.2.4 nanolog_like / zephyr_deferred_like / ulog_async_like / hilog_lite_like

`nanolog_like`（固定字段二进制）：

$$
t_{\text{nano}} = t_{\text{pack\_fixed}} + t_{\text{memcpy}} + t_{\text{write}}
$$

$$
b_{\text{nano}} = \text{const}
$$

`zephyr_deferred_like`（前台入队 + 后台批刷）：

$$
t_{z,\text{front}} = t_{\text{enqueue}} + t_{\text{signal}}
$$

$$
t_{z,\text{back}} = \frac{t_{\text{dequeue}} + t_{\text{batch\_write}}(k_b)}{k_b}
$$

$$
t_z = t_{z,\text{front}} + t_{z,\text{back}}
$$

$$
b_z = b_{\text{queue\_node}} + b_{\text{fixed}} + \frac{b_{\text{batch\_meta}}}{k_b}
$$

`ulog_async_like`（异步队列 + 文本 backend）：

$$
t_{\text{ulog}} =
t_{\text{enqueue}} + t_{\text{cond\_signal}}
+
\frac{t_{\text{dequeue}} + t_{\text{format\_text}} + t_{\text{backend\_write}}}{k_b}
$$

$$
b_{\text{ulog}} = b_{\text{queue\_node}} + b_{\text{text\_payload}}
$$

`hilog_lite_like`（受限字段 + 轻量头）：

$$
t_{\text{hilog}} = t_{\text{compact\_format}} + t_{\text{write}}(h_{\text{lite}} + p_{\text{lite}})
$$

$$
b_{\text{hilog}} = b_{\text{header\_lite}} + b_{\text{payload\_lite}}
$$

### 4.3 空间随 `N` 的详细推导

#### 4.3.1 期望单条字节

对任意模式 $m$，定义：

$$
B_m = \mathbb{E}[b_m(k)] = \sum_k p_k \, b_m(k)
$$

则总空间：

```{=latex}
\begin{equation}
S_m(N) = S_{0,m} + N B_m
\label{eq:space-linear}
\end{equation}
```

这是严格的一阶线性模型，斜率为 $B_m$，截距为 $S_{0,m}$。
见公式（\ref{eq:space-linear}）所示，后续空间分析都可以归结为“固定项 + 单条斜率”两部分。

#### 4.3.2 binary 的显式表达

由 4.2.2 得：

$$
B_{\text{bin}} = 8 + g_{\text{bin}} + \sum_k p_k \widetilde{L}_k
$$

```{=latex}
\begin{equation}
S_{\text{bin}}(N) = S_{\text{shared}} + N \left(8 + g_{\text{bin}} + \sum_k p_k \widetilde{L}_k\right)
\label{eq:bin-space-total}
\end{equation}
```

其中，$g_{\text{bin}}=11$ 对应 payload 内部固定元数据，外层的两个 `4B` 分别对应帧头和校验字段。见公式（\ref{eq:bin-space-total}）所示，`binary` 的总空间同时受“每条记录固定项”和“共享格式文件固定项”影响。

#### 4.3.3 节省率与导数

binary 相对模式 $x$ 的空间节省率：

$$
\eta_{\text{bin}/x}(N) = 1 - \frac{S_{\text{bin}}(N)}{S_x(N)}
$$

若 $S_{0,x}=0$，则

$$
\eta_{\text{bin}/x}(N) = 1 - \frac{B_{\text{bin}} + S_{\text{shared}}/N}{B_x}
$$

求导：

$$
\frac{d\eta_{\text{bin}/x}}{dN} = \frac{S_{\text{shared}}}{N^2 B_x} > 0
$$

说明 $N$ 越大，$S_{\text{shared}}/N$ 越小，binary 空间优势越稳定。
这说明只要边际斜率满足优势条件，共享文件这一固定项就会随着 $N$ 增大而被持续摊薄。

#### 4.3.4 空间交叉点

当比较 $\text{bin}$ 与 $x$：

$$
S_{\text{bin}}(N) \le S_x(N)
$$

$$
S_{\text{shared}} + N B_{\text{bin}} \le S_{0,x} + N B_x
$$

若 $B_x > B_{\text{bin}}$，可得交叉点：

```{=latex}
\begin{equation}
N^*_{\text{space}}(\text{bin},x) = \frac{S_{\text{shared}} - S_{0,x}}{B_x - B_{\text{bin}}}
\label{eq:space-crossover}
\end{equation}
```

当 $N > N^*_{\text{space}}$ 时，binary 空间一定优于 $x$。
因此，空间交叉点的判断可以直接按公式（\ref{eq:space-crossover}）所示进行。

### 4.4 时间随 `N` 的详细推导

#### 4.4.1 从单条时间到总时间

定义：

$$
C_m = \mathbb{E}[t_m(k)] = \sum_k p_k \, t_m(k)
$$

则总时间：

```{=latex}
\begin{equation}
T_m(N) = A_m + N C_m
\label{eq:time-linear}
\end{equation}
```

该形式直接对应实验中的 `elapsed/write_only/end_to_end` 指标。
见公式（\ref{eq:time-linear}）所示，总时间同样由固定项和边际项组成。若统计口径切换到 `end_to_end`，则进程/节点启动、文件准备与调度等待也并入 $A_m$。

#### 4.4.2 吞吐公式与单调性

```{=latex}
\begin{equation}
Q_m(N) = \frac{1000N}{T_m(N)} = \frac{1000}{C_m + A_m/N}
\label{eq:throughput}
\end{equation}
```

对 $N$ 求导：

$$
\frac{dQ_m}{dN} = \frac{1000A_m}{N^2\left(C_m + A_m/N\right)^2} > 0
$$

说明在模型中吞吐随 $N$ 单调上升并渐近：

$$
\lim_{N \to \infty} Q_m(N) = \frac{1000}{C_m}
$$

在 $A_m>0$ 与 $C_m>0$ 条件下，吞吐增速会随 $N$ 增大而下降，表现为“先快后慢”的收益递减，这与实验中大 $N$ 后增益变缓一致。
因此，公式（\ref{eq:throughput}）给出了第 5 章解释“吞吐为何先快后慢趋于平台”的统一表达。

#### 4.4.3 时间交叉点（binary 何时优于其他模式）

比较 $\text{bin}$ 与 $x$：

$$
T_{\text{bin}}(N) \le T_x(N)
$$

$$
A_{\text{bin}} + N C_{\text{bin}} \le A_x + N C_x
$$

当 $C_x > C_{\text{bin}}$ 时：

```{=latex}
\begin{equation}
N^*_{\text{time}}(\text{bin},x) = \frac{A_{\text{bin}} - A_x}{C_x - C_{\text{bin}}}
\label{eq:time-crossover}
\end{equation}
```

解释：
1. 若 $A_{\text{bin}}$ 较大（共享初始化重），小 $N$ 可能不占优。
2. 随 $N$ 增大，$NC$ 项主导，只要 $C_{\text{bin}} < C_x$，最终会跨过交叉点。

因此，某一模式何时在时间上完成反超，可按公式（\ref{eq:time-crossover}）所示理解。

#### 4.4.4 `C_m` 的字段复杂度展开

为把“常数项差异”写清楚，可把边际开销展开到字段数级别：

$$
C_m = c_{m,0} + c_{m,1}\mathbb{E}[e_k] + c_{m,2}\mathbb{E}[L_k] + c_{m,3}\mathbb{E}[\mathrm{syscall}_m]
$$

其中：
1. $\mathbb{E}[e_k]$ 决定字段遍历成本（编码循环次数）。
2. $\mathbb{E}[L_k]$ 决定 memcpy、校验和写出字节相关成本。
3. $\mathbb{E}[\mathrm{syscall}_m]$ 反映模式是否显著依赖系统日志接口或内核路径。

结合当前复现代码，可得到更贴近实现的解释：
1. `text_semantic_like` 主要承担字段格式化与文本展开成本。
2. `binary` 的边际项主要由 typed encode、校验和批量写出摊销组成，固定项则额外包含共享视图与 tag cache 准备。
3. `nanolog_like / zephyr_deferred_like` 走固定宽度打包路径，边际项通常较低，但 `zephyr_deferred_like` 还会受到批刷组织方式影响。
4. `ulog_async_like / syslog / ftrace` 会显著暴露文本展开、系统接口或内核路径的常数开销。
5. `hilog_lite_like` 通过轻量头和受限消息体压缩了单条记录规模，因此常落在二进制直写组与文本系统组之间。

### 4.5 多设备 `D` 的详细推导

#### 4.5.1 总时间分解

设每设备写 `r` 条：

```{=latex}
\begin{equation}
E_m(D,r) = A_{\text{spawn}}(D) + A_m + rC_m + W_m(D,r)
\label{eq:multi-time}
\end{equation}
```

见公式（\ref{eq:multi-time}）所示，多设备总时间除了单机边际项外，还需要显式计入并发等待项。若观察的是 `write_only` 指标，则 $A_{\text{spawn}}(D)$ 会明显缩小；若观察 `end_to_end`，它则不能忽略。

其中 $W_m$ 为竞争等待项（锁、队列、内核资源、I/O 饱和）。

#### 4.5.2 队列化近似（解释吞吐饱和）

把共享瓶颈抽象成单服务台，服务率 $\mu_m$（records/s），到达率 $\lambda = D\nu$（$\nu$ 为单设备提交率）。

稳定条件：

$$
\rho = \frac{\lambda}{\mu_m} < 1
$$

等待时间近似随 $\rho$ 增大而快速上升，可抽象为：

$$
W_m(D,r) \sim r f_m(\rho),
\qquad
f_m'(\rho) > 0,
\qquad
f_m''(\rho) > 0
$$

因此当 $D$ 接近

$$
D_{\text{sat},m} \approx \frac{\mu_m}{\nu}
$$

吞吐进入平台区；超过后出现明显回落。
这里的“服务台”不一定是真实队列，也可以对应共享文件写出通道、后台批刷线程或内核 tracing 接口；该抽象的意义在于用同一符号解释不同模式的饱和与回落。

#### 4.5.3 常用二次近似

在工程拟合中常写作：

$$
E_m(D,r) \approx \beta_m + rC_m + \alpha_m D + \gamma_m D^2
$$

总吞吐：

$$
Q_m(D,r) = \frac{1000Dr}{E_m(D,r)}
$$

当 $\gamma_m > 0$ 时，$D$ 过大必然导致分母增长过快，出现峰值后下降。

进一步对 $D$ 求导：

$$
\frac{dQ_m}{dD}
\propto
\left(\beta_m + rC_m + \alpha_m D + \gamma_m D^2\right)
-
D\left(\alpha_m + 2\gamma_m D\right)
$$

整理得：

$$
\frac{dQ_m}{dD} \propto \beta_m + rC_m - \gamma_m D^2
$$

故吞吐峰值点近似为：

```{=latex}
\begin{equation}
D_{\text{peak},m} = \sqrt{\frac{\beta_m + rC_m}{\gamma_m}}
\label{eq:device-peak}
\end{equation}
```

这给出了可直接与实验扫描结果比对的“理论峰值设备数”。
其中，峰值设备数可直接按公式（\ref{eq:device-peak}）与实验扫描结果对照。

#### 4.5.4 空间随 `D`

$$
S_m(D,r) = S_{0,m} + DrB_m
$$

因此空间对 $D$ 严格线性，差异主要由斜率 $B_m$ 与截距 $S_{0,m}$ 决定。
这一点与当前实验统计口径一致：在记录规模 $r$ 固定时，设备数增加只会线性放大记录总量，而不会改变每条记录的平均编码体积。

### 4.6 模式间理论排序与可检验预测

#### 4.6.1 异语义比较（最短路径口径）

在工程近似层中，各模式的输入语义和存储组织并不完全一致，因此本节只讨论“机制上更可能出现的方向”，而不把它写成固定全序。

从当前复现代码出发，可得到三条较稳定的预期：
1. 固定宽度二进制直写组（如 `nanolog_like`、`zephyr_deferred_like`）通常拥有更低的单位记录空间与更短的边际写入路径。
2. 文本展开或系统路径更重的模式（如 `text/syslog/ftrace/ulog_async_like`）通常拥有更高的常数项。
3. `binary` 与 `hilog_lite_like` 处于二者之间：前者保留共享 schema、自描述帧和校验语义，后者保留轻量头和受限消息体，因此二者的排序会更依赖具体 schema 与统计口径。

可检验预测：
1. nanolog_like 在时间与吞吐上常呈数量级优势。
2. hilog_lite_like 在“受限语义”下应明显优于 text/syslog。
3. ulog_async_like 在吞吐未饱和且后台轻负载时有机会受益；若后台文本后端成为瓶颈，可能慢于 binary/text。
4. 上述优势不代表语义等价下仍保持同倍率。

#### 4.6.2 语义对齐比较（公平口径）

本文所说的“语义对齐”，并不是把所有 peer 强行改写成同一种记录格式，而是在**同一组字段语义、同一组字段值与同一平台**下，让 Optbinlog 与 peer 分别走各自的 native-like 复现路径。于是：

1. `binary` 的差异主要来自共享 schema、frame 组织、per-record CRC 以及字段遍历。
2. peer 的差异主要来自其 native-like 记录布局、批刷/后台路径、文本展开与系统接口路径。

因此，公平口径下的空间关系不再由“语义体量不同”主导，而主要由各模式的**单位记录常数项**主导。对于 `nanolog_like / zephyr_deferred_like` 这类紧凑二进制 peer，$B_{\text{peer}}$ 往往长期小于 $B_{\text{bin}}$；对于 `ulog_async_like / hilog_lite_like / syslog` 这类仍带有消息头、文本展开或系统路径附加开销的 peer，$B_{\text{bin}}$ 更可能在较小 $N$ 后越过交叉点。

$C_{\text{peer}}$ 与 $C_{\text{bin}}$ 的差异同样主要来自实现常数，包括编码循环、缓存命中、批刷组织、格式化路径与校验实现，而不再主要来自“消息语义更多或更少”的偏差。

可检验预测：
1. 时间差异将由各模式的原生热路径决定，而不是由语义字段数决定。
2. 空间差异会在同一语义组内呈现稳定斜率，并在 $N$ 增长时显现是否存在交叉点。

### 4.7 与第5章指标的映射关系

为了让推导可被实验验证，建立公式到指标的映射：

1. $T_m(N)$ 对应 `elapsed_ms` 或 `end_to_end_ms` 的均值趋势。
2. $Q_m$ 对应 `throughput_rps` / `throughput_e2e_rps`。
3. $S_m$ 对应 `total_bytes`。
4. $N^*_{\text{time}}$ 与 $N^*_{\text{space}}$ 对应“某场景下 binary 由劣转优”的阈值。
5. $D_{\text{sat},m}$ 对应“多设备扫描中吞吐峰值后回落”的拐点。

本章给出的是“由算法与数据结构直接推出的理论趋势”；第5章再代入实测数据检验这些趋势是否成立，以及偏离原因。

### 4.8 本章小结

本章完成了三项工作：一是把不同日志模式统一到可比较的时间与空间参数形式；二是给出 `N` 与 `D` 两个核心自变量下的交叉点、峰值点与线性斜率含义；三是建立理论参数与实验指标的映射关系。基于这些结果，第5章将按“单机 -> 本地多设备 -> 真实多设备模拟 -> 日志量扫描”的顺序进行逐层验证。

## 5 实验设计、结果与理论一致性分析

### 5.1 实验设计与评估口径

本章按照“实验口径→结果呈现→理论对照→机制解释”的顺序展开。主证据采用最新一版严格语义对齐数据，统一比较对象为 `text_semantic_like / binary / peer native mode`，并覆盖以下三类场景。\textsuperscript{[20]}
1. 单机高负载（single）。
2. 本地多设备模拟（`D=5,10,20,50` 扫描）。
3. 真实多设备模拟（多台虚拟机节点）并发（`5/10/15/20` 节点，每节点一台设备）。

此外，本文同步报告两组初始化竞争结果（本地多进程与节点级并发）。统计口径统一为 `mean/median/p95/std/95%CI`，先执行 IQR（1.5）异常值过滤，再输出时间、空间、吞吐与改进百分比，以保证不同场景可横向比较。

本轮核心原则为：**Optbinlog 对齐 peer 语义，peer 保持原生存储路径；所有模式共享同一字段值生成器；共享格式文件在集群空间统计中仅计算一次**。在此基础上，本文增加了 `N=1～100000` 的空间扫描，用于回答“何时发生空间反超”。消融数据仅用于解释机制，不作为主结论依据。

为保证分析口径可追溯，本章额外固定“对比对象、运行平台与数据量”三类约束。尤其在真实多设备（多台虚拟机节点）阶段，本轮采用“**全节点共享同一 `shared_eventtag.bin` 格式缓存文件**、节点结果文件独立输出”的方式，重点观察共享元数据在节点级并发中的竞争与收益释放。

| 维度 | 本文实验口径 |
|:---:|---|
| 对比对象 | `text_semantic_like`（文本基线）、`binary`（Optbinlog 最终路径）；single/multi peer：`syslog`、`nanolog`、`zephyr`、`ulog`、`hilog`；l1 peer：`syslog`、`ftrace`、`nanolog`、`zephyr`、`ulog`、`hilog`；peer 侧保持原生存储路径 |
| 单机/本地多设备平台 | macOS 26.3（Darwin 25.3.0，ARM64） |
| 真实多设备平台 | Lima Ubuntu 24.04 ARM64 虚拟机集群；最多 20 节点；节点 CPU 为 2 vCPU（aarch64） |
| 校验实现与硬件路径 | `binary` 默认每条记录 `CRC32C(hw)`；L1 节点 `aarch64` CPU flags 包含 `crc32`，基准按 `-march=armv8-a+crc` 编译 |
| 真实多设备数据量 | 每节点 `records=20000`；节点规模 `5/10/15/20`；`repeats=3`，`warmup=1` |
| 真实多设备总样本 | 六个语义组 × 三种模式 × 节点扫描后，有效统计样本约 `54,000,000` 条日志（不含 warmup） |
| 真实多设备总日志字节 | 有效轮次聚合约 `4.601 GiB`（按 `cluster_total_bytes` 汇总） |
| 真实多设备空间口径 | 集群总空间；共享格式文档在每个模式下仅计算一次 |
| 空间交叉点附加扫描 | `records` 从 `1` 到 `100000` 共 13 个点；`repeats=2`；指标为 `total_bytes` |

图像呈现遵循“主图承载结论、辅图解释机制”的统一规则：每组实验至少保留一张可直接支撑结论的主图，并配一张差值或分解图用于说明原因，避免仅堆叠曲线而缺少结论指向。

本文以严格公平层作为主结论来源，其中 `syslog` 覆盖 single/multi/l1 三类场景，`ftrace` 覆盖真实多设备场景。工程近似层仅用于补充系统全链路代价解释。该分工可将“实现差异”与“平台路径差异”有效分离。

### 5.2 初始化竞争正确性（前置验证）

在进入性能对比之前，本文先验证共享格式文件并发初始化的正确性，以确保后续 single/multi/l1 指标建立在一致的并发语义之上。\textsuperscript{[1,4,15]}

1. 本地多进程初始化竞争。

为了回答“日志量增加到什么程度，Optbinlog 的空间性能会优于其他模式”这一问题，本文在严格语义对齐口径下新增 `N` 扫描实验：单设备单进程，模式固定为 `text_semantic_like / binary / peer native mode`，记录数扫描点为 `1,2,5,10,20,50,100,200,500,1000,5000,20000,100000`，每点 `repeats=2`、`warmup=0`，统计指标为 `total_bytes` 均值。交叉点定义为“最早满足 `binary_bytes <= peer_bytes` 的 `N`”。

| zephyr | 未出现（至 100000） | `-45.2` | 正 `N` 区间无交叉，binary 空间持续劣势 |
| ulog | `10` | `9.7` | `N≈10` 即出现交叉，后续保持空间正收益 |
| hilog | `50` | `29.7` | `N≈30～50` 出现交叉，小 `N` 阶段 peer 仍占优 |

### 5.7 理论—实证对照

综合 5.3～5.6 与第4章模型，可得到五点结论：

1. 空间模型与实测保持一致：`total_bytes` 随 `N` 近似线性增长，不同语义组对应不同斜率 `B_m`。`nanolog/zephyr` 组中 binary 空间持续为负，`syslog/ulog/hilog` 组中 binary 空间持续为正，l1 的 `ftrace` 组同样保持空间正收益。
2. 时间与吞吐交叉点具有“语义组相关、平台相关”特征，而非统一阈值。`nanolog/zephyr` 在 single 可转正，但在 l1 再次转负；`syslog/ulog` 在 l1 保持稳定正收益；`hilog` 呈现中间形态；`ftrace` 在 l1 时间维度持续负区。
3. 竞争项 $W_m(D,r)$ 在本地多设备与真实多设备中均可观测。本地多设备表现为“时间/吞吐振荡、空间斜率稳定”，真实多设备则进一步放大为“持续正区/持续负区/近持平”的结构性分化。
4. 真实多设备不是本地多设备的线性外推。l1 场景引入节点冷启动、VM 调度噪声与进程重建成本，使固定项 `A_m` 占比上升，并直接改变不同模式的收益分布。
5. 日志量扩展实验给出可操作阈值：`syslog/ulog` 在 `N≈10` 左右完成空间反超，`hilog` 反超点约在 `N≈30～50`，`nanolog/zephyr` 在 `N<=100000` 仍未出现空间交叉。

### 5.7.1 原因分析与消融佐证

为了更直接解释上述三层场景的排序差异，本文继续使用严格公平消融运行作为“原因分析层”，但不再把它作为主体结论数据集。该数据集给出的关键消融结果如下：

| zephyr | 0.703 | +22.60% |
| ulog | 0.713 | +4.59% |

如表\ref{tbl:ablation-prep}和表\ref{tbl:ablation-varstr}所示，结合严格对齐结果与消融结果，可以把原因归纳为三点：
1. **校验开销并非统一主导项**：本地五组中，`binary_nocrc` 的影响区间扩大到 `-2.82%～+22.60%`。这说明关闭校验并不会在所有语义组上自动带来收益，真正受益更明显的是 `zephyr` 这类紧凑定长路径。
2. **冷路径差异仍然存在，但不再只集中于文本型 peer**：`syslog` 的 `prep_ms` 仍达到 `8.214ms`，说明系统路径初始化仍然最重；而 `nanolog/zephyr/ulog/hilog` 的 `prep_ms` 已落在 `0.680～0.827ms`，说明单机主导差异更多开始转向热路径组织方式，而不是单纯的初始化成本。
3. **auto-varstr 收益具有明显语义组差异**：`zephyr/hilog` 中 `varstr` 或 `nocrc_varstr` 仍可能带来正收益，但 `nanolog/ulog/syslog` 中单独开启 `varstr` 反而可能转负，说明变长编码并非所有 schema 的统一最优策略，必须结合字段分布来判断。

### 5.8 证据链闭环与外推边界

本文主证据链由“最新严格公平层 + 初始化竞争验证 + 空间交叉点扫描”组成：
1. **严格公平层主数据**：`text_semantic_like / binary / peer native mode` 在 single、本地多设备与真实多设备三层结果。
2. **正确性验证数据**：本地与节点级初始化竞争，验证“单创建者 + 多复用者”并发语义。
3. **阈值验证数据**：`N=1～100000` 空间扫描，验证不同语义组的空间交叉点是否存在。

该证据链使结论具备三重可检验性：指标可复核、机制可解释、阈值可复现。工程近似层在本章中仅用于补充说明，不承担最终排序结论。

### 5.9 机制解释与应用启示

结合最新严格公平结果与消融数据，可将 Optbinlog 的分组结论概括为：
1. **稳定优势组（`syslog/ulog`）**：在 single 与真实多设备均保持清晰正收益，是当前最确定的应用受益区间。
2. **中间过渡组（`hilog`）**：空间长期占优，时间多为小幅占优，吞吐接近持平，体现“可优化但不失稳”的中间态。
3. **高压竞争组（`nanolog/zephyr`）与 tracing 组（`ftrace`）**：前者在当前口径下仍未跨越节点级交叉点，后者维持“空间正收益、时间吞吐负收益”。

由此可见，Optbinlog 的工程价值主要体现在三点：跨进程一致性强、语义扩展成本低、记录级可校验；其后续优化重点应继续聚焦于冷路径压缩、稳态缓存收益释放与批量提交效率提升。

## 6 结论与未来展望

### 6.1 研究结论

基于最新一版严格语义对齐数据，本文得到以下结论：

1. `binary` 相对 `text` 在全部语义组上保持稳定正收益，说明结构化二进制路径在高频写入场景下具备明确基线优势。
2. 在 peer 原生路径对比中，`syslog/ulog` 属于稳定优势组，`hilog` 属于中间过渡组，`nanolog/zephyr` 在当前节点级口径下仍未完成反超，`ftrace` 保持“空间正收益、时间吞吐负收益”。
3. 空间交叉点具有语义组相关性：`syslog/ulog` 在 `N≈10` 出现反超，`hilog` 在 `N≈50` 左右出现反超，`nanolog/zephyr` 在 `N<=100000` 仍未出现交叉。
4. 初始化竞争结果持续满足“单创建者 + 多复用者”约束，说明共享格式文件方案在本地与节点级并发下均保持正确性。

总体而言，理论推导与最新实证趋势一致。Optbinlog 更适用于长期在线、强调跨进程一致性与可追溯性的嵌入式网关与边缘控制场景。

### 6.2 当前局限

1. 受网络与平台限制，尚未完成外部系统官方工程的全栈编译复现（当前为代码级阅读 + 算法级原型复现）。
2. Linux 实验仍以 VM 为主，尚未覆盖 Raspberry Pi / i.MX 实机。
3. 严格公平语义套件已经扩展到 20 节点 VM 级“真实多设备”模拟，但仍未迁移到真实 ARM 板卡或 bare-metal Linux 实机。
4. 当前严格公平套件采用“新进程 + 单次写入”统计口径，因此新加入的进程内 shared/tag cache 收益尚未在长生命周期 steady-state 基准中充分量化。
5. 缺少 `perf` 级 CPU 微观指标，暂不能细粒度解释所有异常点。

### 6.3 未来展望

未来工作可从以下方向持续推进：
1. 在更广泛的平台与负载条件下开展验证，进一步提升结论的外部有效性与可迁移性。
2. 在保持语义一致性的前提下继续优化热路径与稳态路径，缩小与极短路径 peer 在特定场景下的差距。
3. 完善日志可靠性与可恢复性评估，形成覆盖损坏、截断与异常恢复过程的系统化证据。
4. 结合更细粒度观测手段（如 CPU 与调度行为分析），建立从机制到性能现象的更强解释闭环。

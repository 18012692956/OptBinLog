## 摘要

面向嵌入式与边缘设备，日志系统既要满足可观测性需求，又要控制存储、CPU 与并发初始化开销。传统文本日志可读性高，但在高频写入场景下存在体积大、解析慢和跨进程格式同步成本高等问题。本文围绕“事件格式驱动 + 二进制编码 + 共享内存元数据”提出并实现了一套轻量日志方案 Optbinlog：通过解析 eventlog 格式文件构建事件描述表，将描述表写入共享文件并采用偏移量访问，日志写入阶段仅执行按类型定长/变长截断编码，读出阶段依据共享格式表完成解码与校验。

实现上，本文基于 C 语言完成了事件解析模块、共享元数据模块、二进制读写模块及多进程初始化控制逻辑，并构建了单机场景、多设备并发场景、初始化竞态场景和 round-trip 正确性场景的自动化实验脚本。实验采用重复运行、IQR 异常值过滤及均值/中位数/p95/标准差/95% 置信区间统计。结果表明，在“高负载单机”实验（`records=80000`）中，本地平台上 binary 相比 text 的端到端时间下降 **39.07%**、总空间下降 **69.23%**、吞吐提升 **64.13%**；Linux 平台（基线设为 binary）上 ftrace 相比 binary 的端到端时间下降 **50.91%**、吞吐提升 **103.63%**。在多设备场景（1/2/4/8/12/16 设备，每设备 1000 条）中，binary 在空间上保持显著优势，而时间与吞吐受并发调度影响更明显；Linux 上 ftrace 在各设备档位下均保持高于 binary 的吞吐。初始化竞态测试（本地 `procs=32`）显示 `create_success=1`、`open_existing_ok=31`、`init_done=1`，IQR 过滤后平均初始化耗时为 **22.421 ms**，说明共享文件状态机在更高并发下仍能维持单创建者收敛特性。

本文工作验证了“共享格式表 + 二进制编码”在日志系统中的可行性与工程收益，并给出了跨平台测试与外部基线补齐方案。本文进一步从算法路径与数据结构出发推导了日志条数 `N`、设备数 `D` 对时间/空间/吞吐的理论趋势，并与实验结果逐项对照。后续将进一步补全 Linux x86_64 / ARM 嵌入式实机对照和截断日志异常检测能力。

关键词：二进制日志；共享内存；事件格式；并发初始化；性能优化

## Abstract

For embedded and edge systems, logging must balance observability and runtime overhead (storage, CPU, and concurrent initialization). Text-based logs are easy to read but often incur larger footprints and higher processing cost under high-frequency workloads. This thesis presents Optbinlog, a lightweight logging approach based on event-schema metadata in shared memory and binary encoding. Event definitions are parsed from eventlog files and materialized into a shared metadata file; runtime encoding/decoding uses offset-based access to avoid process-address dependency and reduce synchronization overhead.

The implementation includes an event parser, shared metadata manager, binary writer/reader, and concurrent initialization control in C. Automated benchmarks are built for single-node, multi-device concurrency, init-race, and round-trip correctness. Statistical reporting includes IQR outlier filtering and mean/median/p95/std/95% CI. In high-load single-node tests (`records=80000`), binary mode improves over text by **39.07% latency / 69.23% size / 64.13% throughput** on the local platform. On Linux (with binary as Linux baseline), ftrace improves over binary by **50.91% latency** and **103.63% throughput**. In multi-device tests (1/2/4/8/12/16 devices, 1000 records per device), space trends remain stable while time/throughput are more sensitive to concurrency scheduling. Init-race results with `procs=32` show stable one-writer/multi-reader convergence (`create_success=1`, `open_existing_ok=31`, `init_done=1`).

The study confirms the practical value of schema-shared binary logging and provides a reproducible workflow for further Linux x86_64 and ARM embedded validation. It also derives trend formulas from algorithmic/data-structure characteristics first, then validates them against measurements.

Keywords: binary logging; shared memory; event schema; concurrent initialization; performance optimization

## 第2章 需求分析与总体设计

### 2.1 功能需求

系统需满足以下功能：

1. 读取 eventlog 目录中的事件定义文件。
2. 将事件定义初始化到共享文件，供写入/读取进程复用。
3. 支持按事件 ID 查找格式并进行二进制写入与读回。
4. 支持异常输入（非法 tag、元素个数不匹配）检测。

### 2.2 非功能需求

1. **性能**：降低写入耗时与端到端耗时，提升吞吐。
2. **空间**：降低日志总字节数（含共享元数据成本）。
3. **并发**：多进程初始化过程仅允许一个创建者，其余进程可安全复用。
4. **可扩展**：新增事件类型时尽量不改核心读写流程。

### 2.3 总体方案

结合技术文档与现有实现，系统采用“离线格式描述 + 在线共享复用 + 运行时编码”的路径：

1. 事件格式层：沿用 Android EventLog 思想，将 `unit` 改为 `bits`，得到 `(name|type|bits)`。
2. 元数据层：将事件数组、位图与元素描述写入共享文件。
3. 编解码层：写入时按 `type/len` 顺序编码，读取时按共享格式解码并回调。
4. 评测层：通过 Python 脚本统一执行与统计，输出 JSON + SVG。

## 第3章 关键技术与迭代过程

本章依据《hikbinlog技术文档》中的开发流程整理关键迭代。

### 3.1 事件格式从 Android EventLog 到 Hikbinlog

原 EventLog 采用 `(name|type|unit)`，本文改为 `(name|type|bits)`，使编码长度由类型与 bit 宽度直接决定，便于压缩与快速序列化。类型压缩为 `L/D/S` 三类，简化编码分支。

### 3.2 元数据结构迭代

技术文档显示，结构从“每事件单独 struct”演化为“通用 EventTag + EventTagEle + 二级数组 + Bitmap”。该演化的意义在于：

1. 避免事件种类增多后结构体爆炸。
2. 通过 Bitmap 进行 tag 合法性判定。
3. 通过数组分段控制查找复杂度。

### 3.3 共享文件与偏移量访问

共享文件初版存储指针地址时会受到进程映射基址差异影响。最终方案使用“首地址 + 偏移”替代裸指针，核心收益：

1. 跨进程映射地址不同仍可正确访问。
2. 数据结构可序列化、可迁移。
3. 与初始化状态机结合后更适合并发场景。

### 3.4 查找策略迭代

查找路径从 switch-case 到排序二分，再到二维数组索引与偏移直访。最终方案使用：

1. `arr = tag_id / 100` 与 `idx = tag_id % 100` 进行快速定位。
2. Bitmap 校验 tag 合法。
3. 偏移量获取 `EventTag` 与 `EventTagEle`。

该路径将查找代价稳定在常数级，降低大规模事件表下的开销。

## 第4章 系统实现

### 4.1 模块划分

基于当前工程实现，核心模块如下：

1. `optbinlog_eventlog`：解析事件格式目录，构建 `OptbinlogTagList`。
2. `optbinlog_shared`：共享文件初始化、状态控制、tag 查找。
3. `optbinlog_binlog`：二进制写入/读取与回调。
4. `run_bench.py / run_multi_bench.py / run_init_race.py / run_roundtrip.py`：实验驱动与统计可视化。

### 4.2 关键数据结构

`demo/include/optbinlog_shared.h` 给出以下关键结构：

1. `OptbinlogBitmap`：记录 tag 合法位。
2. `OptbinlogEventTagEle`：元素类型与长度（字节）描述。
3. `OptbinlogEventTag`：事件级描述，包含元素偏移。
4. `OptbinlogSharedTag`：共享头，包含状态与偏移入口。

### 4.3 初始化并发控制

`optbinlog_shared_init_from_dir()` 的核心流程为：

1. 若共享文件存在，先尝试只读映射。
2. 若状态为 `INITIALIZING`，短暂等待并重试。
3. 若不存在则创建文件、填充结构、最后置 `INITIALIZED`。

该流程在竞态测试中体现出“单创建者 + 多复用者”特征（第5章给出数据）。

### 4.4 二进制读写器

`optbinlog_binlog_write()` 与 `optbinlog_binlog_read()` 通过共享 tag 缓存减少重复查找，按元素类型写入：

1. `L`：按 `len` 字节写入无符号整数。
2. `D`：固定 8 字节写入 double。
3. `S`：按字段长度写入并截断/补零。

该实现避免每个 tag 定制函数，减少维护成本。

### 5.2 评价指标与统计方法

1. 时间：`write_only_ms`、`end_to_end_ms`、`elapsed_ms`。
2. 空间：`bytes`、`shared_bytes`、`total_bytes`。
3. 吞吐：`throughput_*_rps`。
4. 稳定性：`mean/median/p95/std/95%CI`。
5. 异常值：IQR 过滤（默认系数 1.5）。

统计量定义（设过滤后样本为 `x_1, x_2, ..., x_n`）：

- 均值：`mean = (1/n) * Σ x_i`
- 标准差：`std = sqrt((1/(n-1)) * Σ (x_i - mean)^2)`
- 95%CI：`mean ± 1.96 * std / sqrt(n)`
- IQR：`IQR = Q3 - Q1`，保留区间 `[Q1 - 1.5*IQR, Q3 + 1.5*IQR]`
- p95：按线性插值分位点计算

改进率定义（与脚本实现一致）：

1. 时间改进率：`(baseline - target) / baseline * 100%`（越大越好）。
2. 空间节省率：`(baseline - target) / baseline * 100%`（越大越好）。
3. 吞吐提升率：`(target - baseline) / baseline * 100%`（越大越好）。

### 5.4 基于算法与数据结构的理论推导

本节只做抽象推导，不代入任何具体 eventlog 文件数据。所有常数先用符号表示，后续在 5.6 再代入实验实例。

#### 5.4.1 符号定义

1. `N`：单机日志条数。
2. `D`：并发设备数（进程数）。
3. `r`：每设备日志条数。
4. `T`：tag 类型总数。
5. `E_total`：所有 tag 的元素总数。
6. `e_k`：第 `k` 个 tag 的元素个数，`k=1..T`。
7. `l_{k,j}`：第 `k` 个 tag 第 `j` 个元素的编码字节数。
8. `p_k`：第 `k` 个 tag 被写入的概率（`Σ p_k = 1`）。
9. `H,B,ET,EE`：共享头、Bitmap、EventTag、EventTagEle 结构体大小。
10. `A`：数组段数（`tag_id/100` 分段数）。

#### 5.4.2 各模式算法路径与复杂度

1. `text`：字符串模板格式化 + `fputs`，每条一次格式化、一次追加写。
复杂度：`O(N)`。
2. `binary`：构造 `OptbinlogRecord` -> 按 schema 打包 -> CRC -> 批量写。  
复杂度：`O(N * ē)`，其中 `ē=Σ p_k e_k`；`ē` 有上界时可视为 `O(N)`。
3. `syslog`：每条一次 `syslog()` 调用。复杂度：`O(N)`。
4. `ftrace`：每条两次 `write()`（正文+换行）。复杂度：`O(N)`。

#### 5.4.3 日志条数 N 与空间趋势的纯理论推导

步骤1：先推导 `binary` 的单条字节。

`binary` 帧格式（写入函数）：

`frame = len(4) + payload + crc32(4)`

其中：

`payload = timestamp(8) + tag_id(2) + ele_cnt(1) + Σ_{j=1..e_k} l_{k,j}`

对 tag 分布取期望，有：

`E[payload] = 11 + Σ_{k=1..T} p_k * (Σ_{j=1..e_k} l_{k,j})`

故单条期望字节：

`B_bin = 4 + E[payload] + 4 = 19 + Σ_{k=1..T} p_k * (Σ_{j=1..e_k} l_{k,j})`

步骤2：推导 `binary` 的固定共享开销。

共享文件总字节：

`S_shared = 4 + H + A*B + T*ET + E_total*EE`

步骤3：得到 `binary` 总空间。

`S_binary(N) = N * B_bin + S_shared`

步骤4：推导 `text` 总空间。

设一条文本日志长度为随机变量 `L_text`，则

`B_text = E[L_text]`

`S_text(N) = N * B_text`

步骤5：推导 `syslog/ftrace` 总空间。

设原始行集合为 `L={line_1,...,line_M}`，每条写入 `line + '\n'`，则

`B_sys = E[|line|+1]`

`S_syslog(N) = N * B_sys`

`S_ftrace(N) = N * B_sys`

结论1（空间阶）：

1. 四种模式对 `N` 都是一阶线性。
2. `binary` 与其余模式的关键区别是存在 `S_shared` 固定项，因此小 `N` 时会被固定成本放大，大 `N` 时该项被摊薄。

#### 5.4.4 日志条数 N 与时间趋势的纯理论推导

对任意模式 `m`，端到端时间写成：

`T_m(N) = A_m + N * C_m`

其中：

1. `A_m`：一次性开销（初始化、打开资源、解析等）。
2. `C_m`：每条日志边际成本。

进一步拆解 `binary`：

`C_bin = c_alloc + c_pack + c_crc + c_write`

`A_bin = a_parse + a_shared_init + a_open`

`text/syslog/ftrace` 同理可写成 `A_m + N*C_m`，但各项构成不同。

结论2（时间阶）：

1. 单机时间随 `N` 线性增长。
2. 吞吐 `Q_m(N)=1000N/T_m(N)=1000/(C_m + A_m/N)`，当 `N` 增大时趋近 `1000/C_m`。

#### 5.4.5 设备数 D 与时间/空间/吞吐趋势的纯理论推导

多设备脚本中 `D` 个子进程并发写，主进程等待全部完成。设单设备任务时间：

`τ_m(r,D) = a_m + r*c_m + φ_m(D)`

其中 `φ_m(D)` 表示并发竞争引入的额外代价（调度、共享路径、内核队列等）。

总耗时可写为：

`E_m(D,r) = A_fork(D) + max_{i=1..D} τ_m^{(i)}(r,D)`

在同分布近似下：

`E_m(D,r) ≈ A_fork(D) + a_m + r*c_m + φ_m(D)`

空间：

`S_m(D,r) = D*r*B_m + I_m*S_shared`

其中 `I_m=1` 当且仅当模式使用共享元数据（此处为 `binary`）。

吞吐：

`Q_m(D,r) = 1000*D*r / E_m(D,r)`

若高并发区可近似 `E_m(D,r) ≈ β_m + α_m D`，则

`Q_m(D,r) = 1000*D*r/(β_m + α_m D)`，并在 `D->∞` 时趋于 `1000*r/α_m`（饱和上限）。

### 5.5 基于理论推导的预估结论

由 5.4 的抽象公式，在不代入任何具体数据时可得到：

1. 空间排序由 `B_m` 决定，`S_shared` 仅影响 `binary` 的小样本区。
2. `binary` 对 `text` 的空间节省率为：

`η_{bin/text}(N)=1-(B_bin + S_shared/N)/B_text`

其导数：

`dη_{bin/text}/dN = S_shared/(N^2 * B_text) > 0`

即 `N` 越大，`binary` 空间优势越接近极限值 `1-B_bin/B_text`。
3. `syslog` 与 `ftrace` 在当前实现的空间斜率理论上应一致（同样写原始行集合）。
4. 时间方面，若 `A_bin` 较大，则小 `N` 下 `binary` 可能不占优；当 `N` 足够大，比较重点转为 `C_m`。
5. 多设备吞吐随 `D` 的增长不会无限线性增加；竞争项 `φ_m(D)` 增大后会出现边际递减甚至突变。

### 5.6 理论预估与实验结果对比

#### 5.6.1 空间公式对照（理论 -> 实测）

本节才代入当前实验实例参数（由 eventlog 与样本行统计得到）：

1. 代入步骤A：计算 `B_bin`  
`B_bin = 19 + Σ p_k Σ l_{k,j}`  
其中 `l_{k,j}` 按实现映射为：`L/S -> bits/8`，`D -> 8`。  
本实验按均匀轮询 tag，得到 `Σ p_k Σ l_{k,j}=14.768`，故 `B_bin=33.768 B`。
2. 代入步骤B：计算 `S_shared`  
`S_shared = 4 + H + A*B + T*ET + E_total*EE`  
由 `max_tag=3249` 得 `A=floor(3249/100)+1=33`，并统计得到 `T=250,E_total=626`。  
代入 `H=56,B=13,ET=54,EE=33`，得  
`S_shared = 4 + 56 + 33*13 + 250*54 + 626*33 = 34647 B`。
3. 代入步骤C：计算 `B_text`  
由文本模板得：
`B_text = C_fixed + E[d(seq)] + E[d(code)] + E[d(temp)] + 1`。  
其中：
`C_fixed=81`；  
`E[d(seq)] = Σ_{k=1..19} k * P(10^{k-1} <= seq < 10^k) + 20*P(seq>=10^{19}) = 19.3977`；  
`E[d(code)] = (1*10 + 2*90 + 3*900 + 4*9000 + 5*90000)/100000 = 4.8889`；  
`E[d(temp)] = 0.125*4 + 0.875*5 = 4.875`。  
故 `B_text = 81 + 19.3977 + 4.8889 + 4.875 + 1 = 111.1616 B`。
4. 代入步骤D：计算 `B_sys`  
`B_sys = E[|line|+1] = 71.45 B`。

以 `N=80000` 为例：

1. `S_text = 80000*111.1616 = 8,892,925.24 B`，实测 `8,892,984 B`，误差 `0.0007%`。
2. `S_binary = 80000*33.768 + 34647 = 2,736,087 B`，实测 `2,736,087 B`，误差 `0%`。
3. `S_syslog = S_ftrace = 80000*71.45 = 5,716,000 B`，实测均为 `5,716,000 B`，误差 `0%`。

多设备 `D=1..16,r=1000` 时，`S_m(D,r)` 逐点对照仍一致（`text` 偏差在 `±0.011%` 内）。

#### 5.6.2 时间趋势对照（理论 -> 实测）

理论给出 `T_m(N)=A_m+N*C_m`。在 `N=20000/40000/80000/160000` 扫描中：

1. `text/binary/syslog/binary@linux/ftrace` 的时间-日志数关系均接近线性（`R²≈0.9976/1.0000/0.9995/0.9964/0.9998`）。
2. `T/N` 随 `N` 增大总体下降并趋稳，符合 `A_m/N` 摊薄过程。
3. 由实验反推边际成本（斜率）`C_m`，可得到模式间长期吞吐上限排序；该排序与“路径复杂度 + 调用开销”推断一致。

#### 5.6.3 关键不一致与原因分析

1. Linux 单机中 `ftrace` 的端到端时间显著优于 `binary`。  
理论上两者同为线性，差异来自 `A_m` 与常数项组成。实测中：
`binary`：`write_only=42.062 ms`、`prep=45.088 ms`、`e2e=89.076 ms`；  
`ftrace`：`write_only=41.406 ms`、`prep=2.316 ms`、`e2e=43.727 ms`。  
可见两者写出时间接近，但 `binary` 额外准备开销主导了总差异。
2. local 多设备在 `D=16` 出现突变（例如 text `elapsed` 从 `D=12` 的 `5.639 ms` 跳至 `13.776 ms`）。  
这不违背理论模型，属于 `L_m(D)`（调度与资源竞争项）在高并发区放大后的非线性现象。
3. Linux 多设备中 `ftrace` 吞吐始终高于 `binary`，但两者“每设备吞吐”均随 `D` 增大下降，符合 `Q_m(D,r)` 饱和趋势。  
`binary` 每设备吞吐由 `16161` 降到 `7526`；`ftrace` 由 `70139` 降到 `16480`。

### 5.7 初始化竞态测试（仅本地，procs=32, repeats=20）

2. `wait_events`：mean **6.158**，`wait_total_ms` mean **67.393 ms**。

## 第6章 结论与展望

### 6.1 结论

本文围绕嵌入式日志系统实现了共享格式元数据驱动的二进制日志方案，并完成了从数据结构、查找策略到并发初始化控制的系统化实现与验证。主要结论如下：

1. 在双平台单机高负载场景，local 端 `binary` 相比 `text` 在时间/吞吐/空间上同时占优；Linux 端 `ftrace` 相比 `binary` 在时间与吞吐上显著占优，但空间更大。
2. 在双平台多设备场景，空间随日志量/设备量增长呈稳定线性趋势；local 端在超高并发（`D=16`）出现明显抖动，Linux 端 `ftrace` 在各设备档位吞吐均高于 `binary`。
3. 初始化竞态实验（32 进程）表明共享状态机能稳定收敛到单创建者，但等待成本已具备统计显著性。
4. round-trip 与 bad-tag 检测通过，系统具备基础正确性保证。

### 6.2 后续工作

1. 按开题计划补齐 Linux x86_64 与 ARM（Raspberry Pi / i.MX）实机数据，并记录内核/文件系统/存储介质信息。
2. 引入更完整的外部基线（journald、Android EventLog 设备侧口径）并统一对照条件。
3. 统一采集 CPU 指标（如 `perf stat` 周期、上下文切换、cache miss）以解释多设备退化原因。
4. 完善截断日志检测、损坏恢复与异常路径测试覆盖。
5. 增加更长时长与更高并发压测，形成可复现实验矩阵。

### 6.3 下一步优化计划（对齐开题报告）

结合开题报告中 Android EventLog / Linux ftrace / 嵌入式平台目标，建议下一阶段优先级如下：

1. `P0`：补齐外部对照闭环。  
口径：`text`、`optbinlog binary`、`syslog/journald`、`ftrace`，可行时增加 Android EventLog 设备侧数据。
2. `P0`：多设备性能退化专项优化。  
重点：共享初始化锁等待、文件写入 flush 策略、进程调度与批量写策略。
3. `P1`：双平台实机复现。  
至少覆盖 x86_64 Linux 与 ARM Linux（Raspberry Pi / i.MX），并固化平台元信息。
4. `P1`：统计口径强化。  
除 mean/median/p95/std/95%CI 外，增加 p99、CPU 周期与上下文切换指标。
5. `P2`：工程化复现。  
保留 `run_thesis_suite.py` 为主入口，补充参数模板与自动报告导出。

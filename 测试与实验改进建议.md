# 基于开题报告的测试与实验改进建议

## 1. 结论概览

结合开题报告里可见的研究目标（如与 **Android EventLog / Linux ftrace** 对比、关注 **CPU/空间开销**、并面向 **Linux 嵌入式平台**）和当前仓库里的基准脚本，当前实验已经覆盖了“文本 vs 二进制”写入时间与体积对比，但在**对照组完整性、统计严谨性、正确性验证、可复现实验矩阵**上仍有提升空间。

## 2. 当前实验覆盖情况（基于现有代码）

- 单机单进程基准：`demo/run_bench.py` 调用 `optbinlog_bench`，输出 `elapsed_ms / bytes / shared_bytes / total_bytes / peak_kb`。  
- 多进程多设备场景：`demo/run_multi_bench.py` 调用 `optbinlog_multi_bench`，输出总时间和总字节。  
- 初始化竞态：`demo/run_init_race.py` 调用 `optbinlog_init_race`，统计 `create_success / open_existing_ok / init_done`，并绘制时序图。

## 3. 可改进点（按优先级）

### P0：补齐“对照组”以对齐开题报告目标

开题报告中有 Android EventLog / Linux ftrace 的关键词，但当前脚本只比较 text 与 binary。建议补齐三类对照：

1. `text`（现有）
2. `optbinlog binary`（现有）
3. `syslog/journald` 或 `ftrace`（新增）
4. 若可行，再加 Android EventLog（Android 设备侧）

> 这样论文中的“优于现有机制”结论才有外部基线支撑，而不是仅与自定义文本格式对比。

### P0：从“单次结果”升级为“统计结果”

目前脚本默认单次运行，建议改为每个场景重复 20~30 次，输出：

- `mean / median / p95 / std`
- 95% 置信区间
- 异常值处理（IQR 或 z-score）

并固定测试环境：CPU governor、核心绑定（`taskset`）、禁用后台扰动。避免“偶然一次跑得快”导致结论偏差。

### P0：加入“正确性回归”

当前基准偏性能，缺少强正确性验证。建议新增 round-trip 测试：

- 写入 binary -> 反序列化 -> 与原始记录逐字段比对
- 覆盖边界值：`bits=1/8/16/31/32/63/64`、空字符串、超长字符串、非法 tag_id
- 共享内存损坏/截断文件的异常路径测试

### P1：扩展指标维度（不仅是耗时和文件大小）

建议增加：

- 吞吐：records/s、MB/s
- 单条日志平均 CPU 周期（`perf stat`）
- 延迟分位：p50/p95/p99
- 内存细分：RSS + 匿名页 + 共享页
- 初始化阶段锁等待/重试次数（竞态成本）

这样可支撑“CPU 降低 xx%、空间节省 xx%”这类论文陈述。

### P1：把并发实验做成“参数扫描”

`run_multi_bench.py` 目前固定思路（devices 与 records）。建议做二维扫描：

- 进程数：1, 2, 4, 8, 16, 32
- 每进程日志速率：低/中/高

输出可扩展性曲线（吞吐-并发、时延-并发），更能体现架构上限与退化点。

### P1：平台覆盖对齐开题报告

开题报告提及 Raspberry Pi、NXP i.MX 等嵌入式平台关键词。建议至少形成两类平台结果：

- x86_64 Linux（开发机）
- ARM Linux（Raspberry Pi / i.MX）

并记录：内核版本、文件系统、存储介质（SD/eMMC/NVMe），避免跨平台结论失真。

已补充脚本：`demo/run_platform_suite.py`，可在每台目标机器上执行同一命令，自动采集并固化：

- 架构、内核版本、CPU、内存
- 结果目录所在文件系统类型与挂载点
- 存储介质推断（NVMe / MMC(SD/eMMC) / SATA / USB / HDD）
- `run_bench.py` 与 `run_multi_bench.py` 输出快照

建议在 x86 与 ARM 分别执行（示例）：

```bash
cd demo
OPTBINLOG_MULTI_MODES=text,binary,syslog,ftrace \
OPTBINLOG_SCAN_DEVICES=1,2,4 \
OPTBINLOG_SCAN_RECORDS_PER_DEVICE=500 \
python3 run_platform_suite.py --tag x86_nvme
```

```bash
cd demo
OPTBINLOG_MULTI_MODES=text,binary,syslog,ftrace \
OPTBINLOG_SCAN_DEVICES=1,2,4 \
OPTBINLOG_SCAN_RECORDS_PER_DEVICE=500 \
python3 run_platform_suite.py --tag rpi_sd
```

执行后结果位于：`demo/bench_platform/<tag>/platform_report.json`。  
论文中可直接引用其中 `platform.storage.filesystem`、`platform.storage.medium`、`platform.host.kernel_release` 字段，做跨平台对照表。

### P2：实验可复现工程化

建议新增 `experiments/` 目录统一：

- `config/*.yaml`：场景参数
- `run_all.py`：一键跑全套
- `analyze.py`：统计与画图
- `results/<date>/`：原始数据与图

同时在 README 写明“一键复现实验命令 + 预期耗时 + 依赖版本”。

## 4. 最小可落地改造路线（两周）

### 第 1 周
- 给现有三套脚本加“重复运行 + 统计输出（mean/p95/std）”。
- 增加 round-trip 正确性脚本与边界样例。
- 在 init race 中补充失败重试次数、等待时长统计。

### 第 2 周
- 新增 1 个系统级对照（建议先选 syslog/journald，成本较低）。
- 完成并发参数扫描与自动化绘图。
- 在 x86 + 1 个 ARM 平台各跑一轮，生成对比表。

## 5. 论文写作建议（避免答辩常见问题）

- 明确区分“工程收益”（更省空间、更快）与“学术结论”（在什么负载范围内成立）。
- 所有提升结论都标注场景条件（并发数、消息大小、平台）。
- 对不占优场景也要给出解释（例如低并发小消息时收益不明显）。

\section{绪论}\label{ux7eeaux8bba}

\subsection{研究背景}\label{ux7814ux7a76ux80ccux666f}

嵌入式系统日志常处于``高频写入、有限存储、多进程并发、实时约束''并存的环境。传统文本日志在可读性上有优势，但会带来：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  字符串格式化与解析开销高。
\item
  单条日志冗余字节多，长期空间成本高。
\item
  多进程共享格式信息时容易出现初始化竞争和一致性问题。
\end{enumerate}

因此，本文并不局限于 \texttt{text} 与 \texttt{binary} 的局部比较，而是将问题置于``理论分析、实现机制与分组对照实验''统一框架下进行讨论。

\subsection{研究问题}\label{ux7814ux7a76ux95eeux9898}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  现有日志系统的主流优化点是什么？是否仍有缺口？
\item
  Optbinlog 的实现路径能否覆盖这些缺口？
\item
  从算法与数据结构出发，\texttt{N} 与 \texttt{D} 的时间/空间/吞吐趋势应如何推导？
\item
  推导结论与实验结果哪些一致，哪些不一致，原因是什么？
\end{enumerate}

\subsection{本文工作}\label{ux672cux6587ux5de5ux4f5c}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  完成国内外日志系统调研，并定位代表性优化代码入口。
\item
  设计并实现 Optbinlog（事件格式解析、共享元数据、二进制读写、并发初始化状态机）。
\item
  建立抽象数学模型，推导不同模式在 \texttt{N}、\texttt{D} 维度下的性能趋势。
\item
  组织多轮实验（单机高负载、多设备模拟、真实多设备模拟（多台虚拟机节点）并发、init-race）并进行理论对照分析。
\end{enumerate}

\subsection{研究方法与工作中心}\label{ux7814ux7a76ux65b9ux6cd5ux4e0eux5de5ux4f5cux4e2dux5fc3}

围绕前述研究问题，本文将工作中心放在三条主线上： 1. \textbf{机制主线}：从日志系统算法与数据结构出发，不先下结论，先解释为什么会快/慢、为什么会大/小。 2. \textbf{对照主线}：不只做 \texttt{text} 对比，还和 \texttt{syslog/ftrace/nanolog/zephyr/ulog/hilog} 做同语义分组对照。 3. \textbf{验证主线}：统一统计口径与可复现实验脚本，按``理论推导 -\textgreater{} 单机 -\textgreater{} 多设备 -\textgreater{} 真实多设备模拟（多台虚拟机节点）''逐层验证。

\subsection{研究创新点}\label{ux7814ux7a76ux521bux65b0ux70b9}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  提出``共享 schema + 偏移访问''的用户态二进制日志实现，重点解决多进程初始化一致性与重复解析问题。
\item
  从算法与数据结构角度推导 \texttt{N} 与 \texttt{D} 对时间、空间和吞吐的影响，并给出可检验交叉点与饱和点表达式。
\item
  建立``工程近似层 + 严格公平层''两级实验体系：前者保留真实系统路径差异，后者统一 schema、统一生成值与统一平台，避免把语义体量差或输入来源差误判为实现优劣。
\item
  在真实多设备模拟（多台虚拟机节点）场景下验证初始化竞争与并发写入行为，形成可复现实验流程。
\end{enumerate}

\section{国内外研究调查与缺陷分析}\label{ux56fdux5185ux5916ux7814ux7a76ux8c03ux67e5ux4e0eux7f3aux9677ux5206ux6790}

\subsection{调研范围与方法}\label{ux8c03ux7814ux8303ux56f4ux4e0eux65b9ux6cd5}

本章基于官方文档、论文与代码仓库进行调研，聚焦``热路径优化机制''和``可复现实现入口''。 为保证后续章节的横向可比性，调研统一按四个维度展开：数据结构组织、关键算法路径、并发行为特征与工程部署边界。

调研对象如下： 1. Linux ftrace / kernel ring buffer。 2. LTTng。 3. Zephyr logging subsystem。 4. systemd-journald。 5. NanoLog（USENIX ATC 2018）。 6. RT-Thread ulog。 7. OpenHarmony HiLog Lite。

\subsection{国际代表系统与优化思路（代码级）}\label{ux56fdux9645ux4ee3ux8868ux7cfbux7edfux4e0eux4f18ux5316ux601dux8defux4ee3ux7801ux7ea7}

\subsubsection{Linux ftrace（内核追踪）}\label{linux-ftraceux5185ux6838ux8ffdux8e2a}

代码与文档显示其核心是 \texttt{per-CPU\ lockless\ ring\ buffer}。在 \texttt{ring\_buffer.c} 中，写入路径围绕``预留 -\textgreater{} 填充 -\textgreater{} 提交''组织，关键是把竞争局部化到 CPU 本地缓冲，降低全局锁竞争。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：每 CPU buffer + page 链式组织，维护 head/commit/tail。
\item
  算法逻辑：优先本地写入、批量可见性提交，读写阶段解耦。
\item
  场景定位：内核 tracing、高并发诊断；功能强但接入复杂度高于应用层日志库。
\end{enumerate}

\subsubsection{LTTng（Linux Tracing Toolkit Next Generation）}\label{lttnglinux-tracing-toolkit-next-generation}

LTTng 采用 tracepoint instrumentation + lockless buffer 管线，强调高吞吐追踪和离线分析。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：channel/sub-buffer 等追踪缓冲结构。
\item
  算法逻辑：将事件采集与后处理分层，追踪点开销可控。
\item
  场景定位：系统级性能分析与可观测平台，部署和运维要求较高。
\end{enumerate}

\subsubsection{Zephyr logging（RTOS 场景）}\label{zephyr-loggingrtos-ux573aux666f}

在 Zephyr \texttt{Kconfig.mode} 与 \texttt{log\_msg.c} 可见，关键机制是 deferred logging。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：\texttt{log\_msg} 对象 + 消息池（MPSC packet buffer）。
\item
  算法逻辑：前台线程只做封包入队，后台线程统一输出；支持 overflow/drop 策略。
\item
  场景定位：RTOS 中实时任务优先，要求前台日志路径尽量短、可控。
\end{enumerate}

\subsubsection{systemd-journald}\label{systemd-journald}

其 \texttt{JOURNAL\_FILE\_FORMAT} 与 \texttt{journald-server} 路径强调二进制对象与索引。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：header + objects + 哈希/索引。
\item
  算法逻辑：写入与检索并重，保证一致性与查询能力。
\item
  场景定位：通用 Linux 系统服务日志；对轻量嵌入式而言依赖偏重。
\end{enumerate}

\subsubsection{NanoLog（研究型高性能日志）}\label{nanologux7814ux7a76ux578bux9ad8ux6027ux80fdux65e5ux5fd7}

从 \texttt{NanoLogCpp17.h}、\texttt{RuntimeLogger.cc} 可见，NanoLog 核心是``编译期提取格式 + 运行期仅记录动态参数''。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：staging buffer、runtime logger 队列/缓冲。
\item
  算法逻辑：前台路径尽量只做参数打包，压缩与格式恢复转移到后台/离线。
\item
  场景定位：超高频日志、允许离线恢复文本的性能敏感场景。
\end{enumerate}

\subsection{国内生态系统与优化思路（代码级）}\label{ux56fdux5185ux751fux6001ux7cfbux7edfux4e0eux4f18ux5316ux601dux8defux4ee3ux7801ux7ea7}

\subsubsection{RT-Thread ulog}\label{rt-thread-ulog}

\texttt{ulog.c} 提供异步输出和多 backend 抽象，思路是``前台快写缓冲，后台慢速输出''。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：ringbuffer/rbb、backend 列表。
\item
  算法逻辑：异步线程统一调度输出，减轻业务线程阻塞。
\item
  场景定位：RT-Thread 设备侧调试与在线日志。
\end{enumerate}

\subsubsection{OpenHarmony HiLog Lite}\label{openharmony-hilog-lite}

HiLog Lite 在格式能力上做约束（减少复杂格式），换取更低资源开销。

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：轻量日志缓冲和简化格式存储。
\item
  算法逻辑：限制格式复杂度，降低格式化路径成本。
\item
  场景定位：资源受限设备的系统日志基础设施。
\end{enumerate}

\subsection{使用场景与工作中心差异}\label{ux4f7fux7528ux573aux666fux4e0eux5de5ux4f5cux4e2dux5fc3ux5deeux5f02}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  若目标是\textbf{内核态追踪}，\texttt{ftrace} 仍是优先选择；若目标是\textbf{应用层结构化日志}，Optbinlog 更匹配。

\item
  若系统强调\textbf{运维生态兼容}，\texttt{syslog/journald} 优势明显；若系统强调\textbf{嵌入式长期在线与空间效率}，Optbinlog 更具针对性。
\end{enumerate}

\subsection{现有方案缺陷归纳}\label{ux73b0ux6709ux65b9ux6848ux7f3aux9677ux5f52ux7eb3}

从嵌入式角度可抽象出四类常见缺口：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{格式化开销仍在热路径}：text/syslog 在高频场景下受字符串格式化与内核接口调用影响明显。
\item
  \textbf{元数据复用不足}：多进程系统若每进程重复解析格式，启动和内存成本上升。
\item
  \textbf{并发初始化一致性风险}：共享资源初次创建缺少状态机时，容易出现竞态。
\item
  \textbf{跨平台复现实验缺失}：只在单平台得到的结论容易失真，外推边界较弱。
\end{enumerate}

\subsection{Optbinlog 对缺陷的对应关系}\label{optbinlog-ux5bf9ux7f3aux9677ux7684ux5bf9ux5e94ux5173ux7cfb}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  用``事件格式文件 + 二进制编码''替代热路径字符串格式化。
\item
  用共享元数据文件持久化 tag/element 描述，进程间偏移复用。
\item
  用 \texttt{INITIALIZING\ -\textgreater{}\ INITIALIZED} 状态机控制并发初始化。
\item
  用统一脚本输出 mean/median/p95/std/95\%CI 与 IQR 过滤结果，支持多轮迭代对比。
\end{enumerate}

\section{Optbinlog 设计迭代与工程实现}\label{optbinlog-ux8bbeux8ba1ux8fedux4ee3ux4e0eux5de5ux7a0bux5b9eux73b0}

\subsection{设计目标}\label{ux8bbeux8ba1ux76eeux6807}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  降低单条日志热路径开销（时间）。
\item
  降低长期写入空间消耗（空间）。
\item
  支持多进程共享元数据与安全初始化（并发）。
\item
  对齐嵌入式可部署需求（工程可落地）。
\end{enumerate}

上述目标在后续章节中分别对应为可测指标：时间目标映射到 \texttt{elapsed/end\_to\_end}，空间目标映射到 \texttt{total\_bytes}，并发目标映射到初始化竞争正确性与节点级稳定性，工程目标映射到跨场景复现与部署可迁移性。

\subsection{事件格式与元数据结构}\label{ux4e8bux4ef6ux683cux5f0fux4e0eux5143ux6570ux636eux7ed3ux6784}

当前 Optbinlog 采用``共享元数据文件 + 进程内共享视图缓存 + 自描述二进制帧 + \texttt{CRC32C(hw)} + string-heavy 场景自动 \texttt{varstr}''的实现方式。

事件定义采用 \texttt{(name\textbar{}type\textbar{}bits)} 形式，字段类型简化为 \texttt{L/D/S}：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \texttt{L}：定长整数，按 schema 指定字节数写入。
\item
  \texttt{D}：双精度浮点，固定写入 8 字节。
\item
  \texttt{S}：字符串字段；当前实现同时支持固定宽度字符串编码和 \texttt{varstr} 编码。
\end{enumerate}

共享元数据文件内仍由以下结构组织：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \texttt{OptbinlogSharedTag}：共享文件头与初始化状态。
\item
  \texttt{OptbinlogBitmap}：tag 合法性位图。
\item
  \texttt{OptbinlogEventTag}：事件定义入口。
\item
  \texttt{OptbinlogEventTagEle}：元素类型与长度描述。
\end{enumerate}

在共享文件之上，系统进一步引入了\textbf{进程内共享视图缓存层}。该缓存按共享文件的 \texttt{dev/inode/size/mtime} 判定是否可复用，并缓存：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  已映射的共享文件视图。
\item
  每个 tag 的 \texttt{OptbinlogTagCacheEntry}。
\item
  schema 级统计量，例如 tag 数、总 payload 字节数、字符串字段总宽度、字符串字段数和最大字符串字段宽度。
\end{enumerate}

这些统计量会直接参与写路径中的 \texttt{varstr} 自动决策，因此它们属于当前热路径优化的一部分。

\subsection{写入/读取路径}\label{ux5199ux5165ux8bfbux53d6ux8defux5f84}

写入与读取路径遵循``先判定元数据状态、再执行编码/解码、最后进行一致性校验''的统一流程。

写入路径： 1. 通过 \texttt{acquire\_shared\_view()} 获取共享文件视图；若共享文件未变化，则直接复用进程内缓存的 tag cache。 2. 根据 schema 统计量判断是否启用 \texttt{varstr}；当字符串字段占比高或最大定长字符串字段较大时，默认切换到 \texttt{varstr}。 3. 依据 tag cache 顺序编码 payload。 4. 构造 4 字节自描述帧头，其中同时编码 \texttt{payload\_len}、\texttt{varstr} 标志位和校验类型。 5. 对单条记录计算校验值；默认采用 \texttt{CRC32C(hw)}，仅在硬件不可用时回退到软件实现。 6. 将 \texttt{frame\_header\ +\ payload\ +\ checksum} 写入用户态缓冲区，按批次落盘。

读取路径： 1. 读取 4 字节帧头，并解析 \texttt{payload\_len}、\texttt{varstr} 标志位与 \texttt{checksum\_type}。 2. 读取 payload 与校验字段，并按帧头声明的校验类型执行校验。 3. 按共享格式解码：整数和浮点按固定宽度读取，字符串则根据 \texttt{varstr} 标志选择``长度前缀 + 数据''或``固定宽度截断/补零''方式。 4. 将解码后的记录交给统一回调接口。

因此，当前记录格式可写作：

\[
\mathrm{frame} = \mathrm{frame\_header} \;\Vert\; \mathrm{payload} \;\Vert\; \mathrm{checksum}
\]

其中：

\[
\mathrm{frame\_header}
=
\mathrm{payload\_len}
\;\cup\;
\mathrm{varstr\_bit}
\;\cup\;
\mathrm{checksum\_type}
\]

\subsection{关键迭代}\label{ux5173ux952eux8fedux4ee3}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  从``每事件独立结构''迭代为``统一结构 + 位图 + 偏移表''。
\item
  从进程内指针共享迭代为``偏移量共享''（解决跨进程映射基址差异）。
\item
  从 switch-case 查找迭代为 \texttt{arr=id/100,\ idx=id\%100} 的常数级路径。
\item
  从软件 \texttt{CRC32} 迭代为默认 \texttt{CRC32C(hw)}；在 ARMv8 与 SSE4.2 平台优先使用硬件指令，软件实现只作为回退路径。
\item
  新增自描述帧头，使 \texttt{CRC32\ /\ CRC32C\ /\ none} 与 \texttt{fixed-string\ /\ varstr} 可以在同一读取逻辑中兼容。
\item
  新增进程内共享视图缓存和 tag cache，把共享文件 reopen、tag 遍历与 schema 统计从稳态热路径迁出。
\item
  新增 schema 级自动 \texttt{varstr} 策略：当字符串字段占比高时，从固定宽度字符串编码切换到``长度前缀 + 实际长度''编码。
\item
  增加并发初始化状态机和 init-race 压力验证，保证共享文件只被单创建者初始化，其余进程或节点复用既有元数据。
\end{enumerate}

综合以上迭代，当前实现可归纳为三条关键收敛点：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{校验语义与热路径代价兼顾}：系统保留每条记录校验，同时通过 \texttt{CRC32C(hw)} 将校验成本压缩到更适合热路径的水平。
\item
  \textbf{共享元数据的稳态收益进入真实路径}：共享视图缓存和 tag cache 使 schema 解析与 tag 遍历不再重复进入每次写入。
\item
  \textbf{字符串主导 schema 的冗余写入被抑制}：在 string-heavy 场景下自动启用 \texttt{varstr}，把固定宽度补零和无效拷贝降到更低。
\end{enumerate}

\subsection{与实际场景的匹配性}\label{ux4e0eux5b9eux9645ux573aux666fux7684ux5339ux914dux6027}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  小样本 \texttt{N}：共享文件打开、共享视图校验、文件打开和缓冲准备等固定项占比仍然较高，时间优势不一定立刻出现。
\item
  中大样本 \texttt{N}：固定项被摊薄，自描述帧头和 \texttt{CRC32C(hw)} 的额外代价收敛到常数级，binary 的时间和空间优势更稳定。
\item
  string-heavy schema：自动 \texttt{varstr} 会显著减少补零写入和冗余字节，因此空间优势更容易显现。
\item
  高并发 \texttt{D}：即使校验和 schema 解析成本被压低，调度、文件竞争和写出路径仍会主导吞吐峰值与回落点。
\item
  多进程/多节点部署：共享 schema、进程内缓存复用和节点级初始化竞争正确性已经形成闭环，因此该设计更贴近真实工程部署。
\end{enumerate}

因此，第4章将把上述实现行为进一步抽象为 \texttt{A\_m/C\_m/B\_m/W\_m} 等参数形式，便于与第5章实验结果逐项对照。

\subsection{工程化读取能力补全}\label{ux5de5ux7a0bux5316ux8bfbux53d6ux80fdux529bux8865ux5168}

为提升实验数据复核效率，本文在写入与解码库之外补充了 \texttt{optbinlog\_read} 工具，用于``共享格式文件 + 二进制日志文件''的直接可视化读取。该工具具备三项能力： 1. 按 tag 与字段名输出记录，避免只看到匿名值序列。 2. 支持 \texttt{table/jsonl} 两种格式，便于命令行排错与自动化分析脚本复用。 3. 支持 \texttt{-\/-limit} 与 \texttt{-\/-summary}，可快速截取样本并统计各 tag 分布。

该工具把``论文中的统计结果''与``底层二进制记录''建立了可追溯链路，使实验结论不只停留在汇总图表层，而能够直接回溯到具体记录级证据。

\section{理论模型与数学推导}\label{ux7406ux8bbaux6a21ux578bux4e0eux6570ux5b66ux63a8ux5bfc}

本章先做抽象推导，不代入具体事件文件数值。 为保证推导可解释且可验证，本文采用以下前提：在单次统计窗口内事件分布近似稳定；各模式在同一场景下使用一致的输入语义；模型参数以期望意义解释而非单点极值解释。

\subsection{符号定义}\label{ux7b26ux53f7ux5b9aux4e49}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(N\)：总日志条数。
\item
  \(D\)：并发设备数（或并发进程数）。
\item
  \(r\)：每设备日志条数，满足 \(N = D \cdot r\)。
\item
  \(k\)：事件类型（tag）索引，\(k \in \{1,\dots,T\}\)。
\item
  \(p_k\)：事件类型 \(k\) 的概率，满足 \(\sum_k p_k = 1\)。
\item
  \(e_k\)：事件 \(k\) 的字段数。
\item
  \(l_{k,j}\)：事件 \(k\) 第 \(j\) 个字段编码字节数。
\item
  \(L_k = \sum_{j=1}^{e_k} l_{k,j}\)：事件 \(k\) 的有效载荷字节数。
\item
  \(A_m\)：模式 \(m\) 的固定时间开销（初始化、打开文件、预热）。
\item
  \(C_m\)：模式 \(m\) 的单条边际时间开销。
\item
  \(B_m\)：模式 \(m\) 的单条边际空间开销。
\item
  \(S_{0,m}\)：模式 \(m\) 的固定空间开销（例如共享元数据）。
\item
  \(Q_m\)：吞吐（records/s）。
\end{enumerate}

\subsection{从算法与数据结构到单条代价}\label{ux4eceux7b97ux6cd5ux4e0eux6570ux636eux7ed3ux6784ux5230ux5355ux6761ux4ee3ux4ef7}

为避免``只看结果不看机制''，先把每种模式拆成操作级成本。以下表达式均以单条记录期望开销为核心，再扩展到总量关系。

\subsubsection{text}\label{text}

单条记录包含： 1. 读取输入字段。 2. \texttt{snprintf}/拼接格式串。 3. \texttt{write} 到文件。

时间可写为：

\[
t_{\text{text}}(k) = t_{\text{parse}}(k) + t_{\text{fmt}}(k) + t_{\text{write}}(\mathrm{bytes}_{\text{text}}(k))
\]

空间可写为：

\[
b_{\text{text}}(k) = b_{\text{prefix}} + b_{\text{delim}} + b_{\text{num}}(k) + b_{\text{str}}(k)
\]

其中 \(b_{\text{num}}(k)\) 是数字转十进制后的字符长度，随数值大小缓慢变化。

\subsubsection{binary（optbinlog）}\label{binaryoptbinlog}

单条记录包含： 1. tag 查找（数组/偏移访问）。 2. 按字段类型编码 payload。 3. 计算校验值。 4. 写帧 \texttt{len\ +\ payload\ +\ crc}。

时间：

\[
t_{\text{bin}}(k) = t_{\text{lookup}} + t_{\text{encode}}(k) + t_{\text{checksum}}(L_k) + t_{\text{write}}(4 + h_{\text{bin}} + L_k + 4)
\]

空间：

\[
b_{\text{bin}}(k) = 4 + h_{\text{bin}} + L_k + 4
\]

固定空间：

\[
S_{0,\text{bin}} = S_{\text{shared}}
\]

若 schema 触发 \texttt{varstr}，则字符串字段的有效载荷不再按固定宽度写出，而是写为``长度前缀 + 实际长度''：

\[
L_k^{(\text{varstr})}
=
\sum_{j \in \mathcal{N}_k} l_{k,j}
\;+\;
\sum_{j \in \mathcal{S}_k} \left( 2 + \hat{l}_{k,j} \right),
\qquad
\hat{l}_{k,j} \le l_{k,j}
\]

\subsubsection{syslog / ftrace}\label{syslog-ftrace}

抽象为``系统日志接口 + 内核路径''：

\[
t_{\text{sys}}(k) = t_{\text{format\_sys}}(k) + t_{\text{syscall}} + t_{\text{kernel\_path}} + t_{\text{write\_kernel}}
\]

\[
t_{\text{ftrace}}(k) = t_{\text{format\_ftrace}}(k) + t_{\text{trace\_write}} + t_{\text{ring\_or\_file\_path}}
\]

空间：

\[
b_{\text{sys}}(k) = b_{\text{sys\_header}} + b_{\text{sys\_payload}}(k)
\]

\[
b_{\text{ftrace}}(k) = b_{\text{ftrace\_header}} + b_{\text{ftrace\_payload}}(k)
\]

\subsubsection{nanolog\_like / zephyr\_deferred\_like / ulog\_async\_like / hilog\_lite\_like}\label{nanolog_like-zephyr_deferred_like-ulog_async_like-hilog_lite_like}

\texttt{nanolog\_like}（固定字段二进制）：

\[
t_{\text{nano}} = t_{\text{pack\_fixed}} + t_{\text{memcpy}} + t_{\text{write}}
\]

\[
b_{\text{nano}} = \text{const}
\]

\texttt{zephyr\_deferred\_like}（前台入队 + 后台批刷）：

\[
t_{z,\text{front}} = t_{\text{enqueue}} + t_{\text{signal}}
\]

\[
t_{z,\text{back}} = \frac{t_{\text{dequeue}} + t_{\text{batch\_write}}(k_b)}{k_b}
\]

\[
t_z = t_{z,\text{front}} + t_{z,\text{back}}
\]

\[
b_z = b_{\text{queue\_node}} + b_{\text{fixed}} + \frac{b_{\text{batch\_meta}}}{k_b}
\]

\texttt{ulog\_async\_like}（异步队列 + 文本 backend）：

\[
t_{\text{ulog}} =
t_{\text{enqueue}} + t_{\text{cond\_signal}}
+
\frac{t_{\text{dequeue}} + t_{\text{format\_text}} + t_{\text{backend\_write}}}{k_b}
\]

\[
b_{\text{ulog}} = b_{\text{queue\_node}} + b_{\text{text\_payload}}
\]

\texttt{hilog\_lite\_like}（受限字段 + 轻量头）：

\[
t_{\text{hilog}} = t_{\text{compact\_format}} + t_{\text{write}}(h_{\text{lite}} + p_{\text{lite}})
\]

\[
b_{\text{hilog}} = b_{\text{header\_lite}} + b_{\text{payload\_lite}}
\]

\subsection{\texorpdfstring{空间随 \texttt{N} 的详细推导}{空间随 N 的详细推导}}\label{ux7a7aux95f4ux968f-n-ux7684ux8be6ux7ec6ux63a8ux5bfc}

\subsubsection{期望单条字节}\label{ux671fux671bux5355ux6761ux5b57ux8282}

对任意模式 \(m\)，定义：

\[
B_m = \mathbb{E}[b_m(k)] = \sum_k p_k \, b_m(k)
\]

则总空间：

\[
S_m(N) = S_{0,m} + N B_m
\]

这是严格的一阶线性模型，斜率为 \(B_m\)，截距为 \(S_{0,m}\)。

\subsubsection{binary 的显式表达}\label{binary-ux7684ux663eux5f0fux8868ux8fbe}

由 4.2.2 得：

\[
b_{\text{bin}}(k) = 8 + h_{\text{bin}} + L_k
\]

\[
B_{\text{bin}} = 8 + h_{\text{bin}} + \sum_k p_k L_k
\]

\[
S_{\text{bin}}(N) = S_{\text{shared}} + N \left(8 + h_{\text{bin}} + \sum_k p_k L_k\right)
\]

\subsubsection{节省率与导数}\label{ux8282ux7701ux7387ux4e0eux5bfcux6570}

binary 相对模式 \(x\) 的空间节省率：

\[
\eta_{\text{bin}/x}(N) = 1 - \frac{S_{\text{bin}}(N)}{S_x(N)}
\]

若 \(S_{0,x}=0\)，则

\[
\eta_{\text{bin}/x}(N) = 1 - \frac{B_{\text{bin}} + S_{\text{shared}}/N}{B_x}
\]

求导：

\[
\frac{d\eta_{\text{bin}/x}}{dN} = \frac{S_{\text{shared}}}{N^2 B_x} > 0
\]

说明 \(N\) 越大，\(S_{\text{shared}}/N\) 越小，binary 空间优势越稳定。

\subsubsection{空间交叉点}\label{ux7a7aux95f4ux4ea4ux53c9ux70b9}

当比较 \(\text{bin}\) 与 \(x\)：

\[
S_{\text{bin}}(N) \le S_x(N)
\]

\[
S_{\text{shared}} + N B_{\text{bin}} \le S_{0,x} + N B_x
\]

若 \(B_x > B_{\text{bin}}\)，可得交叉点：

\[
N^*_{\text{space}}(\text{bin},x) = \frac{S_{\text{shared}} - S_{0,x}}{B_x - B_{\text{bin}}}
\]

当 \(N > N^*_{\text{space}}\) 时，binary 空间一定优于 \(x\)。

\subsection{\texorpdfstring{时间随 \texttt{N} 的详细推导}{时间随 N 的详细推导}}\label{ux65f6ux95f4ux968f-n-ux7684ux8be6ux7ec6ux63a8ux5bfc}

\subsubsection{从单条时间到总时间}\label{ux4eceux5355ux6761ux65f6ux95f4ux5230ux603bux65f6ux95f4}

定义：

\[
C_m = \mathbb{E}[t_m(k)] = \sum_k p_k \, t_m(k)
\]

则总时间：

\[
T_m(N) = A_m + N C_m
\]

该形式直接对应实验中的 \texttt{elapsed/write\_only/end\_to\_end} 指标。

\subsubsection{吞吐公式与单调性}\label{ux541eux5410ux516cux5f0fux4e0eux5355ux8c03ux6027}

\[
Q_m(N) = \frac{1000N}{T_m(N)} = \frac{1000}{C_m + A_m/N}
\]

对 \(N\) 求导：

\[
\frac{dQ_m}{dN} = \frac{1000A_m}{N^2\left(C_m + A_m/N\right)^2} > 0
\]

说明在模型中吞吐随 \(N\) 单调上升并渐近：

\[
\lim_{N \to \infty} Q_m(N) = \frac{1000}{C_m}
\]

在 \(A_m>0\) 与 \(C_m>0\) 条件下，吞吐增速会随 \(N\) 增大而下降，表现为``先快后慢''的收益递减，这与实验中大 \(N\) 后增益变缓一致。

\subsubsection{时间交叉点（binary 何时优于其他模式）}\label{ux65f6ux95f4ux4ea4ux53c9ux70b9binary-ux4f55ux65f6ux4f18ux4e8eux5176ux4ed6ux6a21ux5f0f}

比较 \(\text{bin}\) 与 \(x\)：

\[
T_{\text{bin}}(N) \le T_x(N)
\]

\[
A_{\text{bin}} + N C_{\text{bin}} \le A_x + N C_x
\]

当 \(C_x > C_{\text{bin}}\) 时：

\[
N^*_{\text{time}}(\text{bin},x) = \frac{A_{\text{bin}} - A_x}{C_x - C_{\text{bin}}}
\]

解释： 1. 若 \(A_{\text{bin}}\) 较大（共享初始化重），小 \(N\) 可能不占优。 2. 随 \(N\) 增大，\(NC\) 项主导，只要 \(C_{\text{bin}} < C_x\)，最终会跨过交叉点。

\subsubsection{\texorpdfstring{\texttt{C\_m} 的字段复杂度展开}{C\_m 的字段复杂度展开}}\label{c_m-ux7684ux5b57ux6bb5ux590dux6742ux5ea6ux5c55ux5f00}

为把``常数项差异''写清楚，可把边际开销展开到字段数级别：

\[
C_m = c_{m,0} + c_{m,1}\mathbb{E}[e_k] + c_{m,2}\mathbb{E}[L_k] + c_{m,3}\mathbb{E}[\mathrm{syscall}_m]
\]

其中： 1. \(\mathbb{E}[e_k]\) 决定字段遍历成本（编码循环次数）。 2. \(\mathbb{E}[L_k]\) 决定 memcpy、校验和写入字节相关成本。 3. \(\mathbb{E}[\mathrm{syscall}_m]\) 反映模式是否强依赖系统调用路径。

可得典型关系： 1. text/syslog：\(c_{m,1}\) 与 \(c_{m,3}\) 较大（格式化 + syscall）。 2. binary：\(c_{m,1}\) 中等，\(c_{m,2}\) 受字段编码与校验影响，另有 \(A_{\text{bin}}\) 较大（共享文件和缓存准备）。 3. nanolog\_like：\(\mathbb{E}[e_k]\) 近似常数且小，\(\mathbb{E}[\mathrm{syscall}_m]\) 低，\(C_m\) 最小。 4. zephyr\_deferred\_like：前台 \(C\) 小，但高并发时等待项会转移到 \(W_m(D,r)\)。 5. ulog\_async\_like：前台 \(C\) 小，但后台文本格式化与后端写入使 \(W_m(D,r)\) 与 \(C_m\) 的常数项上升。 6. hilog\_lite\_like：通过限制格式能力降低 \(\mathbb{E}[e_k]\) 与 \(\mathbb{E}[L_k]\)，通常较 text/syslog 更轻。

\subsection{\texorpdfstring{多设备 \texttt{D} 的详细推导}{多设备 D 的详细推导}}\label{ux591aux8bbeux5907-d-ux7684ux8be6ux7ec6ux63a8ux5bfc}

\subsubsection{总时间分解}\label{ux603bux65f6ux95f4ux5206ux89e3}

设每设备写 \texttt{r} 条：

\[
E_m(D,r) = A_{\text{spawn}}(D) + A_m + rC_m + W_m(D,r)
\]

其中 \(W_m\) 为竞争等待项（锁、队列、内核资源、I/O 饱和）。

\subsubsection{队列化近似（解释吞吐饱和）}\label{ux961fux5217ux5316ux8fd1ux4f3cux89e3ux91caux541eux5410ux9971ux548c}

把共享瓶颈抽象成单服务台，服务率 \(\mu_m\)（records/s），到达率 \(\lambda = D\nu\)（\(\nu\) 为单设备提交率）。

稳定条件：

\[
\rho = \frac{\lambda}{\mu_m} < 1
\]

等待时间近似随 \(\rho\) 增大而快速上升，可抽象为：

\[
W_m(D,r) \sim r f_m(\rho),
\qquad
f_m'(\rho) > 0,
\qquad
f_m''(\rho) > 0
\]

因此当 \(D\) 接近

\[
D_{\text{sat},m} \approx \frac{\mu_m}{\nu}
\]

吞吐进入平台区；超过后出现明显回落。

\subsubsection{常用二次近似}\label{ux5e38ux7528ux4e8cux6b21ux8fd1ux4f3c}

在工程拟合中常写作：

\[
E_m(D,r) \approx \beta_m + rC_m + \alpha_m D + \gamma_m D^2
\]

总吞吐：

\[
Q_m(D,r) = \frac{1000Dr}{E_m(D,r)}
\]

当 \(\gamma_m > 0\) 时，\(D\) 过大必然导致分母增长过快，出现峰值后下降。

进一步对 \(D\) 求导：

\[
\frac{dQ_m}{dD}
\propto
\left(\beta_m + rC_m + \alpha_m D + \gamma_m D^2\right)
-
D\left(\alpha_m + 2\gamma_m D\right)
\]

整理得：

\[
\frac{dQ_m}{dD} \propto \beta_m + rC_m - \gamma_m D^2
\]

故吞吐峰值点近似为：

\[
D_{\text{peak},m} = \sqrt{\frac{\beta_m + rC_m}{\gamma_m}}
\]

这给出了可直接与实验扫描结果比对的``理论峰值设备数''。

\subsubsection{\texorpdfstring{空间随 \texttt{D}}{空间随 D}}\label{ux7a7aux95f4ux968f-d}

\[
S_m(D,r) = S_{0,m} + DrB_m
\]

因此空间对 \(D\) 严格线性，差异主要由斜率 \(B_m\) 与截距 \(S_{0,m}\) 决定。

\subsection{模式间理论排序与可检验预测}\label{ux6a21ux5f0fux95f4ux7406ux8bbaux6392ux5e8fux4e0eux53efux68c0ux9a8cux9884ux6d4b}

\subsubsection{异语义比较（最短路径口径）}\label{ux5f02ux8bedux4e49ux6bd4ux8f83ux6700ux77edux8defux5f84ux53e3ux5f84}

由单条代价可得典型关系：

\[
B_{\text{nano}} \sim B_z \lesssim B_{\text{hilog}} < B_{\text{bin}} < B_{\text{sys}} \sim B_{\text{text}} \lesssim B_{\text{ulog}}
\]

\(C_{\text{nano}}\) 常最小，\(C_{\text{hilog}}\) 次之；\(C_z\) 与 \(C_{\text{ulog}}\) 受队列和后台行为影响；\(C_{\text{text}}/C_{\text{sys}}/C_{\text{ftrace}}\) 受格式化与系统路径影响更大。

可检验预测： 1. nanolog\_like 在时间与吞吐上常呈数量级优势。 2. hilog\_lite\_like 在``受限语义''下应明显优于 text/syslog。 3. ulog\_async\_like 在吞吐未饱和且后台轻负载时有机会受益；若后台文本后端成为瓶颈，可能慢于 binary/text。 4. 上述优势不代表语义等价下仍保持同倍率。

\subsubsection{语义对齐比较（公平口径）}\label{ux8bedux4e49ux5bf9ux9f50ux6bd4ux8f83ux516cux5e73ux53e3ux5f84}

当统一为 \texttt{len\ +\ payload\ +\ crc} 并统一字段集时：

\[
B_{\text{sem}}(\text{bin}) \approx B_{\text{sem}}(\text{nano}) \approx B_{\text{sem}}(\text{zephyr})
\]

\(C_{\text{sem}}\) 的差异主要来自实现常数（编码循环、队列同步、缓存命中和校验实现），不再是语义体量差异。

可检验预测： 1. 空间差异收敛到小常数级。 2. 时间差异收敛到同一量级，而非数量级差异。

\subsection{与第5章指标的映射关系}\label{ux4e0eux7b2c5ux7ae0ux6307ux6807ux7684ux6620ux5c04ux5173ux7cfb}

为了让推导可被实验验证，建立公式到指标的映射：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T_m(N)\) 对应 \texttt{elapsed\_ms} 或 \texttt{end\_to\_end\_ms} 的均值趋势。
\item
  \(Q_m\) 对应 \texttt{throughput\_rps} / \texttt{throughput\_e2e\_rps}。
\item
  \(S_m\) 对应 \texttt{total\_bytes}。
\item
  \(N^*_{\text{time}}\) 与 \(N^*_{\text{space}}\) 对应``某场景下 binary 由劣转优''的阈值。
\item
  \(D_{\text{sat},m}\) 对应``多设备扫描中吞吐峰值后回落''的拐点。
\end{enumerate}

本章给出的是``由算法与数据结构直接推出的理论趋势''；第5章再代入实测数据检验这些趋势是否成立，以及偏离原因。

\subsection{本章小结}\label{ux672cux7ae0ux5c0fux7ed3}

本章完成了三项工作：一是把不同日志模式统一到可比较的时间与空间参数形式；二是给出 \texttt{N} 与 \texttt{D} 两个核心自变量下的交叉点、峰值点与线性斜率含义；三是建立理论参数与实验指标的映射关系。基于这些结果，第5章将按``单机 -\textgreater{} 本地多设备 -\textgreater{} 真实多设备模拟 -\textgreater{} 日志量扫描''的顺序进行逐层验证。

\subsection{实验设计与证据组织}\label{ux5b9eux9a8cux8bbeux8ba1ux4e0eux8bc1ux636eux7ec4ux7ec7}

本章按``实验口径定义 -\textgreater{} 分场景结果 -\textgreater{} 理论对照 -\textgreater{} 原因解释''的顺序展开，避免将语义体量差异误判为实现差异。实验主线分为两层：工程近似层回答``真实系统路径代价如何排序''；严格公平层回答``同语义、同平台、同统计口径下 Optbinlog 与 peer 的本征差异''，统一以 \texttt{text\_semantic\_like\ /\ binary\ /\ peer\ semantic\_like} 为比较对象，覆盖以下三类场景： 1. 单机高负载（single）。 2. 本地多设备模拟（\texttt{D=5,10,20,50} 扫描）。 3. 真实多设备模拟（多台虚拟机节点）并发（\texttt{5/10/15/20} 节点，每节点一台设备）。

\subsection{理论---实证对照}\label{ux7406ux8bbaux5b9eux8bc1ux5bf9ux7167}

综合 5.2～5.5 与第4章模型，可得到五点结论：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  空间模型与实测一致：\texttt{total\_bytes} 随 \texttt{N} 线性增长，不同语义组对应不同 \texttt{B\_m}。在 \texttt{nanolog/zephyr} 组中，binary 相对 peer 的空间仍为负；在 \texttt{ulog/hilog/syslog} 组中，binary 在 single、multi 与 l1 三层口径下均为正收益；在 l1 的 \texttt{ftrace} 组中，binary 空间同样保持稳定正收益。
\item
  时间交叉点是``按语义组出现''的，而不是``按模式统一出现''的。\texttt{ulog/hilog/syslog} 在 single 与 multi 场景总体保持正收益；\texttt{zephyr} 在本地与真实多节点场景长期负区；\texttt{nanolog} 在 single 转正但在 multi/l1 波动并多点为负；\texttt{ftrace} 在 l1 时间维度持续深负。
\item
  竞争项 \texttt{W\_m(D,r)} 在本地多设备与真实多设备场景中都可被直接观察到。本地多设备出现的``点位振荡''与真实多设备出现的``持续负区/持续正区''，共同说明竞争项与冷路径项会在不同平台层级被放大为不同形态：前者更像局部抖动，后者更像结构性偏置。
\item
  ``真实多设备''不是简单复制``本地多设备''。L1 场景额外引入了节点级冷启动、VM 调度噪声和每节点新进程启动，因此 \texttt{A\_m} 的占比会明显抬升；这正是 \texttt{zephyr/nanolog} 在 L1 更难转正、\texttt{ulog/hilog/syslog} 在三指标上仍可保持正收益，以及 \texttt{ftrace} 呈现``空间优先但时延劣势''的主要原因。
\item
  日志量增长实验进一步给出了空间交叉点阈值：\texttt{ulog/hilog/syslog} 在 \texttt{N≈10} 附近即可完成 \texttt{binary\ \textless{}=\ peer} 的空间反超，而 \texttt{nanolog/zephyr} 在 \texttt{N\textless{}=100000} 仍未出现交叉点，说明``空间何时反超''同样是语义组相关结论。
\end{enumerate}

\subsection{证据链组织与外推边界}\label{ux8bc1ux636eux94feux7ec4ux7ec7ux4e0eux5916ux63a8ux8fb9ux754c}

为了让结论形成``可复核、可解释、可外推''的证据链，本文将数据集按用途组织为三类： 1. \textbf{工程近似层}：保留 \texttt{syslog/ftrace} 以及 \texttt{*\_like} 路径，观察真实工程风格系统的整体代价排序。 2. \textbf{严格公平层}：统一使用 \texttt{text\_semantic\_like\ /\ binary\ /\ peer\ semantic\_like}，在相同 schema、相同字段值和相同平台上报告单机、本地多设备与真实多设备结果。 3. \textbf{原因分析层}：保留 \texttt{binary\_hotpath\ /\ binary\_nocrc\ /\ binary\_varstr} 等消融模式，用于解释当前实现中的主要时间来源和空间来源。

其中，工程近似层回答``真实系统路径如何排序''，严格公平层回答``同语义条件下 Optbinlog 与 peer 如何分化''，原因分析层回答``差异主要来自哪些结构性开销''。三类数据分工明确，可避免把部署差异、语义差异与实现差异混写在同一结论中。

因此，严格公平层主结论由 \texttt{nanolog/zephyr/ulog/hilog/syslog} 在 single/multi/l1 三层结果与 \texttt{ftrace} 在 l1 结果共同支撑；工程近似层仅承担工程部署视角的补充说明。

\subsection{机制解释与应用启示}\label{ux673aux5236ux89e3ux91caux4e0eux5e94ux7528ux542fux793a}

结合工程近似对照、严格对齐套件和消融结果，可将 \texttt{Optbinlog} 的差异点归纳为三层： 1. \textbf{文本扩展型语义组}：\texttt{ulog} 的主要矛盾是文本扩展与记录膨胀，因此 binary 在 single、local multi 和 real multi-node 三层场景中都保持稳定优势；\texttt{hilog} 在 single 与真实多设备中也保持明显正收益，但在本地高设备数点位会被短记录热路径压缩到近似持平。这仍然是 Optbinlog 当前最稳的受益区间。 2. \textbf{极短热路径语义组}：\texttt{nanolog/zephyr} 已经把热路径压得很短。对于这类 peer，Optbinlog 的剩余通用开销主要来自 schema 访问、字段遍历、frame 组织和冷启动，因此本地多设备与真实多设备都更容易出现负区或振荡；空间扫描也显示这两组在 \texttt{N\textless{}=100000} 范围内仍不存在交叉点。 3. \textbf{平台与并发层}：进入真实多设备后，\texttt{W\_m(D,r)} 和冷启动项会把细小常数项放大，因此``单机反超''不应被直接外推为``节点级并发立即反超''。\texttt{syslog} 在该层体现为``三指标均可转正''，\texttt{ftrace} 体现为``空间正收益但时间吞吐深负''，说明平台路径差异会直接改变三指标的收益方向。

Optbinlog 的应用优势主要体现在： 1. \textbf{跨进程一致性}：共享 schema + 偏移访问保证多进程解码一致，适合网关/边缘设备的多服务部署。 2. \textbf{工程可迁移性}：不依赖编译期代码生成链路，新增日志事件只需更新 schema 文件，适合迭代频繁的业务系统。 3. \textbf{可验证性}：二进制帧包含长度与 per-record CRC32C，损坏检测与回归测试路径清晰，便于工程质量控制。 4. \textbf{对文本型 peer 的稳定优势}：\texttt{ulog/hilog} 三层场景的对齐结果表明，binary 对文本扩展更重的 peer 风格路径具有稳定时间、空间和吞吐优势。 5. \textbf{优化路径清晰}：实验已经把剩余开销收敛到 \texttt{cold-path\ +\ 通用打包路径\ +\ 稳态缓存收益释放不足}，后续优化可以围绕长生命周期进程、steady-state fast path 和批量提交策略展开。

完整分析表、最新可视化和关键结论汇总见严格对齐套件生成的报告；工程近似层的补充说明见实验补充分析报告。

\section{结论与展望}\label{ux7ed3ux8bbaux4e0eux5c55ux671b}

\subsection{研究结论}\label{ux7814ux7a76ux7ed3ux8bba}

本文形成了``调研 -\textgreater{} 设计 -\textgreater{} 建模 -\textgreater{} 验证 -\textgreater{} 解释''的完整研究链条：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  调研层：识别了代表系统的共性优化机制（deferred、ring buffer、结构化二进制、热路径迁移）及其适用边界。
\item
  设计层：Optbinlog 通过共享 schema 与偏移访问实现跨进程一致性，并以自描述帧与硬件 CRC32C 保持可校验性。
\item
  理论层：建立了 \texttt{N}、\texttt{D} 与时间/空间/吞吐之间的关系模型，给出可检验的交叉点与饱和点预测。
\item
  实验层：在 single、本地多设备与真实多设备模拟三层场景完成严格语义对齐验证，并通过初始化竞争和消融实验解释差异来源。
\end{enumerate}

从结果上看，\texttt{binary} 相对 \texttt{text} 在本轮纳入的各语义组上均为正收益；对 \texttt{ulog/hilog/syslog} 的优势在三层场景中总体稳定；\texttt{ftrace} 在真实多设备中表现为``空间正收益、时间吞吐负收益''。空间扫描显示 \texttt{ulog/hilog/syslog} 在 \texttt{N≈10} 即出现反超，而 \texttt{nanolog/zephyr} 在 \texttt{N\textless{}=100000} 仍未出现空间交叉。上述结论与第 4 章模型保持一致，说明 Optbinlog 更适合``长期在线、版本持续演进、强调跨进程一致性''的嵌入式网关与边缘控制器场景。

\subsection{当前局限}\label{ux5f53ux524dux5c40ux9650}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  受网络与平台限制，尚未完成外部系统官方工程的全栈编译复现（当前为代码级阅读 + 算法级原型复现）。
\item
  Linux 实验仍以 VM 为主，尚未覆盖 Raspberry Pi / i.MX 实机。
\item
  严格公平语义套件已经扩展到 20 节点 VM 级``真实多设备''模拟，但仍未迁移到真实 ARM 板卡或 bare-metal Linux 实机。
\item
  当前严格公平套件采用``新进程 + 单次写入''统计口径，因此新加入的进程内 shared/tag cache 收益只在代码层完成接入，尚未通过长生命周期 steady-state 基准充分量化。
\item
  缺少 \texttt{perf} 级 CPU 微观指标，暂不能细粒度解释所有异常点。
\end{enumerate}

\subsection{未来展望}\label{ux672aux6765ux5c55ux671b}

未来工作可从以下方向持续推进： 1. 在更广泛的平台与负载条件下开展验证，进一步提升结论的外部有效性与可迁移性。 2. 在保持语义一致性的前提下继续优化热路径与稳态路径，缩小与极短路径 peer 在特定场景下的差距。 3. 完善日志可靠性与可恢复性评估，形成覆盖损坏、截断与异常恢复过程的系统化证据。 4. 结合更细粒度观测手段（如 CPU 与调度行为分析），建立从机制到性能现象的更强解释闭环。

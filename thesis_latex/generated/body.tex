\section{绪论}\label{ux7eeaux8bba}

\subsection{研究背景}\label{ux7814ux7a76ux80ccux666f}

随着嵌入式设备、边缘网关和轻量控制节点持续承担更复杂的在线业务，日志已经不再只是离线排障时的附属产物，而是系统运行观测、故障定位和行为复核的重要基础设施。\textsuperscript{[4,12,14,20]} 与通用服务器环境相比，这类设备通常同时受到 CPU 性能、内存容量、持久化空间和实时响应要求的约束，因此日志系统不仅要``能记录''，还要尽可能缩短热路径、控制空间增量，并在长期运行中保持稳定的并发行为。

传统文本日志在可读性、工具兼容性和运维生态方面具有明显优势，但在高频写入场景下，字符串格式化、冗余字符存储和重复解析会持续推高时间与空间成本。对于多进程协作或多节点部署场景，如果各执行单元分别解析格式信息、独立维护元数据，不仅启动成本难以收敛，还可能在共享资源首次创建时引入初始化竞争和状态不一致问题。\textsuperscript{[1,3,7,8]}

围绕降低日志开销这一目标，现有研究已经形成了多条技术路径，例如编译期模板提取、无锁缓冲、异步批刷和离线恢复文本等。然而，这些方案往往分别强调极短热路径、高吞吐追踪或后处理能力，对于``共享格式元数据、跨进程一致性与结构化语义可恢复''这一组工程需求，尚缺少统一的机制设计与系统化验证。基于此，本文将研究重点放在面向嵌入式部署的结构化二进制编码与共享元数据机制上，并尝试在统一的理论---实证框架中回答其性能边界与适用条件。

\subsection{研究目标与问题定义}\label{ux7814ux7a76ux76eeux6807ux4e0eux95eeux9898ux5b9aux4e49}

基于上述背景，本文希望解决的并不是单一的``写得更快''问题，而是在语义完整性、并发一致性和综合开销之间寻找可解释、可验证的工程折中。围绕这一目标，本文聚焦以下四个核心问题：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  现有代表性日志系统在热路径优化、并发组织与空间控制方面分别采用了哪些机制，其适用边界是什么。
\item
  Optbinlog 的共享元数据与二进制编码路径能否在保持语义完整性的前提下弥补现有方案缺口。
\item
  在日志条数 \(N\) 与并发设备（节点）数 \(D\) 变化时，时间、空间与吞吐应呈现怎样的理论趋势与交叉关系。
\item
  理论预测与实验观测在何处一致、何处偏离，偏离由哪些实现常数项与系统路径因素导致。
\end{enumerate}

\subsection{主要贡献}\label{ux4e3bux8981ux8d21ux732e}

围绕上述问题，本文的主要工作与贡献可归纳为以下四点：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  设计并实现面向嵌入式多进程场景的 Optbinlog 系统，形成由共享 schema、偏移访问、自描述帧和初始化状态机组成的完整实现方案。
\item
  建立由 \(N\) 与 \(D\) 驱动的统一分析模型，对时间、空间和吞吐的变化趋势、交叉点与竞争放大效应给出可检验表达。
\item
  在统一统计规则和严格语义对齐条件下，完成单机、本地多设备、真实多设备代理场景、空间交叉点扫描与初始化竞争验证，形成闭环证据链。
\item
  基于实验结果识别出不同 peer 的分组特征与适用边界，说明 Optbinlog 在 \thesisterm{syslog/ulog} 场景中的稳定优势、在 \thesisterm{hilog} 场景中的过渡特征，以及在 \thesisterm{nanolog/zephyr} 场景下仍需继续优化的方向。
\end{enumerate}

\subsection{研究方法与技术路线}\label{ux7814ux7a76ux65b9ux6cd5ux4e0eux6280ux672fux8defux7ebf}

本文采用``问题驱动、机制设计先行、分层验证跟进''的研究路径。与单纯围绕实验结果组织论文不同，本文更强调先解释代价来源，再讨论实验排序。具体而言，研究方法可概括为三条主线：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{机制分析主线}：以算法与数据结构为起点，先解释代价来源，再进行性能比较。
\item
  \textbf{公平对照主线}：在保留工程近似层的同时，引入严格语义对齐层，避免将语义差异误判为实现差异。
\item
  \textbf{理论---实证闭环主线}：统一统计口径与复现实验脚本，按``理论推导→单机→多设备→真实多设备''逐层验证。
\end{enumerate}

\subsection{创新点与研究价值}\label{ux521bux65b0ux70b9ux4e0eux7814ux7a76ux4ef7ux503c}

在工程论文写作中，``创新''不仅体现在提出新机制，也体现在能否建立与实验一致的解释框架。本文的创新点与研究价值主要体现在以下几个方面：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  提出面向用户态的``共享 schema + 偏移访问''二进制日志机制，缓解多进程初始化一致性与重复解析问题。
\item
  给出 \(N\)、\(D\) 驱动下的时间/空间/吞吐统一推导框架，并提供可直接检验的交叉点与峰值点表达。
\item
  建立``工程近似层 + 严格公平层''的双层证据体系，提高结论的可解释性与可迁移性。
\item
  在真实多设备节点级并发场景下完成共享格式文件与初始化竞争验证，形成可复现的工程化实验流程。
\end{enumerate}

\section{国内外研究现状与问题归纳}\label{ux56fdux5185ux5916ux7814ux7a76ux73b0ux72b6ux4e0eux95eeux9898ux5f52ux7eb3}

\subsection{研究对象与评估维度}\label{ux7814ux7a76ux5bf9ux8c61ux4e0eux8bc4ux4f30ux7ef4ux5ea6}

本章的目标不是简单罗列相关系统，而是回答两个与本文设计直接相关的问题：现有日志系统主要在什么层面降低开销，以及这些机制为什么不足以直接覆盖本文关注的共享元数据场景。为此，本文基于官方文档、公开论文与代码仓库开展调研，重点关注``热路径优化方式''和``可复现实验所能对应的实现依据''。\textsuperscript{[1,3-4,7-8,12,14]} 为保证后续比较具有可比性，所有系统统一从四个维度展开：数据结构组织、关键算法路径、并发行为特征与工程部署边界。

调研对象如下： 1. Linux ftrace / kernel ring buffer。 2. LTTng。 3. Zephyr logging subsystem。 4. systemd-journald。 5. NanoLog（USENIX ATC 2018）。 6. RT-Thread ulog。 7. OpenHarmony HiLog Lite。

\subsection{国际代表系统：机制与实现路径（代码级）}\label{ux56fdux9645ux4ee3ux8868ux7cfbux7edfux673aux5236ux4e0eux5b9eux73b0ux8defux5f84ux4ee3ux7801ux7ea7}

\subsubsection{Linux ftrace（内核追踪）}\label{linux-ftraceux5185ux6838ux8ffdux8e2a}

代码与文档显示其核心是 \thesisterm{per-CPU lockless ring buffer}。在 \thesisterm{ring\_buffer.c} 中，写入路径围绕``预留 \textrightarrow{} 填充 \textrightarrow{} 提交''组织，关键是把竞争局部化到 CPU 本地缓冲，降低全局锁竞争。\textsuperscript{[1-2,15]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：每 CPU buffer + page 链式组织，维护 head/commit/tail。
\item
  算法逻辑：优先本地写入、批量可见性提交，读写阶段解耦。
\item
  场景定位：内核 tracing、高并发诊断；功能强但接入复杂度高于应用层日志库。
\end{enumerate}

\subsubsection{LTTng（Linux Tracing Toolkit Next Generation）}\label{lttnglinux-tracing-toolkit-next-generation}

LTTng 采用 tracepoint instrumentation + lockless buffer 管线，强调高吞吐追踪和离线分析。\textsuperscript{[3]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：channel/sub-buffer 等追踪缓冲结构。
\item
  算法逻辑：将事件采集与后处理分层，追踪点开销可控。
\item
  场景定位：系统级性能分析与可观测平台，部署和运维要求较高。
\end{enumerate}

\subsubsection{Zephyr logging（RTOS 场景）}\label{zephyr-loggingrtos-ux573aux666f}

在 Zephyr \thesisterm{Kconfig.mode} 与 \thesisterm{log\_msg.c} 可见，关键机制是 deferred logging。\textsuperscript{[4-6]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：\thesisterm{log\_msg} 对象 + 消息池（MPSC packet buffer）。
\item
  算法逻辑：前台线程只做封包入队，后台线程统一输出；支持 overflow/drop 策略。
\item
  场景定位：RTOS 中实时任务优先，要求前台日志路径尽量短、可控。
\end{enumerate}

\subsubsection{systemd-journald}\label{systemd-journald}

其 \thesisterm{JOURNAL\_FILE\_FORMAT} 与 \thesisterm{journald-server} 路径强调二进制对象与索引。\textsuperscript{[7,16]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：header + objects + 哈希/索引。
\item
  算法逻辑：写入与检索并重，保证一致性与查询能力。
\item
  场景定位：通用 Linux 系统服务日志；对轻量嵌入式而言依赖偏重。
\end{enumerate}

\subsubsection{NanoLog（研究型高性能日志）}\label{nanologux7814ux7a76ux578bux9ad8ux6027ux80fdux65e5ux5fd7}

从 \thesisterm{NanoLogCpp17.h}、\thesisterm{RuntimeLogger.cc} 可见，NanoLog 核心是``编译期提取格式 + 运行期仅记录动态参数''。\textsuperscript{[8-10]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：staging buffer、runtime logger 队列/缓冲。
\item
  算法逻辑：前台路径尽量只做参数打包，压缩与格式恢复转移到后台/离线。
\item
  场景定位：超高频日志、允许离线恢复文本的性能敏感场景。
\end{enumerate}

\subsection{国内代表系统：机制与实现路径（代码级）}\label{ux56fdux5185ux4ee3ux8868ux7cfbux7edfux673aux5236ux4e0eux5b9eux73b0ux8defux5f84ux4ee3ux7801ux7ea7}

\subsubsection{RT-Thread ulog}\label{rt-thread-ulog}

\thesisterm{ulog.c} 提供异步输出和多 backend 抽象，思路是``前台快写缓冲，后台慢速输出''。\textsuperscript{[12-13]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：ringbuffer/rbb、backend 列表。
\item
  算法逻辑：异步线程统一调度输出，减轻业务线程阻塞。
\item
  场景定位：RT-Thread 设备侧调试与在线日志。
\end{enumerate}

\subsubsection{OpenHarmony HiLog Lite}\label{openharmony-hilog-lite}

HiLog Lite 在格式能力上做约束（减少复杂格式），换取更低资源开销。\textsuperscript{[14]}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  数据结构：轻量日志缓冲和简化格式存储。
\item
  算法逻辑：限制格式复杂度，降低格式化路径成本。
\item
  场景定位：资源受限设备的系统日志基础设施。
\end{enumerate}

\subsubsection{二进制日志优化研究特征与 Optbinlog 对比}\label{ux4e8cux8fdbux5236ux65e5ux5fd7ux4f18ux5316ux7814ux7a76ux7279ux5f81ux4e0e-optbinlog-ux5bf9ux6bd4}

结合前文调研对象与公开实现路径，现有``二进制日志优化研究''大体可以归纳为三类技术取向： 1. \textbf{热路径极短化}：通过编译期提取静态模板、运行期只记录动态参数，最大化降低调用点常数开销（典型如 NanoLog）。\textsuperscript{[8-10]} 2. \textbf{并发写出解耦化}：通过无锁环形缓冲或 deferred 队列，将前台写入与后台落盘拆分，降低前台阻塞（典型如 ftrace/LTTng/Zephyr，以及 ETW 采集优化工具 Kellect）。\textsuperscript{[1,3-4,15,21]} 3. \textbf{后处理集中化}：把文本恢复、检索或压缩能力后移到离线/后台，以换取在线写入吞吐（典型如 NanoLog、集群日志无损压缩研究与 Binlog 工程方案）。\textsuperscript{[3,8,22-23]}

对应地，这类研究的优化目标通常集中在两点：一是尽可能压缩单位日志写入代价，尤其是热路径延迟；二是在高并发条件下维持吞吐稳定，避免锁竞争或后台拥塞显著放大。相比之下，Optbinlog 并不追求单一指标上的极端最优，而是希望在较低写入开销的基础上，同时保留跨进程共享语义、记录级校验和在线恢复能力。

\subsection{典型优化代码入口（可追溯）}\label{ux5178ux578bux4f18ux5316ux4ee3ux7801ux5165ux53e3ux53efux8ffdux6eaf}

为保证比较分析具有可追溯性，本文在代码级调研中选取了一组具有代表性的实现入口与官方格式说明，并据此建立``设计机制---实现路径---实验指标''之间的对应关系。这里所说的``代码入口''，并非简单的资料索引，而是用于识别各系统优化重点、解释实验差异来源以及限定复现边界的直接依据。

从分析作用看，这些入口主要服务于三个层面。其一，\thesisterm{ring\_buffer.c}、\thesisterm{log\_msg.c}、\thesisterm{RuntimeLogger.cc} 等文件体现了写入热路径的组织方式，例如缓冲结构、前后台分工、模板展开和记录提交流程，因此可用于解释时间开销与吞吐表现的差异。其二，\thesisterm{JOURNAL\_FILE\_FORMAT}、\thesisterm{ulog.c} 以及 HiLog Lite 相关文档反映了记录头部、字段布局、存储约束和恢复方式，可用于分析单位记录字节开销及空间斜率差异。其三，这些入口共同界定了本文的复现边界：第5章并不是对所有外部系统进行完整移植，而是在可控环境下提取其代表性机制，构造 native-like 对照实现。\textsuperscript{[1,4-5,7-10,12-16]}

基于上述处理，后文关于``共性不足''和``针对性设计''的论述都可以回溯到具体实现路径，而非停留在抽象层面的经验判断。这种写法有助于增强比较分析的证据基础，并为第5章实验结果的机制解释提供可核查支撑。

\subsection{现有方案的共性不足}\label{ux73b0ux6709ux65b9ux6848ux7684ux5171ux6027ux4e0dux8db3}

综合前述调研，可以看出不同系统虽然优化重点各异，但若从嵌入式多进程部署这一目标出发，仍存在四类共性不足：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{热路径优化目标较为单一}：部分方案重点压缩调用点延迟，但没有同时考虑结构化语义、共享元数据和记录级校验之间的平衡。
\item
  \textbf{共享元数据复用机制不足}：多进程环境下若各进程独立解析事件格式，启动与内存成本难以收敛，也不利于长期稳定运行。
\item
  \textbf{共享资源初始化缺乏显式一致性约束}：当格式文件或共享结构需要首次创建时，若缺少明确的状态机和协调过程，容易出现竞态。
\item
  \textbf{统一公平比较框架较少}：许多系统结论建立在各自原生语义和原生平台之上，缺少在相同语义体量与统一统计口径下的直接比较。
\end{enumerate}

\subsection{Optbinlog 的针对性设计}\label{optbinlog-ux7684ux9488ux5bf9ux6027ux8bbeux8ba1}

围绕上述缺口，Optbinlog 的设计并非简单叠加现有优化手段，而是针对嵌入式共享场景作出如下回应：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  用``事件格式文件 + 二进制编码''替代热路径中的字符串格式化，使稳态写入更多退化为字段遍历、编码与批量写出。
\item
  用共享元数据文件持久化 tag 和 element 描述，并通过偏移访问实现跨进程复用，避免指针语义带来的映射不一致问题。
\item
  用 \thesisterm{INITIALIZING} \textrightarrow{} \thesisterm{INITIALIZED} 状态机约束共享资源首次创建过程，保证单创建者与多复用者之间的并发语义清晰可检验。
\item
  用统一实验脚本输出 \thesisterm{mean/median/p95/std/95\%CI} 及 IQR 过滤后的统计结果，使理论推导、实现机制与实验结论能够在同一口径下对照。
\end{enumerate}

\subsection{复现边界与结论适用范围}\label{ux590dux73b0ux8fb9ux754cux4e0eux7ed3ux8bbaux9002ux7528ux8303ux56f4}

为了避免结论外推过度，本文在此明确说明复现边界。整体工作遵循``代码阅读先行、实验复现跟进''的顺序开展：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  已完成 ftrace、Zephyr、NanoLog、ulog 等代表性系统的官方代码级阅读，并据此提取与本文比较直接相关的机制特征。
\item
  受平台和环境限制，本文无法在当前工作站上完成所有外部系统的全栈编译与原生部署，尤其无法复现完整 Linux tracing 生态。
\item
  已在可控环境中完成 \thesisterm{nanolog\_like}、\thesisterm{zephyr\_deferred\_like}、\thesisterm{ulog\_async\_like}、\thesisterm{hilog\_lite\_like} 及其语义对齐版本的算法级复现，第5章所有定量结论均建立在该边界内。
\end{enumerate}

上述约束在第5章实验方法中均有显式记录，用于区分``工程可复现边界''和``算法层结论边界''。因此，本文的排序结论仅在相应实现口径与实验条件下成立，不对未覆盖平台和未复现路径作超范围外推。

\section{Optbinlog 系统设计与工程实现}\label{optbinlog-ux7cfbux7edfux8bbeux8ba1ux4e0eux5de5ux7a0bux5b9eux73b0}

本章围绕``从机制到实现、从实现到可验证结果''的逻辑展开。具体而言，先给出系统目标，再说明事件格式与共享元数据组织；随后描述写入与读取流程，并进一步讨论关键优化收敛路径、工程适配边界以及面向掉电场景的恢复机制。该安排对应第4章模型参数与第5章实验指标，可形成完整证据链。

\subsection{设计目标}\label{ux8bbeux8ba1ux76eeux6807}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  降低单条日志热路径开销（时间）。
\item
  降低长期写入空间消耗（空间）。
\item
  支持多进程共享元数据与安全初始化（并发）。
\item
  对齐嵌入式可部署需求（工程可落地）。
\end{enumerate}

上述目标在后续章节中分别对应为可测指标：时间目标映射到 \thesisterm{elapsed/end\_to\_end}，空间目标映射到 \thesisterm{total\_bytes}，并发目标映射到初始化竞争正确性与节点级稳定性，工程目标映射到跨场景复现与部署可迁移性。

\subsection{事件格式与元数据结构}\label{ux4e8bux4ef6ux683cux5f0fux4e0eux5143ux6570ux636eux7ed3ux6784}

当前 Optbinlog 采用``共享元数据文件 + 进程内共享视图缓存 + 自描述二进制帧 + \thesisterm{CRC32C(hw)} + string-heavy 场景自动 \thesisterm{varstr}''的一体化实现路径；其事件定义思路与通用事件日志标签机制保持一致。\textsuperscript{[11,20]}

\begin{figure}[H]
\centering
\includesvg[width=0.94\linewidth]{./demo/results/paper_dataset/supplement/figures/new_fig3_1_architecture.svg}
\caption{Optbinlog 系统总体架构}
\label{fig:optbinlog-design-overview}
\end{figure}

如图\ref{fig:optbinlog-design-overview}所示，系统以共享格式文件为中心，把``schema 解析、tag 组织、字段统计''前移到共享与缓存层，稳态写入只保留编码、帧构造、校验与批量落盘这四类关键操作。该组织方式直接对应第4章中的``固定项与边际项分离''思想。

事件定义采用 \thesisterm{(name\textbar{}type\textbar{}bits)} 形式，字段类型简化为 \thesisterm{L/D/S}：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \thesisterm{L}：定长整数，按 schema 指定字节数写入。
\item
  \(D\)：双精度浮点，固定写入 8 字节。
\item
  \thesisterm{S}：字符串字段；当前实现同时支持固定宽度字符串编码和 \thesisterm{varstr} 编码。
\end{enumerate}

共享元数据文件内仍由以下结构组织：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \thesisterm{OptbinlogSharedTag}：共享文件头与初始化状态。
\item
  \thesisterm{OptbinlogBitmap}：tag 合法性位图。
\item
  \thesisterm{OptbinlogEventTag}：事件定义入口。
\item
  \thesisterm{OptbinlogEventTagEle}：元素类型与长度描述。
\end{enumerate}

在共享文件之上，系统进一步引入了\textbf{进程内共享视图缓存层}。该缓存按共享文件的 \thesisterm{dev/inode/size/mtime} 判定是否可复用，并缓存：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  已映射的共享文件视图。
\item
  每个 tag 的 \thesisterm{OptbinlogTagCacheEntry}。
\item
  schema 级统计量，例如 tag 数、总 payload 字节数、字符串字段总宽度、字符串字段数和最大字符串字段宽度。
\end{enumerate}

这些统计量会直接参与写路径中的 \thesisterm{varstr} 自动决策，因此它们属于当前热路径优化的一部分。

\begin{figure}[H]
\centering
\includesvg[width=0.94\linewidth]{./demo/results/paper_dataset/supplement/figures/new_fig3_2_frame.svg}
\caption{Optbinlog 共享元数据与记录帧结构}
\label{fig:optbinlog-data-structure}
\end{figure}

如图\ref{fig:optbinlog-data-structure}所示，左侧共享元数据布局保证``跨进程可移植''，右侧记录帧布局保证``按帧可校验、按 schema 可还原''，二者共同构成第5章实验统计可追溯的结构基础。

\subsection{写入/读取路径}\label{ux5199ux5165ux8bfbux53d6ux8defux5f84}

写入与读取遵循``元数据状态判定→编码/解码执行→一致性校验''的统一流程。

\begin{figure}[H]
\centering
\includesvg[width=0.95\linewidth]{./demo/results/paper_dataset/supplement/figures/new_fig3_3_flow.svg}
\caption{Optbinlog 写入、读取与恢复流程}
\label{fig:optbinlog-rw-flow}
\end{figure}

如图\ref{fig:optbinlog-rw-flow}所示，写入与读取并非两条割裂路径，而是由同一共享 schema 贯通。该流程设计使``语义对齐实验''和``记录级复核''能够使用同一套格式定义，减少统计口径漂移。

写入路径： 1. 通过 \thesisterm{acquire\_shared\_view()} 获取共享文件视图；若共享文件未变化，则直接复用进程内缓存的 tag cache。 2. 根据 schema 统计量判断是否启用 \thesisterm{varstr}；当字符串字段占比高或最大定长字符串字段较大时，默认切换到 \thesisterm{varstr}。 3. 依据 tag cache 顺序编码 payload。 4. 构造 4 字节自描述帧头，其中同时编码 \thesisterm{payload\_len}、\thesisterm{varstr} 标志位和校验类型。 5. 对单条记录计算校验值；默认采用 \thesisterm{CRC32C(hw)}，仅在硬件不可用时回退到软件实现。 6. 将 \thesisterm{frame\_header + payload + checksum} 写入用户态缓冲区，按批次落盘。

读取路径： 1. 读取 4 字节帧头，并解析 \thesisterm{payload\_len}、\thesisterm{varstr} 标志位与 \thesisterm{checksum\_type}。 2. 读取 payload 与校验字段，并按帧头声明的校验类型执行校验。 3. 按共享格式解码：整数和浮点按固定宽度读取，字符串则根据 \thesisterm{varstr} 标志选择``长度前缀 + 数据''或``固定宽度截断/补零''方式。 4. 将解码后的记录交给统一回调接口。

因此，当前记录格式可写作：

\[
\mathrm{frame} = \mathrm{frame\_header} \;\Vert\; \mathrm{payload} \;\Vert\; \mathrm{checksum}
\]

其中：

\[
\mathrm{frame\_header}
=
\mathrm{payload\_len}
\;\cup\;
\mathrm{varstr\_bit}
\;\cup\;
\mathrm{checksum\_type}
\]

\subsection{关键优化路径与收敛}\label{ux5173ux952eux4f18ux5316ux8defux5f84ux4e0eux6536ux655b}

为突出与最终结果直接相关的设计取舍，本文将实现演化归纳为三个阶段：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{结构一致性阶段}：将``每事件独立结构''统一为``共享结构 + 位图 + 偏移表''，并将跨进程共享从指针语义改为偏移语义，解决映射基址不一致导致的可移植性问题。
\item
  \textbf{热路径压缩阶段}：将 tag 查找改为常数级索引路径，引入进程内共享视图缓存与 tag cache，把共享文件 reopen、tag 遍历与 schema 统计迁出稳态写路径。
\item
  \textbf{可靠性与空间协同阶段}：从软件 \thesisterm{CRC32} 迭代为默认 \thesisterm{CRC32C(hw)}，引入自描述帧头与 \thesisterm{varstr} 自动策略，并通过初始化状态机与 init-race 验证保证``单创建者 + 多复用者''并发语义。
\end{enumerate}

对应到当前版本，系统收敛点可概括为：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{可靠性不降、校验成本下降}：保留每条记录校验语义，同时以 \thesisterm{CRC32C(hw)} 降低热路径校验开销。
\item
  \textbf{共享元数据收益进入稳态}：共享视图缓存使 schema 解析与 tag 遍历不再重复进入每次写入。
\item
  \textbf{字符串冗余持续收缩}：string-heavy 场景下自动启用 \thesisterm{varstr}，减少固定宽度补零与无效拷贝。
\end{enumerate}

\subsection{工程场景适配性}\label{ux5de5ux7a0bux573aux666fux9002ux914dux6027}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  小样本 \(N\)：共享文件打开、共享视图校验、文件打开和缓冲准备等固定项占比仍然较高，时间优势不一定立刻出现。
\item
  中大样本 \(N\)：固定项被摊薄，自描述帧头和 \thesisterm{CRC32C(hw)} 的额外代价收敛到常数级，binary 的时间和空间优势更稳定。
\item
  string-heavy schema：自动 \thesisterm{varstr} 会显著减少补零写入和冗余字节，因此空间优势更容易显现。
\item
  高并发 \(D\)：即使校验和 schema 解析成本被压低，调度、文件竞争和写出路径仍会主导吞吐峰值与回落点。
\item
  多进程/多节点部署：共享 schema、进程内缓存复用和节点级初始化竞争正确性已经形成闭环，因此该设计更贴近真实工程部署。
\end{enumerate}

因此，第4章将把上述实现行为进一步抽象为 \(A_m/C_m/B_m/W_m\) 等参数形式，便于与第5章实验结果逐项对照。

\subsection{读取与复核能力设计}\label{ux8bfbux53d6ux4e0eux590dux6838ux80fdux529bux8bbeux8ba1}

为提升实验结果的可复核性，本文在写入与解码库之外提供了 \thesisterm{optbinlog\_read} 读取工具，用于对``共享格式文件 + 二进制日志文件''进行直接验证。该工具具备三项能力： 1. 按 tag 与字段名输出记录，避免只看到匿名值序列。 2. 支持 \thesisterm{table/jsonl} 两种格式，便于命令行排错与自动化分析脚本复用。 3. 支持 \thesisterm{--limit} 与 \thesisterm{--summary}，可快速截取样本并统计各 tag 分布。

该工具将``统计结果''与``底层记录''建立起可追溯映射，使第5章中的对比结论可回退到记录级证据，而非仅停留在汇总表与图形层面。

\subsection{断电场景下的截断恢复机制}\label{ux65adux7535ux573aux666fux4e0bux7684ux622aux65adux6062ux590dux673aux5236}

嵌入式设备掉电时，最常见的损坏形态是``尾帧半写入''：帧头、payload 或校验字段未完整落盘。若在该状态下直接续写，新数据将附着在损坏尾部之后，读取阶段会触发 \thesisterm{truncated} 或 \thesisterm{crc mismatch}，进而影响后续完整日志的可读性。

针对该问题，本文在写入路径中采用``尾部恢复后再续写''的处理策略，其核心原则如下：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  只保留\textbf{最后一条完整帧之前}的数据；
\item
  对损坏尾部执行一次性截断；
\item
  截断后再恢复 append 写入。
\end{enumerate}

实现流程可概括为：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  顺序扫描 \thesisterm{run.bin}，按 \texttt{frame\_header\ \textrightarrow{}\ payload\ \textrightarrow{}\ checksum} 逐帧校验；
\item
  记录``最后一条完整帧结束偏移'' \thesisterm{valid\_end}；
\item
  一旦遇到以下任一情况即判定尾部损坏： \thesisterm{frame length} 非法、\thesisterm{checksum\_type} 非法、payload 截断、CRC 截断、CRC 不一致；
\item
  将文件截断到 \thesisterm{valid\_end}；
\item
  在截断完成后恢复 append，确保后续日志可连续写入。
\end{enumerate}

工程上，append 模式在进入稳定写入前会先执行尾部一致性检查；当检测到半帧时先裁剪坏尾，再进入续写阶段。这样在``供电恢复/进程重启''后，系统能够优先消除上一次掉电遗留的尾部损坏，避免错误链式扩散。

该机制的语义边界为``可检测、可修复、可继续''：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  正在写入且被断电打断的那条日志视为不可恢复数据，将被丢弃；
\item
  已完整落盘的历史帧全部保留；
\item
  修复后后续日志可继续追加并被正常解码。
\end{enumerate}

因此，Optbinlog 在可靠性层面从``仅检测损坏''扩展为``检测 + 恢复 + 续写''的闭环处理，更符合真实嵌入式供电波动场景。

\section{理论模型与数学推导}\label{ux7406ux8bbaux6a21ux578bux4e0eux6570ux5b66ux63a8ux5bfc}

本章先做抽象推导，不代入具体事件文件数值。 为保证推导可解释且可验证，本文采用以下前提：在单次统计窗口内事件分布近似稳定；各模式在同一场景下使用一致的输入语义；模型参数以期望意义解释而非单点极值解释。

\subsection{符号定义}\label{ux7b26ux53f7ux5b9aux4e49}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(N\)：总日志条数。
\item
  \(D\)：并发设备数（或并发进程数）。
\item
  \(r\)：每设备日志条数，满足 \(N = D \cdot r\)。
\item
  \(k\)：事件类型（tag）索引，\(k \in \{1,\dots,T\}\)。
\item
  \(p_k\)：事件类型 \(k\) 的概率，满足 \(\sum_k p_k = 1\)。
\item
  \(e_k\)：事件 \(k\) 的字段数。
\item
  \(l_{k,j}\)：事件 \(k\) 第 \(j\) 个字段编码字节数。
\item
  \(L_k = \sum_{j=1}^{e_k} l_{k,j}\)：事件 \(k\) 的有效载荷字节数。
\item
  \(A_m\)：模式 \(m\) 的固定时间开销（初始化、打开文件、预热）。
\item
  \(C_m\)：模式 \(m\) 的单条边际时间开销。
\item
  \(B_m\)：模式 \(m\) 的单条边际空间开销。
\item
  \(S_{0,m}\)：模式 \(m\) 的固定空间开销（例如共享元数据）。
\item
  \(Q_m\)：吞吐（records/s）。
\end{enumerate}

除特别说明外，本章的``空间''均对应实验统计中的 \thesisterm{total\_bytes}：对 \thesisterm{binary} 为``日志文件 + 共享格式文件且共享文件只计一次''，对其他模式为其实际输出文件大小；``时间''默认对应单轮写入阶段的均值趋势，若切换到 \thesisterm{end\_to\_end} 口径，则把进程/节点启动与调度准备吸收到固定项中。

\subsection{从算法与数据结构到单条代价}\label{ux4eceux7b97ux6cd5ux4e0eux6570ux636eux7ed3ux6784ux5230ux5355ux6761ux4ee3ux4ef7}

为避免``只看结果不看机制''，先把每种模式拆成操作级成本。以下表达式均以单条记录期望开销为核心，再扩展到总量关系。

\subsubsection{text}\label{text}

单条记录包含： 1. 读取输入字段。 2. \thesisterm{snprintf}/拼接格式串。 3. 写入用户态缓冲并最终落盘。

时间可写为：

\[
t_{\text{text}}(k) = t_{\text{parse}}(k) + t_{\text{fmt}}(k) + t_{\text{write}}(\mathrm{bytes}_{\text{text}}(k))
\]

空间可写为：

\[
b_{\text{text}}(k) = b_{\text{prefix}} + b_{\text{delim}} + b_{\text{num}}(k) + b_{\text{str}}(k)
\]

其中 \(b_{\text{num}}(k)\) 是数字转十进制后的字符长度，随数值大小缓慢变化。

\subsubsection{binary（optbinlog）}\label{binaryoptbinlog}

当前 \thesisterm{optbinlog\_binlog\_write()} 的稳态路径可分为四步： 1. 复用共享视图与 tag cache，并按 \thesisterm{tag\_id} 完成常数级查找。 2. 按字段类型编码 payload。 3. 根据校验模式计算记录校验。 4. 将记录写入用户态批量缓冲区，缓冲区满后统一写出。

与代码一致，单条记录的物理布局为：

\[
\mathrm{frame}
=
\mathrm{frame\_header}_{4B}
\;\Vert\;
\mathrm{payload}
\;\Vert\;
\mathrm{checksum}_{4B}
\]

其中 payload 并不只是 schema 字段本身，还额外包含：

\[
\mathrm{payload}
=
\mathrm{timestamp}_{8B}
\;\Vert\;
\mathrm{tag\_id}_{2B}
\;\Vert\;
\mathrm{ele\_count}_{1B}
\;\Vert\;
\mathrm{fields}
\]

记记录内部固定元数据开销为：

\[
g_{\text{bin}} = 8 + 2 + 1 = 11
\]

若字符串字段按固定宽度写出，则 payload 中 schema 字段的有效载荷仍记为 \(L_k\)；若 schema 触发 \thesisterm{varstr}，则对应地记为 \(\widetilde{L}_k\)。于是单条时间可写为：

\[
t_{\text{bin}}(k)
=
t_{\text{lookup}}
+
t_{\text{encode}}(k)
+
t_{\text{checksum}}(k)
+
\bar{t}_{\text{write}}(b_{\text{bin}}(k))
\]

其中 \(\bar{t}_{\text{write}}(\cdot)\) 表示``经用户态缓冲摊薄后的平均写出代价''，比逐条系统调用更接近当前实现。

单条空间则可写为：

\[
b_{\text{bin}}(k) = 4 + g_{\text{bin}} + \widetilde{L}_k + 4
\]

固定空间只在共享格式文件层面体现，因此有：

\[
S_{0,\text{bin}} = S_{\text{shared}}
\]

若 schema 触发 \thesisterm{varstr}，则字符串字段的有效载荷不再按固定宽度写出，而是写为``长度前缀 + 实际长度''：

\[
\widetilde{L}_k
=
\sum_{j \in \mathcal{N}_k} l_{k,j}
\;+\;
\sum_{j \in \mathcal{S}_k} \left( 2 + \hat{l}_{k,j} \right),
\qquad
\hat{l}_{k,j} \le l_{k,j}
\]

还需说明两点与代码实现直接对应的口径： 1. 默认 \thesisterm{CRC32C(hw)} 覆盖的是``帧头 + payload''；兼容模式下 \thesisterm{CRC32} 覆盖 payload，\thesisterm{none} 则不做校验。 2. 共享文件的打开、映射与 tag cache 构建只在视图失效时重新执行，因此它们主要进入固定项 \(A_{\text{bin}}\)，而不是每条记录的边际项。

\subsubsection{syslog / ftrace}\label{syslog-ftrace}

抽象为``系统日志接口 + 内核路径''：

\[
t_{\text{sys}}(k) = t_{\text{format\_sys}}(k) + t_{\text{syscall}} + t_{\text{kernel\_path}} + t_{\text{write\_kernel}}
\]

\[
t_{\text{ftrace}}(k) = t_{\text{format\_ftrace}}(k) + t_{\text{trace\_write}} + t_{\text{ring\_or\_file\_path}}
\]

空间：

\[
b_{\text{sys}}(k) = b_{\text{sys\_header}} + b_{\text{sys\_payload}}(k)
\]

\[
b_{\text{ftrace}}(k) = b_{\text{ftrace\_header}} + b_{\text{ftrace\_payload}}(k)
\]

\subsubsection{nanolog\_like / zephyr\_deferred\_like / ulog\_async\_like / hilog\_lite\_like}\label{nanolog_like-zephyr_deferred_like-ulog_async_like-hilog_lite_like}

\thesisterm{nanolog\_like}（固定字段二进制）：

\[
t_{\text{nano}} = t_{\text{pack\_fixed}} + t_{\text{memcpy}} + t_{\text{write}}
\]

\[
b_{\text{nano}} = \text{const}
\]

\thesisterm{zephyr\_deferred\_like}（前台入队 + 后台批刷）：

\[
t_{z,\text{front}} = t_{\text{enqueue}} + t_{\text{signal}}
\]

\[
t_{z,\text{back}} = \frac{t_{\text{dequeue}} + t_{\text{batch\_write}}(k_b)}{k_b}
\]

\[
t_z = t_{z,\text{front}} + t_{z,\text{back}}
\]

\[
b_z = b_{\text{queue\_node}} + b_{\text{fixed}} + \frac{b_{\text{batch\_meta}}}{k_b}
\]

\thesisterm{ulog\_async\_like}（异步队列 + 文本 backend）：

\[
t_{\text{ulog}} =
t_{\text{enqueue}} + t_{\text{cond\_signal}}
+
\frac{t_{\text{dequeue}} + t_{\text{format\_text}} + t_{\text{backend\_write}}}{k_b}
\]

\[
b_{\text{ulog}} = b_{\text{queue\_node}} + b_{\text{text\_payload}}
\]

\thesisterm{hilog\_lite\_like}（受限字段 + 轻量头）：

\[
t_{\text{hilog}} = t_{\text{compact\_format}} + t_{\text{write}}(h_{\text{lite}} + p_{\text{lite}})
\]

\[
b_{\text{hilog}} = b_{\text{header\_lite}} + b_{\text{payload\_lite}}
\]

\subsection{\texorpdfstring{空间随 \(N\) 的详细推导}{空间随 N 的详细推导}}\label{ux7a7aux95f4ux968f-n-ux7684ux8be6ux7ec6ux63a8ux5bfc}

\subsubsection{期望单条字节}\label{ux671fux671bux5355ux6761ux5b57ux8282}

对任意模式 \(m\)，定义：

\[
B_m = \mathbb{E}[b_m(k)] = \sum_k p_k \, b_m(k)
\]

则总空间：

\begin{equation}
S_m(N) = S_{0,m} + N B_m
\label{eq:space-linear}
\end{equation}

这是严格的一阶线性模型，斜率为 \(B_m\)，截距为 \(S_{0,m}\)。 见公式（\ref{eq:space-linear}）所示，后续空间分析都可以归结为``固定项 + 单条斜率''两部分。

\subsubsection{binary 的显式表达}\label{binary-ux7684ux663eux5f0fux8868ux8fbe}

由 4.2.2 得：

\[
B_{\text{bin}} = 8 + g_{\text{bin}} + \sum_k p_k \widetilde{L}_k
\]

\begin{equation}
S_{\text{bin}}(N) = S_{\text{shared}} + N \left(8 + g_{\text{bin}} + \sum_k p_k \widetilde{L}_k\right)
\label{eq:bin-space-total}
\end{equation}

其中，\(g_{\text{bin}}=11\) 对应 payload 内部固定元数据，外层的两个 \thesisterm{4B} 分别对应帧头和校验字段。见公式（\ref{eq:bin-space-total}）所示，\thesisterm{binary} 的总空间同时受``每条记录固定项''和``共享格式文件固定项''影响。

\subsubsection{节省率与导数}\label{ux8282ux7701ux7387ux4e0eux5bfcux6570}

binary 相对模式 \(x\) 的空间节省率：

\[
\eta_{\text{bin}/x}(N) = 1 - \frac{S_{\text{bin}}(N)}{S_x(N)}
\]

若 \(S_{0,x}=0\)，则

\[
\eta_{\text{bin}/x}(N) = 1 - \frac{B_{\text{bin}} + S_{\text{shared}}/N}{B_x}
\]

求导：

\[
\frac{d\eta_{\text{bin}/x}}{dN} = \frac{S_{\text{shared}}}{N^2 B_x} > 0
\]

说明 \(N\) 越大，\(S_{\text{shared}}/N\) 越小，binary 空间优势越稳定。 这说明只要边际斜率满足优势条件，共享文件这一固定项就会随着 \(N\) 增大而被持续摊薄。

\subsubsection{空间交叉点}\label{ux7a7aux95f4ux4ea4ux53c9ux70b9}

当比较 \(\text{bin}\) 与 \(x\)：

\[
S_{\text{bin}}(N) \le S_x(N)
\]

\[
S_{\text{shared}} + N B_{\text{bin}} \le S_{0,x} + N B_x
\]

若 \(B_x > B_{\text{bin}}\)，可得交叉点：

\begin{equation}
N^*_{\text{space}}(\text{bin},x) = \frac{S_{\text{shared}} - S_{0,x}}{B_x - B_{\text{bin}}}
\label{eq:space-crossover}
\end{equation}

当 \(N > N^*_{\text{space}}\) 时，binary 空间一定优于 \(x\)。 因此，空间交叉点的判断可以直接按公式（\ref{eq:space-crossover}）所示进行。

\subsection{\texorpdfstring{时间随 \(N\) 的详细推导}{时间随 N 的详细推导}}\label{ux65f6ux95f4ux968f-n-ux7684ux8be6ux7ec6ux63a8ux5bfc}

\subsubsection{从单条时间到总时间}\label{ux4eceux5355ux6761ux65f6ux95f4ux5230ux603bux65f6ux95f4}

定义：

\[
C_m = \mathbb{E}[t_m(k)] = \sum_k p_k \, t_m(k)
\]

则总时间：

\begin{equation}
T_m(N) = A_m + N C_m
\label{eq:time-linear}
\end{equation}

该形式直接对应实验中的 \thesisterm{elapsed/write\_only/end\_to\_end} 指标。 见公式（\ref{eq:time-linear}）所示，总时间同样由固定项和边际项组成。若统计口径切换到 \thesisterm{end\_to\_end}，则进程/节点启动、文件准备与调度等待也并入 \(A_m\)。

\subsubsection{吞吐公式与单调性}\label{ux541eux5410ux516cux5f0fux4e0eux5355ux8c03ux6027}

\begin{equation}
Q_m(N) = \frac{1000N}{T_m(N)} = \frac{1000}{C_m + A_m/N}
\label{eq:throughput}
\end{equation}

对 \(N\) 求导：

\[
\frac{dQ_m}{dN} = \frac{1000A_m}{N^2\left(C_m + A_m/N\right)^2} > 0
\]

说明在模型中吞吐随 \(N\) 单调上升并渐近：

\[
\lim_{N \to \infty} Q_m(N) = \frac{1000}{C_m}
\]

在 \(A_m>0\) 与 \(C_m>0\) 条件下，吞吐增速会随 \(N\) 增大而下降，表现为``先快后慢''的收益递减，这与实验中大 \(N\) 后增益变缓一致。 因此，公式（\ref{eq:throughput}）给出了第 5 章解释``吞吐为何先快后慢趋于平台''的统一表达。

\subsubsection{时间交叉点（binary 何时优于其他模式）}\label{ux65f6ux95f4ux4ea4ux53c9ux70b9binary-ux4f55ux65f6ux4f18ux4e8eux5176ux4ed6ux6a21ux5f0f}

比较 \(\text{bin}\) 与 \(x\)：

\[
T_{\text{bin}}(N) \le T_x(N)
\]

\[
A_{\text{bin}} + N C_{\text{bin}} \le A_x + N C_x
\]

当 \(C_x > C_{\text{bin}}\) 时：

\begin{equation}
N^*_{\text{time}}(\text{bin},x) = \frac{A_{\text{bin}} - A_x}{C_x - C_{\text{bin}}}
\label{eq:time-crossover}
\end{equation}

解释： 1. 若 \(A_{\text{bin}}\) 较大（共享初始化重），小 \(N\) 可能不占优。 2. 随 \(N\) 增大，\(NC\) 项主导，只要 \(C_{\text{bin}} < C_x\)，最终会跨过交叉点。

因此，某一模式何时在时间上完成反超，可按公式（\ref{eq:time-crossover}）所示理解。

\subsubsection{\texorpdfstring{\(C_m\) 的字段复杂度展开}{C\_m 的字段复杂度展开}}\label{c_m-ux7684ux5b57ux6bb5ux590dux6742ux5ea6ux5c55ux5f00}

为把``常数项差异''写清楚，可把边际开销展开到字段数级别：

\[
C_m = c_{m,0} + c_{m,1}\mathbb{E}[e_k] + c_{m,2}\mathbb{E}[L_k] + c_{m,3}\mathbb{E}[\mathrm{syscall}_m]
\]

其中： 1. \(\mathbb{E}[e_k]\) 决定字段遍历成本（编码循环次数）。 2. \(\mathbb{E}[L_k]\) 决定 memcpy、校验和写出字节相关成本。 3. \(\mathbb{E}[\mathrm{syscall}_m]\) 反映模式是否显著依赖系统日志接口或内核路径。

结合当前复现代码，可得到更贴近实现的解释： 1. \thesisterm{text\_semantic\_like} 主要承担字段格式化与文本展开成本。 2. \thesisterm{binary} 的边际项主要由 typed encode、校验和批量写出摊销组成，固定项则额外包含共享视图与 tag cache 准备。 3. \thesisterm{nanolog\_like / zephyr\_deferred\_like} 走固定宽度打包路径，边际项通常较低，但 \thesisterm{zephyr\_deferred\_like} 还会受到批刷组织方式影响。 4. \thesisterm{ulog\_async\_like / syslog / ftrace} 会显著暴露文本展开、系统接口或内核路径的常数开销。 5. \thesisterm{hilog\_lite\_like} 通过轻量头和受限消息体压缩了单条记录规模，因此常落在二进制直写组与文本系统组之间。

\subsection{\texorpdfstring{多设备 \(D\) 的详细推导}{多设备 D 的详细推导}}\label{ux591aux8bbeux5907-d-ux7684ux8be6ux7ec6ux63a8ux5bfc}

\subsubsection{总时间分解}\label{ux603bux65f6ux95f4ux5206ux89e3}

设每设备写 \(r\) 条：

\begin{equation}
E_m(D,r) = A_{\text{spawn}}(D) + A_m + rC_m + W_m(D,r)
\label{eq:multi-time}
\end{equation}

见公式（\ref{eq:multi-time}）所示，多设备总时间除了单机边际项外，还需要显式计入并发等待项。若观察的是 \thesisterm{write\_only} 指标，则 \(A_{\text{spawn}}(D)\) 会明显缩小；若观察 \thesisterm{end\_to\_end}，它则不能忽略。

其中 \(W_m\) 为竞争等待项（锁、队列、内核资源、I/O 饱和）。

\subsubsection{队列化近似（解释吞吐饱和）}\label{ux961fux5217ux5316ux8fd1ux4f3cux89e3ux91caux541eux5410ux9971ux548c}

把共享瓶颈抽象成单服务台，服务率 \(\mu_m\)（records/s），到达率 \(\lambda = D\nu\)（\(\nu\) 为单设备提交率）。

稳定条件：

\[
\rho = \frac{\lambda}{\mu_m} < 1
\]

等待时间近似随 \(\rho\) 增大而快速上升，可抽象为：

\[
W_m(D,r) \sim r f_m(\rho),
\qquad
f_m'(\rho) > 0,
\qquad
f_m''(\rho) > 0
\]

因此当 \(D\) 接近

\[
D_{\text{sat},m} \approx \frac{\mu_m}{\nu}
\]

吞吐进入平台区；超过后出现明显回落。 这里的``服务台''不一定是真实队列，也可以对应共享文件写出通道、后台批刷线程或内核 tracing 接口；该抽象的意义在于用同一符号解释不同模式的饱和与回落。

\subsubsection{常用二次近似}\label{ux5e38ux7528ux4e8cux6b21ux8fd1ux4f3c}

在工程拟合中常写作：

\[
E_m(D,r) \approx \beta_m + rC_m + \alpha_m D + \gamma_m D^2
\]

总吞吐：

\[
Q_m(D,r) = \frac{1000Dr}{E_m(D,r)}
\]

当 \(\gamma_m > 0\) 时，\(D\) 过大必然导致分母增长过快，出现峰值后下降。

进一步对 \(D\) 求导：

\[
\frac{dQ_m}{dD}
\propto
\left(\beta_m + rC_m + \alpha_m D + \gamma_m D^2\right)
-
D\left(\alpha_m + 2\gamma_m D\right)
\]

整理得：

\[
\frac{dQ_m}{dD} \propto \beta_m + rC_m - \gamma_m D^2
\]

故吞吐峰值点近似为：

\begin{equation}
D_{\text{peak},m} = \sqrt{\frac{\beta_m + rC_m}{\gamma_m}}
\label{eq:device-peak}
\end{equation}

这给出了可直接与实验扫描结果比对的``理论峰值设备数''。 其中，峰值设备数可直接按公式（\ref{eq:device-peak}）与实验扫描结果对照。

\subsubsection{\texorpdfstring{空间随 \(D\)}{空间随 D}}\label{ux7a7aux95f4ux968f-d}

\[
S_m(D,r) = S_{0,m} + DrB_m
\]

因此空间对 \(D\) 严格线性，差异主要由斜率 \(B_m\) 与截距 \(S_{0,m}\) 决定。 这一点与当前实验统计口径一致：在记录规模 \(r\) 固定时，设备数增加只会线性放大记录总量，而不会改变每条记录的平均编码体积。

\subsection{模式间理论排序与可检验预测}\label{ux6a21ux5f0fux95f4ux7406ux8bbaux6392ux5e8fux4e0eux53efux68c0ux9a8cux9884ux6d4b}

\subsubsection{异语义比较（最短路径口径）}\label{ux5f02ux8bedux4e49ux6bd4ux8f83ux6700ux77edux8defux5f84ux53e3ux5f84}

在工程近似层中，各模式的输入语义和存储组织并不完全一致，因此本节只讨论``机制上更可能出现的方向''，而不把它写成固定全序。

从当前复现代码出发，可得到三条较稳定的预期： 1. 固定宽度二进制直写组（如 \thesisterm{nanolog\_like}、\thesisterm{zephyr\_deferred\_like}）通常拥有更低的单位记录空间与更短的边际写入路径。 2. 文本展开或系统路径更重的模式（如 \thesisterm{text/syslog/ftrace/ulog\_async\_like}）通常拥有更高的常数项。 3. \thesisterm{binary} 与 \thesisterm{hilog\_lite\_like} 处于二者之间：前者保留共享 schema、自描述帧和校验语义，后者保留轻量头和受限消息体，因此二者的排序会更依赖具体 schema 与统计口径。

可检验预测： 1. nanolog\_like 在时间与吞吐上常呈数量级优势。 2. hilog\_lite\_like 在``受限语义''下应明显优于 text/syslog。 3. ulog\_async\_like 在吞吐未饱和且后台轻负载时有机会受益；若后台文本后端成为瓶颈，可能慢于 binary/text。 4. 上述优势不代表语义等价下仍保持同倍率。

\subsubsection{语义对齐比较（公平口径）}\label{ux8bedux4e49ux5bf9ux9f50ux6bd4ux8f83ux516cux5e73ux53e3ux5f84}

本文所说的``语义对齐''，并不是把所有 peer 强行改写成同一种记录格式，而是在\textbf{同一组字段语义、同一组字段值与同一平台}下，让 Optbinlog 与 peer 分别走各自的 native-like 复现路径。于是：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \thesisterm{binary} 的差异主要来自共享 schema、frame 组织、per-record CRC 以及字段遍历。
\item
  peer 的差异主要来自其 native-like 记录布局、批刷/后台路径、文本展开与系统接口路径。
\end{enumerate}

因此，公平口径下的空间关系不再由``语义体量不同''主导，而主要由各模式的\textbf{单位记录常数项}主导。对于 \thesisterm{nanolog\_like / zephyr\_deferred\_like} 这类紧凑二进制 peer，\(B_{\text{peer}}\) 往往长期小于 \(B_{\text{bin}}\)；对于 \thesisterm{ulog\_async\_like / hilog\_lite\_like / syslog} 这类仍带有消息头、文本展开或系统路径附加开销的 peer，\(B_{\text{bin}}\) 更可能在较小 \(N\) 后越过交叉点。

\(C_{\text{peer}}\) 与 \(C_{\text{bin}}\) 的差异同样主要来自实现常数，包括编码循环、缓存命中、批刷组织、格式化路径与校验实现，而不再主要来自``消息语义更多或更少''的偏差。

可检验预测： 1. 时间差异将由各模式的原生热路径决定，而不是由语义字段数决定。 2. 空间差异会在同一语义组内呈现稳定斜率，并在 \(N\) 增长时显现是否存在交叉点。

\subsection{与第5章指标的映射关系}\label{ux4e0eux7b2c5ux7ae0ux6307ux6807ux7684ux6620ux5c04ux5173ux7cfb}

为了让推导可被实验验证，建立公式到指标的映射：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \(T_m(N)\) 对应 \thesisterm{elapsed\_ms} 或 \thesisterm{end\_to\_end\_ms} 的均值趋势。
\item
  \(Q_m\) 对应 \thesisterm{throughput\_rps} / \thesisterm{throughput\_e2e\_rps}。
\item
  \(S_m\) 对应 \thesisterm{total\_bytes}。
\item
  \(N^*_{\text{time}}\) 与 \(N^*_{\text{space}}\) 对应``某场景下 binary 由劣转优''的阈值。
\item
  \(D_{\text{sat},m}\) 对应``多设备扫描中吞吐峰值后回落''的拐点。
\end{enumerate}

本章给出的是``由算法与数据结构直接推出的理论趋势''；第5章再代入实测数据检验这些趋势是否成立，以及偏离原因。

\subsection{本章小结}\label{ux672cux7ae0ux5c0fux7ed3}

本章完成了三项工作：一是把不同日志模式统一到可比较的时间与空间参数形式；二是给出 \(N\) 与 \(D\) 两个核心自变量下的交叉点、峰值点与线性斜率含义；三是建立理论参数与实验指标的映射关系。基于这些结果，第5章将按``单机 \textrightarrow{} 本地多设备 \textrightarrow{} 真实多设备模拟 \textrightarrow{} 日志量扫描''的顺序进行逐层验证。

\subsection{理论---实证对照}\label{ux7406ux8bbaux5b9eux8bc1ux5bf9ux7167}

综合 5.3～5.6 与第4章模型，可得到五点结论：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  空间模型与实测保持一致：\thesisterm{total\_bytes} 随 \(N\) 近似线性增长，不同语义组对应不同斜率 \(B_m\)。\thesisterm{nanolog/zephyr} 组中 binary 空间持续为负，\thesisterm{syslog/ulog/hilog} 组中 binary 空间持续为正，l1 的 \thesisterm{ftrace} 组同样保持空间正收益。
\item
  时间与吞吐交叉点具有``语义组相关、平台相关''特征，而非统一阈值。\thesisterm{nanolog/zephyr} 在 single 可转正，但在 l1 再次转负；\thesisterm{syslog/ulog} 在 l1 保持稳定正收益；\thesisterm{hilog} 在 l1 呈``空间正、时间吞吐负''的分化形态；\thesisterm{ftrace} 在 l1 时间维度持续负区。
\item
  竞争项 \(W_m(D,r)\) 在本地多设备与真实多设备中均可观测。本地多设备表现为``时间/吞吐振荡、空间斜率稳定''，真实多设备则进一步放大为``持续正区/持续负区/近持平''的结构性分化。
\item
  真实多设备不是本地多设备的线性外推。l1 场景引入节点冷启动、VM 调度噪声与进程重建成本，使固定项 \(A_m\) 占比上升，并直接改变不同模式的收益分布。
\item
  日志量扩展实验给出可操作阈值：\thesisterm{syslog/ulog} 在 \(N\approx 10\) 左右完成空间反超，\thesisterm{hilog} 反超点约在 \(N\approx 30\text{～}50\)，\thesisterm{nanolog/zephyr} 在 \(N\le 100000\) 仍未出现空间交叉。
\end{enumerate}

\subsection{证据链闭环与外推边界}\label{ux8bc1ux636eux94feux95edux73afux4e0eux5916ux63a8ux8fb9ux754c}

本文主证据链由``最新严格公平层 + 初始化竞争验证 + 空间交叉点扫描''组成： 1. \textbf{严格公平层主数据}：\thesisterm{text\_semantic\_like / binary / peer native mode} 在 single、本地多设备与真实多设备三层结果。 2. \textbf{正确性验证数据}：本地与节点级初始化竞争，验证``单创建者 + 多复用者''并发语义。 3. \textbf{阈值验证数据}：\(N=1\text{～}100000\) 空间扫描，验证不同语义组的空间交叉点是否存在。

该证据链使结论具备三重可检验性：指标可复核、机制可解释、阈值可复现。工程近似层在本章中仅用于补充说明，不承担最终排序结论。

\subsection{机制解释与应用启示}\label{ux673aux5236ux89e3ux91caux4e0eux5e94ux7528ux542fux793a}

结合最新严格公平结果与消融数据，可将 Optbinlog 的分组结论概括为： 1. \textbf{稳定优势组（\thesisterm{syslog/ulog}）}：在 single 与真实多设备均保持清晰正收益，是当前最确定的应用受益区间。 2. \textbf{中间过渡组（\thesisterm{hilog}）}：空间长期占优，但在节点级并发下时间与吞吐转为负区，体现``空间稳、时间敏感''的过渡态。 3. \textbf{高压竞争组（\thesisterm{nanolog/zephyr}）与 tracing 组（\thesisterm{ftrace}）}：前者在当前口径下仍未跨越节点级交叉点，后者维持``空间正收益、时间吞吐负收益''。

由此可见，Optbinlog 的工程价值主要体现在三点：跨进程一致性强、语义扩展成本低、记录级可校验；其后续优化重点应继续聚焦于冷路径压缩、稳态缓存收益释放与批量提交效率提升。

\section{结论与未来展望}\label{ux7ed3ux8bbaux4e0eux672aux6765ux5c55ux671b}

\subsection{研究结论}\label{ux7814ux7a76ux7ed3ux8bba}

基于最新一版严格语义对齐数据以及初始化竞争、空间交叉点等辅助证据，本文得到以下结论：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \thesisterm{binary} 相对 \thesisterm{text} 在全部语义组上保持稳定正收益，说明结构化二进制路径在高频写入场景下具备明确基线优势。
\item
  在 peer 原生路径对比中，\thesisterm{syslog/ulog} 属于稳定优势组，\thesisterm{hilog} 呈现``空间优势稳定、时间吞吐受节点级并发抑制''的过渡特征，\thesisterm{nanolog/zephyr} 在当前节点级口径下仍未完成反超，\thesisterm{ftrace} 保持``空间正收益、时间吞吐负收益''。
\item
  空间交叉点具有语义组相关性：\thesisterm{syslog/ulog} 在 \(N\approx 10\) 出现反超，\thesisterm{hilog} 在 \(N\approx 50\) 左右出现反超，\thesisterm{nanolog/zephyr} 在 \(N\le 100000\) 仍未出现交叉。
\item
  初始化竞争结果持续满足``单创建者 + 多复用者''约束，说明共享格式文件方案在本地与节点级并发下均保持正确性。
\end{enumerate}

总体而言，本文提出的理论模型能够较好解释最新实验趋势，说明 Optbinlog 更适用于长期在线、强调跨进程一致性、结构化语义和结果可追溯性的嵌入式网关与边缘控制场景。

\subsection{当前局限}\label{ux5f53ux524dux5c40ux9650}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  受网络与平台限制，尚未完成外部系统官方工程的全栈编译复现（当前为代码级阅读 + 算法级原型复现）。
\item
  Linux 实验仍以 VM 为主，尚未覆盖 Raspberry Pi / i.MX 实机。
\item
  严格公平语义套件已经扩展到 20 节点 VM 级节点代理场景，但仍未迁移到真实 ARM 板卡或 bare-metal Linux 实机。
\item
  当前严格公平套件采用``新进程 + 单次写入''的统计口径，因此新加入的进程内 shared/tag cache 收益尚未在长生命周期 steady-state 基准中得到充分量化。
\item
  缺少 \thesisterm{perf} 级 CPU 微观指标，暂不能细粒度解释所有异常点。
\end{enumerate}

\subsection{未来展望}\label{ux672aux6765ux5c55ux671b}

面向后续研究，本文认为可以从以下几个方向继续推进： 1. 在更广泛的平台与负载条件下开展验证，进一步提升结论的外部有效性与可迁移性。 2. 在保持语义一致性的前提下继续优化热路径与稳态路径，缩小与极短路径 peer 在特定场景下的差距。 3. 完善日志可靠性与可恢复性评估，形成覆盖损坏、截断与异常恢复过程的系统化证据。 4. 结合更细粒度观测手段（如 CPU 与调度行为分析），建立从机制到性能现象的更强解释闭环。

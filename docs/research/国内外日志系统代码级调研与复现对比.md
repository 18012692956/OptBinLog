## 2. 代码级实现分析（算法逻辑 + 数据结构）

### 2.1 Linux ftrace：per-CPU ring buffer（无锁写路径）

从 `ring_buffer.c` 与官方设计文档可见，核心数据结构是“每 CPU 一个 ring buffer + page 链表/提交状态”。

1. 数据结构特征
- `ring_buffer_per_cpu` 持有每 CPU 的写入上下文。
- ring 中 page 作为基本管理单元，维护 `head/commit/tail` 等状态。

2. 算法特征
- 写入不是全局大锁串行，而是优先在本 CPU 缓冲进行预留与提交。
- 通过提交序列与事件长度信息控制并发可见性。

3. 与 optbinlog 的主要差异
- ftrace 优先优化“高并发内核 trace 写路径”；
- optbinlog 优先优化“应用侧结构化编码 + 空间效率 + 跨进程元数据复用”。

### 2.2 Zephyr logging：deferred logging + message pool

从 `Kconfig.mode` 和 `log_msg.c` 可见，Zephyr 的关键是把前台任务日志写入动作“轻量化”，把真正输出延后。

1. 数据结构特征
- `log_msg` 消息对象 + 消息池（pool）。
- 实际后端输出在后台上下文处理。

2. 算法特征
- deferred 模式下，前台路径做“封包/入队”，避免同步慢 I/O。
- 可配置 overflow/drop 策略，在资源受限时保证系统可运行性。

3. 与 optbinlog 的主要差异
- Zephyr 重点在“实时线程不被日志阻塞”；
- optbinlog 重点在“编码压缩 + 可回放 + 共享格式表”。

### 2.3 NanoLog：编译期提取格式 + 运行期只写动态参数

从 `NanoLogCpp17.h` / `RuntimeLogger.cc` 可见，NanoLog 核心策略是把格式字符串处理迁出热路径。

1. 数据结构特征
- 运行时记录包含日志 ID 与动态参数，不保存完整格式文本。
- `RuntimeLogger` 使用 `StagingBuffer`、后台压缩线程（`compressionThreadMain`）等组件。

2. 算法特征
- 前台路径尽量只做参数打包。
- 后台线程做批量压缩与落盘，离线恢复文本。

3. 与 optbinlog 的主要差异
- NanoLog 依赖编译期插桩/宏展开体系；
- optbinlog 依赖事件定义文件 + 共享元数据，工程接入成本更低。

### 2.4 RT-Thread ulog：异步输出与 backend 抽象

从 `ulog.c` 可见 ulog 提供异步输出入口（含异步线程/缓冲），并抽象多个 backend。

1. 数据结构特征
- ringbuffer/rbb 作为暂存区。
- backend 列表管理输出目的地。

2. 算法特征
- 前台写入缓冲；后台线程统一输出。
- 目标是减小实时线程日志抖动。

3. 与 optbinlog 的主要差异
- ulog 优先调度友好与插件化输出；
- optbinlog 优先结构化编码效率与空间压缩。

### 2.5 systemd-journald：系统级二进制索引日志

从 `JOURNAL_FILE_FORMAT` 与 `journald-server.c` 可见，journald 是“系统级、可检索、带索引”的日志体系。

1. 数据结构特征
- journal 文件中存在 header、对象和索引结构。

2. 算法特征
- 强调一致性与检索能力，而非仅单条写入极限速度。

3. 与 optbinlog 的主要差异
- journald 更重系统功能完备性；
- optbinlog 更偏轻量嵌入式高频记录。

## 3. 与 optbinlog 的统一对比

| 维度 | optbinlog | ftrace | Zephyr deferred | NanoLog | RT-Thread ulog | journald |
|---|---|---|---|---|---|---|
| 热路径 | 按 schema 二进制编码 | per-CPU ring reserve/commit | 入队+后端线程 | 参数打包，格式离线恢复 | 入缓冲+后端线程 | 系统级对象写入 |
| 元数据 | 共享文件 + 偏移访问 | 内核 ring/page 元数据 | msg pool/队列元数据 | 编译期静态元数据 | backend/ringbuffer 元数据 | 文件对象与索引 |
| 核心优化目标 | 空间 + 一致性 + 编解码效率 | 并发 trace 写入 | 减少前台阻塞 | 极低调用点开销 | 实时线程友好 | 检索/持久化一致性 |
| 嵌入式接入成本 | 中（事件文件+库） | 高（内核设施） | 中（RTOS 配置） | 较高（编译链耦合） | 低-中 | 较高 |

## 4. 本地复现的典型案例与实现映射

由于当前环境存在外网拉仓库和 Linux tracing 工具限制，采用“算法复现原型”方式落地了四类典型案例：

1. NanoLog-like 原型（`nanolog_like`）
- 文件：`demo/research/research_prototypes.c`
- 思路映射：只写紧凑二进制参数，不在热路径做文本格式化。

2. Zephyr-deferred-like 原型（`zephyr_deferred_like`）
- 文件：`demo/research/research_prototypes.c`
- 思路映射：生产者入队 + 后台线程落盘，前台与慢 I/O 解耦。

3. RT-Thread ulog-async-like 原型（`ulog_async_like`）
- 文件：`demo/research/research_prototypes.c`
- 思路映射：前台只做入队，后台线程完成文本格式化与 backend 写入，复现异步日志线程模型。

4. HiLog-Lite-like 原型（`hilog_lite_like`）
- 文件：`demo/research/research_prototypes.c`
- 思路映射：受限字段 + 轻量头（`sec/nsec/domain/tag/level/msg_len`）+ 短消息体，复现 HiLog Lite 的“受限格式、轻量记录”路径。

5. 自动化对比脚本
- 文件：`demo/run_research_compare.py`
- 对比模式：  
  `text / optbinlog binary / syslog / research_text / nanolog_like / zephyr_deferred_like / ulog_async_like / hilog_lite_like / nanolog_semantic_like / zephyr_deferred_semantic_like / binary_semantic_like`
- 统计：IQR 异常值处理 + `mean/median/p95/std/95%CI` + 吞吐/空间对比。

## 6. 结果解释与边界

这些“研究原型”用于验证算法趋势，不等价于原系统全栈复现：

1. `nanolog_like` 没有完整复现 NanoLog 的编译期 instrumentation 与离线解码器，仅复现“动态参数二进制记录”这一核心思想。
2. `zephyr_deferred_like` 复现的是“前台入队+后台落盘”机制，不等价于 Zephyr 内核对象/调度器实现细节。
3. `ulog_async_like` 复现的是 ulog 的异步线程 + 文本 backend 思路，不含 RT-Thread 内核调度与设备驱动细节。
4. `hilog_lite_like` 复现的是“受限格式 + 轻量头 + 短消息体”的日志编码策略，不等价于 OpenHarmony HiLog Lite 的系统全栈实现。
5. `*_semantic_like` 进一步进行了字段与帧语义拉齐，能更公平地比较“实现开销差异”，但仍不是官方工程全栈复现。
6. 因此绝对数值不能直接当作“原系统官方性能”，但可用于和 optbinlog 做“算法机制级”对照。
